# EXAMPLES.md

> **用途**：配合根 [`AGENTS.md`](./AGENTS.md) **L1** 的示意代码与对照表；供代理与代码审查引用。  
> **非**：人类上手指南（请用 [`README.md`](./README.md)）；非可拷贝进业务的实现。

正文为 **编码与审查准则** 的正反例示意（说明意图，非可运行业务代码）。审查时对照：是否出现同类误区。

---

## 1. 思考（先澄清，勿默认）

**场景**：用户说「导出用户数据」。

| ❌ 常见误区 | ✅ 更合适 |
|------------|-----------|
| 直接写「导出全部用户到本地文件」，默认字段与路径 | 先列出假设：**范围**（全量/筛选）、**交付形态**（下载/API/异步任务）、**字段与合规**、**量级**；再选最小方案 |
| 用户说「让搜索更快」就同时上缓存、索引、异步 | 并列可能含义（延迟 / 吞吐 / 感知速度），问清指标再动刀 |

**要点**：多解时**写出来并列**，不要静默选一种「看起来对」的实现。

---

## 2. 简洁（最小可行，不预支复杂度）

**场景**：「算一笔折扣金额」。

| ❌ 常见误区 | ✅ 更合适 |
|------------|-----------|
| 一上来 `Strategy` / `Protocol` / 多类继承，单场景也要可配置一切 | 一个函数解决当前规则；**真有**多种规则再抽类型 |
| 「保存用户偏好」顺手加缓存、通知、合并策略、通用校验器 | 只做「写入 DB」这一件事；其余等**明确需求**再加 |

**要点**：复杂度要**晚于**需求出现，而不是早于。

---

## 3. 手术式修改（只动该动的）

**场景**：「空邮箱导致校验崩溃」。

| ❌ 常见误区 | ✅ 更合适 |
|------------|-----------|
| 顺手改邮箱正则、加用户名校验、重排注释、统一引号风格 | **只改**与空值相关的分支；风格与周边文件一致 |
| 「给上传加日志」时全局改类型标注、返回值写法、引号 | **只加**日志与必要 `import`；其余保持原样 |

**要点**：每一行 diff 都应能指向**本条需求**；本 diff 产生的未使用符号须清理。

---

## 4. 可验证目标（先定义「怎样算做完」）

**场景**：「修一下认证」或「接口要限流」。

| ❌ 常见误区 | ✅ 更合适 |
|------------|-----------|
| 「先通读代码再整体优化」，无验收标准 | 问清具体故障或指标 → **先写失败用例或复现步骤** → 再改 → 再跑全量相关测试 |
| 一次提交里 Redis、配置中心、监控全上 | 拆步：**每步有可独立验证的结果**（例如先单端点内存限流，再推广 middleware） |

**场景**：「同分排序不稳定」。

| ❌ 常见误区 | ✅ 更合适 |
|------------|-----------|
| 直接改 `sort`，无测试 | 先写**稳定复现**（同输入多次运行应一致），再改比较键（如次级按 `name`） |

**要点**：弱标准（「弄好就行」）→ 强标准（「这条测试绿 / 这组 curl 行为符合」）。

## 5. 注释与审查（意图与契约要说清）

本仓库要求：关键逻辑、公共 API、边界与非显而易见处须有 **TSDoc/JSDoc** 等约定注释；**不足以说明意图与契约的变更不得视为通过审查。**

**示例（TypeScript 示意）**

```ts
// ❌ 公共导出却无说明：调用方不知道约束与失败形态
export function resolvePageToken(raw: string | undefined) {
  if (raw == null || raw === "") return { cursor: null };
  // ...
}

/**
 * 将客户端传入的分页 token 解析为服务端游标。
 *
 * @param raw - 查询串中的 `pageToken`；缺省或空串表示首页
 * @returns 解析结果；非法格式时抛出 {@link PageTokenError}
 * @throws {PageTokenError} token 非空但无法解析时
 */
export function resolvePageToken(raw: string | undefined) {
  if (raw == null || raw === "") return { cursor: null };
  // ...
}
```

**要点**：注释服务**读者与审查**——说明「约定、边界、失败方式」，而非复述代码字面。

## 对照速查

| 准则 | 反模式 | 方向 |
|------|--------|------|
| 思考 | 默认范围、格式、字段 | 列假设、多问一句 |
| 简洁 | 为单场景建框架 | 最小实现，需求变再抽 |
| 手术式 | 修 bug 时改风格、加无关校验 | 只改相关行，风格跟现有代码 |
| 可验证 | 无测试/无复现的「重构」 | 先失败用例或步骤，再实现 |
| 注释 | 公共 API 无文档、关键分支无语义说明 | TSDoc/JSDoc + 关键行说明意图 |

**一句话**：用**今天**的最小正确方案解决问题；复杂度与抽象**滞后**于真实需求，不提前透支。
