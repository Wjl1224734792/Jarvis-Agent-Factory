# 代码审查与质量

## 概述

代码审查不是找茬，是**质量保证的最后一道防线**。本技能提供结构化的审查方法论，确保每段代码在合并前都经过多维度评估。

**核心原则：**

1. **证据驱动** — 审查检查的是代码质量，不是作者的偏好。每条 finding 必须有证据支撑（文件路径、行号、命令输出）。

2. **发现与过滤分离** — 发现阶段不预过滤：发现所有问题（含低严重度和不确定的），不做预判。过滤阶段由下游执行：审查报告消费者（编排者/qa-review-expert）负责过滤和优先级排序。"宁可发现一个最终被过滤掉的问题，也不因自我审查而漏掉真正的 bug。"——覆盖率是发现阶段的目标，精确度是过滤阶段的目标。

3. **自查与校验** — 输出前必须自检：每条 CRITICAL/BLOCKED 发现问自己"作者能立即反驳吗？"，区分 FLAW（缺陷）vs PREFERENCE（偏好），LOW 置信度的严重发现应降级到 Open Questions。

**必需背景：** 你必须理解 `behavioral-guidelines`（准则 3：精准修改）。

## 何时使用

**适用场景：**
- PR / diff 代码审查
- 实现交付后的质量评审
- 代码重构后的安全审查
- 新人代码的导师审查
- 合并前的最后检查

**不适用场景：**
- 项目级架构审查（交给 `project-review-expert`）
- 性能专项审查（交给 `perf-review-expert`）
- 审查范围超过 1000 行变更（建议拆分审查）

## 五轴审查框架

每个变更至少从以下五个维度评估：

### 1. 正确性

| 检查项 | 问题示例 |
|--------|---------|
| 是否符合需求规格？ | 有没有遗漏验收标准？ |
| 边界条件是否处理？ | 空值、最大值、负数、零值 |
| 错误路径是否覆盖？ | 异常是否被正确处理？有没有空的 catch 块？ |
| 是否存在竞态条件？ | 异步操作是否考虑了并发？ |
| 数据是否一致？ | 写入的数据能否正确读取？事务边界是否合理？ |
| 是否幂等？ | 重复执行是否产生副作用？ |

### 2. 可读性与简洁性

| 检查项 | 问题示例 |
|--------|---------|
| 命名是否清晰？ | 函数名描述"做什么"而非"怎么做" |
| 控制流是否直观？ | 深层嵌套、过多分支、goto 式跳转 |
| 是否能用更少代码？ | 冗余变量、不必要的中间步骤、重复逻辑 |
| 逻辑是否自解释？ | 需要注释解释的代码通常是不够清晰的代码 |
| 文件职责是否单一？ | 一个文件做了太多不相关的事 |

### 3. 架构

| 检查项 | 问题示例 |
|--------|---------|
| 是否契合系统设计？ | 新代码是否违反了现有的分层约定？ |
| 模块边界是否清楚？ | 跨层调用、循环依赖、紧耦合 |
| 抽象层级是否适当？ | 为单点使用创建过度抽象？ |
| 依赖方向是否合理？ | 底层模块依赖了上层模块？ |

### 4. 安全

| 检查项 | 问题示例 |
|--------|---------|
| 输入是否验证？ | SQL 注入、XSS、命令注入风险 |
| 密钥是否安全？ | 硬编码密钥、密钥在日志中出现 |
| 鉴权是否到位？ | 缺少权限检查、可被越权访问 |
| 查询是否参数化？ | 拼接 SQL、拼接命令 |
| 外部数据是否视为不可信？ | 前端传来的数据直接用于后端逻辑 |

### 5. 性能

| 检查项 | 问题示例 |
|--------|---------|
| 是否存在 N+1 查询？ | 循环内发起数据库查询 |
| 是否有无限循环风险？ | while(true) 缺少退出条件 |
| 同步操作是否应为异步？ | 阻塞主线程的同步 I/O |
| 是否缺少分页？ | 全量查询可能返回百万条记录 |
| 是否存在内存泄漏？ | 事件监听未清理、闭包持有大对象 |

## 变更规模标准

| 变更行数 | 评估 | 操作 |
|---------|------|------|
| ≤100 行 | 良好 | 一次审查可完成 |
| 100-300 行 | 可接受 | 需是一次逻辑变更（不能是多个不相关改动拼凑） |
| 300-1000 行 | 偏大 | 建议拆分为多次独立审查 |
| >1000 行 | 过大 | 要求拆分，否则审查质量无法保证 |

## 严重度分级

每条 finding 使用**严重度 + 置信度**双维度标注：

### 严重度

| 标签 | 含义 | 作者操作 |
|------|------|---------|
| **[CRITICAL]** | 阻塞合并 | 必须修复：安全漏洞、数据丢失、功能破坏 |
| **[IMPORTANT]** | 应修复 | 必须修复：潜在 bug、架构问题、测试缺口 |
| **[SUGGESTION]** | 建议改进 | 值得考虑但不必强制修复 |
| **[NIT]** | 轻微偏好 | 作者可忽略的风格偏好 |

### 置信度

| 标签 | 含义 | 决策规则 |
|------|------|---------|
| **[CONFIDENCE:HIGH]** | 有直接证据，确定是问题 | 正常处理 |
| **[CONFIDENCE:MEDIUM]** | 间接证据或推理，大概率有问题 | 正常处理但注明推理路径 |
| **[CONFIDENCE:LOW]** | 可疑但无法确认 | **移到 Open Questions，不阻断通过** |

### 置信度规则

- **LOW 置信度的 CRITICAL/IMPORTANT 发现不阻断 pipeline。** 它们进入 Open Questions 区，提醒注意但不阻塞。
- 只有 HIGH 置信度的 CRITICAL/IMPORTANT 发现才构成 BLOCKED 判定。
- 审查员不得将所有发现默认标为 HIGH——必须能够说出"如果我错了，什么证据能推翻这个发现"。

### 发现格式

```
- [CRITICAL] [CONFIDENCE:HIGH] 标题
  文件: path:line
  证据: <具体代码片段或命令输出>
  如果我错了: <什么证据能推翻此发现>
  影响: <对系统的影响>
  建议: <修复建议>
```

**注意：** 不要把风格偏好（NIT）伪装成缺陷（IMPORTANT）。只有影响正确性、安全性、可维护性的才算缺陷。

## 依赖审查

引入新依赖时检查：

1. **必要性**：现有技术栈能否解决？是否为了一个函数引入整个库？
2. **体积**：对 bundle size 有多大影响？
3. **活跃度**：最近 3 个月有 commit 吗？issue 有人回复吗？
4. **安全性**：`npm audit` / `pip audit` 是否有已知漏洞？
5. **许可证**：是否与项目许可证兼容？

## 死代码处理

发现疑似死代码时的标准做法：

```
⚠️ 疑似死代码：
- function legacyFormat() in src/utils/date.ts
  → 搜索整个仓库无引用
  → 请确认：这些可以安全删除吗？
```

**不要直接删除。** 先报告，让作者确认。可能是：
- 即将上线的功能预留
- 配置项/约定的一部分
- 外部脚本的依赖

## 审查输出格式

```
## Findings
- [P1/CRITICAL] [CONFIDENCE:HIGH] 标题
  文件: path:line
  证据: <具体代码片段或命令输出>
  如果我错了: <什么证据能推翻此发现>
  影响: <对系统的影响>
  建议: <修复建议>

- [P2/IMPORTANT] [CONFIDENCE:MEDIUM] 标题
  ...

## Open Questions
- [LOW CONFIDENCE] <疑似问题，置信度不足，不阻断>
- <需要作者澄清的问题>

## Test Gaps
<未覆盖的测试场景>

## Residual Risk
<审查后仍存在的风险>

## 多视角审查记录
- 安全工程师视角: <发现数>
- 新人视角: <发现数>
- 运维工程师视角: <发现数>
```

没有发现问题时，明确写"未发现阻塞问题"，并列出未覆盖的验证范围。

## 多视角审查协议

审查时至少从以下三个视角切换审视（不仅依赖默认视角）：

### 代码审查三视角

| 视角 | 角色 | 核心问题 |
|------|------|---------|
| **安全工程师** | 攻击者思维 | 哪些信任边界被跨越？什么输入未验证？什么可被利用？ |
| **新人** | 零上下文思维 | 不熟悉此代码库的人能理解吗？哪些上下文假设没有明说？ |
| **运维工程师** | 故障思维 | 大规模运行时会发生什么？依赖故障时如何降级？ |

### 计划审查三视角（适用于 Gate C 计划审查）

| 视角 | 角色 | 核心问题 |
|------|------|---------|
| **执行者** | "我能否只靠写下来的内容完成每一步？" | 步骤是否足够具体？缺失什么信息？ |
| **利益相关方** | "这解决了正确的问题吗？" | 方案是否匹配问题？是否有更简单的替代方案？ |
| **怀疑者** | "这个方案失败的最强论据是什么？" | 最大风险是什么？哪些假设最脆弱？ |

### 执行规则

1. 每个视角至少产生 1 条独立的发现或确认声明
2. 视角切换后发现的问题单独标注来源视角
3. 视角切换不是可选项——跳过视角 = 审查不完整

## 自查与现实主义校验

审查完成后、输出前，必须执行以下校验：

### 步骤 1：置信度自检

对每条 CRITICAL/IMPORTANT 发现：
- 这条发现作者能立即反驳吗？（如果能，降低置信度或删除）
- 这是 FLAW（缺陷）还是 PREFERENCE（偏好）？（偏好降级到 SUGGESTION/NIT）
- 如果我错了，什么具体证据能推翻？（写进"如果我错了"字段）

### 步骤 2：现实主义压力测试

对每条严重发现：
- **真实最坏情况是什么？**（不是理论最大值，是实际能发生的）
- **有哪些缓解因素？**（现有测试、部署门禁、监控告警、功能开关）
- **多快能检测到？**（立即爆炸 vs 静默数据损坏 vs 下一版本才发现）
- **我是否因审查压力而抬高了严重度？**（"我要找出问题" → 把可疑当确定）

### 步骤 3：重新校准

- 如果真实影响可控 → 降低严重度（含"Mitigated by: ..."说明）
- 如果是 PREFERENCE → 降级到 SUGGESTION/NIT
- 如果置信度 LOW → 移到 Open Questions
- **绝不降级：** 数据丢失、安全漏洞、财务影响的发现（即使置信度 LOW，保留在 Open Questions 而非删除）

## 常见借口

| 合理化借口 | 现实 |
|-----------|------|
| "代码能跑，审查何必较真" | 能跑的代码如果不可读、不安全、架构错了，就是在制造复利债务。 |
| "我自己写的，我知道没问题" | 作者对自己的假设有盲区。每段代码都需要另一双眼睛。 |
| "AI 生成的代码应该没问题" | AI 代码需要更严格的审查——自信但可能出错，合理但不一定正确。 |
| "测试通过了，就是好的" | 测试是必要但不充分的。测试不检查架构问题、安全问题、可读性问题。 |
| "这个 PR 太大了，但就是一次逻辑变更" | 超过 300 行的即使是单次变更也要求拆分。大 PR 隐藏 bug。 |
| "没发现问题，就是通过了" | 明确写"未发现阻塞问题"。列出未覆盖的验证范围。不把"没测"说成"没问题"。 |

## 红线——停下来，按流程走

如果你发现自己在想：
- "这段改得挺对的，不用细看了"
- "没有发现问题"（但没有逐条过五轴检查）
- "这个 finding 太小了，不值得写"
- "审查太费时间，简单过一下就行"
- "我和作者关系好，不用太严格"

**以上这些都意味着：审查不完整。回到五轴框架，逐项评估。**

## 验证清单

审查完成后确认：

- [ ] 五轴（正确性/可读性/架构/安全/性能）均已评估
- [ ] 每条 finding 有文件路径和行号作为证据
- [ ] 严重度分级正确（CRITICAL/IMPORTANT/SUGGESTION/NIT）
- [ ] **置信度已标注（HIGH/MEDIUM/LOW）且 LOW 置信度严重发现已移入 Open Questions**
- [ ] **发现/过滤分离——未预过滤低严重度或不确定的发现**
- [ ] **多视角审查已完成（安全工程师/新人/运维工程师）**
- [ ] **自查校验完成——FLAW vs PREFERENCE 区分、现实主义压力测试、重新校准**
- [ ] 变更规模已评估（超过 300 行已建议拆分）
- [ ] 新引入的依赖已审查
- [ ] 未覆盖的验证范围已明确列出
- [ ] 审查结果已按标准格式输出（含置信度和多视角记录）
