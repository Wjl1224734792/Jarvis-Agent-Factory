# 需求探索

立即执行以下初始化步骤：

## 步骤 0：加载技能 + 注册引擎
```
Skill("behavioral-guidelines")
Skill("idea-refine")
Skill("spec-driven-development")
```

**引擎会话注册**（硬约束——引擎确保探索操作按 Gate 权限执行）：
- `mcp__jarvis-engine__session_join({ platform: "claude", pipeline_type: "explore", task_name: "需求探索: <需求主题>" })`
- 每个 Gate 开始前调用 `mcp__jarvis-engine__pipeline_guide()` 获取当前 Gate 允许的操作
- 写文档前调用 `mcp__jarvis-engine__gate_check({ operation: "write_doc" })`

代码注释语言：遵从 `behavioral-guidelines` 准则 5（注释语言约定）。

> **核心理念**：通过苏格拉底式提问，将模糊的想法逐步澄清为结构化的需求规格。不预设答案，不替用户做决策，而是通过提问帮助用户自己想清楚。

---

## X0：问题澄清

**Gate 检查条件**：问题空间已澄清，含核心痛点+上下文约束

### 步骤

1. **理解初始想法**——解析用户的输入，识别：
   - 用户想要达成什么？（目标）
   - 为什么现在需要这个？（动机）
   - 当前有什么痛点？（问题）

2. **探索现有上下文**（棕地项目）——spawn `code-explore-expert` 了解：
   - 当前项目的技术栈和架构
   - 相关的现有模块和代码
   - 可能受影响的区域

3. **苏格拉底式提问**——使用 `AskUserQuestion` 逐轮澄清（每轮 1-2 个问题）：
   - **目标层**：这个功能最终要达成什么业务目标？
   - **用户层**：谁会使用这个功能？在什么场景下使用？
   - **约束层**：有哪些已知的技术约束、时间约束、资源约束？
   - **边界层**：什么明确不在范围内？

4. **问题空间总结**——确认理解正确后，记录：
   ```markdown
   # 问题空间
   ## 核心痛点
   ## 目标用户
   ## 业务目标
   ## 约束条件
   ## 明确排除
   ```

5. 输出 `docs/YYYY-MM-DD/requirements/problem-space.md`

**引擎推进**：`mcp__jarvis-engine__advance_gate({ gate: "X1" })`

---

## X1：场景挖掘

**Gate 检查条件**：使用场景清单已产出，含主流程+边缘场景

### 步骤

1. **主流程场景**（Happy Path）——至少 3 个：
   ```
   | 场景ID | 场景描述 | 前置条件 | 操作步骤 | 预期结果 |
   |--------|---------|---------|---------|---------|
   | SC-01  | 用户首次使用核心功能 | 已登录 | 1.打开页面 2.点击... | 看到... |
   | SC-02  | ... | ... | ... | ... |
   ```

2. **边缘场景**（Edge Cases）——至少覆盖：
   - 空数据/首次使用
   - 大量数据/极限情况
   - 网络异常/超时
   - 权限不足/未登录
   - 并发操作/重复提交

3. **用户旅程映射**（可选，复杂功能时使用）：
   ```
   用户目标 → 触发动作 → 系统响应 → 用户反馈 → 目标达成
   ```

4. 输出 `docs/YYYY-MM-DD/requirements/scenario-inventory.md`

**引擎推进**：`mcp__jarvis-engine__advance_gate({ gate: "X2" })`

---

## X2：需求收敛

**Gate 检查条件**：需求优先级矩阵已产出，含MVP范围定义

### 步骤

1. **需求清单**——从 X1 的场景中提取功能需求：
   | 需求ID | 需求描述 | 来源场景 | 优先级 |
   |--------|---------|---------|--------|
   | FR-01  | ...     | SC-01   | P0     |

2. **优先级矩阵**（MoSCoW 法）：
   ```
   | 优先级 | 含义 | 需求数 | 说明 |
   |--------|------|--------|------|
   | P0 (Must Have) | 必须有，否则产品不可用 | __ | MVP核心 |
   | P1 (Should Have) | 应该有，影响用户体验 | __ | MVP+ |
   | P2 (Could Have) | 可以有，锦上添花 | __ | 后续迭代 |
   | P3 (Won't Have) | 明确不做（本次） | __ | 记录以防遗忘 |
   ```

3. **MVP 范围定义**：
   - MVP 包含：P0 需求清单
   - MVP 验收标准：用户能完成核心流程的最小闭环
   - 预计迭代周期

4. 输出 `docs/YYYY-MM-DD/requirements/requirements-priority.md`

**引擎推进**：`mcp__jarvis-engine__advance_gate({ gate: "X3" })`

## 与 `/jarvis` 的衔接

需求探索完成后，可直接进入实现阶段：

```
/explore "模糊想法"
  → X0→X1→X2→X3 需求规格（结构化文档）
  → /jarvis "实现 FR-01 FR-02..."  （启动全流程编排）
```

也可以将需求规格作为 `/task-ddd` 或 `/task-bdd` 的输入。

---

## 红线
- 跳过问题澄清直接写需求（没想清楚就写规格必然返工）
- 替用户做决策（探索者的职责是提问和呈现选项，不是代替决策）
- 一次性问太多问题（每轮至多 2 个问题，确保深入而非广撒网）
- 在 X0 阶段就承诺技术方案（先搞清楚"做什么"，再想"怎么做"）
- 需求规格写得像技术设计（需求文档描述"什么"，不描述"如何实现"）
- 忽略边缘场景（Happy Path 只占 20% 的工作量，Bug 都在边缘场景里）
