## RS0：课题定义

**Gate 检查条件**：研究课题定义文档已产出，含研究范围+方法论+成功标准

**🔴 前置约束**：若用户输入模糊（课题不明确/范围不清），必须先使用 `AskUserQuestion` 与用户确认研究课题和范围，确认后再产出课题定义文档。跳过确认直接写文档 → 违反红线。

### 步骤
0. **确认研究课题**——若未明确，`AskUserQuestion` 确认：
   - 要研究的核心问题是什么？
   - 研究的范围和深度边界？
   - 预期的产出形式（报告/方案/对比分析）？

1. 明确研究课题和核心问题：
   - 要回答什么问题？
   - 为什么这个研究很重要？
   - 预期的研究成果是什么？

2. 定义研究范围与边界：
   - 哪些方面在范围内？
   - 哪些明确排除？
   - 研究的时间/深度边界？

3. 选择研究方法论：
   | 研究类型 | 方法论 | 适用场景 |
   |---------|--------|---------|
   | 技术调研 | 代码库探索 + 文档分析 + 网络调研 | 了解现有实现、技术栈 |
   | 方案评估 | 原型验证 + 基准测试 + 对比分析 | 技术选型、架构决策 |
   | 问题诊断 | 日志分析 + 根因追溯 + 复现验证 | Bug根因、性能瓶颈 |
   | 趋势分析 | 文献综述 + 社区调研 + 专家访谈 | 技术方向、最佳实践 |

4. 定义成功标准：
   - 回答了哪些核心问题？
   - 提供了哪些可执行的建议？
   - 遗留了哪些待进一步研究的问题？

5. 输出 `.jarvis/YYYY-MM-DD/research/research-scope.md`

**引擎推进**：`mcp__jarvis-engine__advance_gate({ gate: "RS1" })`

## RS1：信息收集

**Gate 检查条件**：信息收集清单已产出，含代码库/文档/网络资源索引

### 步骤
1. **代码库探索**——spawn `code-explore-expert` 扫描相关代码：
   - 识别关键模块、文件和函数
   - 分析依赖关系和调用链
   - 评估代码质量和潜在风险点

2. **文档调研**——spawn `external-resource-expert` 搜索外部资源：
   - 官方文档和 API 参考
   - 社区最佳实践和案例
   - 相关论文和技术博客

3. **网络搜索**（按需）——使用 WebSearch/WebFetch：
   - 搜索同类问题的解决方案
   - 收集性能基准数据
   - 对比不同技术方案

4. 汇总信息收集清单到 `.jarvis/YYYY-MM-DD/research/information-inventory.md`：
   ```markdown
   # 信息收集清单
   ## 代码库发现
   | 文件/模块 | 关键发现 | 相关度 |
   ## 文档/资源
   | 资源链接 | 关键内容 | 可信度 |
   ## 初步假设
   | 假设 | 支撑证据 | 待验证 |
   ```

**引擎推进**：`mcp__jarvis-engine__advance_gate({ gate: "RS2" })`

## RS2：深度分析

**Gate 检查条件**：多维度分析已产出，含竞争假设+证据矩阵

### 步骤
1. 从 RS1 的初步假设出发，提出**竞争假设**（至少 2-3 个）：
   ```
   | 假设 | 描述 | 支撑证据 | 反驳证据 | 置信度 |
   |------|------|---------|---------|--------|
   | H1  | ...  | ...     | ...     | 高/中/低 |
   | H2  | ...  | ...     | ...     | 高/中/低 |
   ```

2. 多维度分析（至少 4 个维度）：
   | 维度 | 权重 | 分析要点 |
   |------|------|---------|
   | 技术可行性 | 30% | 实现复杂度、技术成熟度、可维护性 |
   | 性能影响 | 25% | 延迟、吞吐量、资源消耗 |
   | 安全可靠 | 20% | 攻击面、容错性、数据一致性 |
   | 成本收益 | 25% | 开发成本、运维成本、长期收益 |

3. 证据矩阵——每个维度对每个假设进行评分（1-5 分）：
   ```
   | 维度 | H1 | H2 | H3 |
   |------|----|----|-----|
   | 技术可行性 | _/5 | _/5 | _/5 |
   | 性能影响 | _/5 | _/5 | _/5 |
   | 安全可靠 | _/5 | _/5 | _/5 |
   | 成本收益 | _/5 | _/5 | _/5 |
   | **加权总分** | __ | __ | __ |
   ```

4. 输出 `.jarvis/YYYY-MM-DD/research/deep-analysis.md`

**引擎推进**：`mcp__jarvis-engine__advance_gate({ gate: "RS3" })`

## RS3：假设验证

**Gate 检查条件**：假设验证已执行，证据充分支持或否定各假设

### 步骤
1. 对 RS2 得分最高的假设（排名前 2）进行**深度验证**：
   - 收集更多针对性证据
   - 执行最小化验证实验（如编写原型代码、运行基准测试）
   - spawn 测试 Agent 验证技术假设（如性能测试、安全扫描）

2. 对每个假设做出明确结论：
   ```
   | 假设 | 验证方式 | 验证结果 | 结论 |
   |------|---------|---------|------|
   | H1  | 原型+基准 | 响应时间 < 50ms | ✅ 成立 |
   | H2  | 文档调研 | API不兼容 | ❌ 否定 |
   | H3  | 代码审查 | 复杂度高但可行 | ⚠️ 条件成立 |
   ```

3. 输出 `.jarvis/YYYY-MM-DD/research/hypothesis-validation.md`

**引擎推进**：`mcp__jarvis-engine__advance_gate({ gate: "RS4" })`

|------|------|----|----|-----|

## 红线
- **跳过课题确认直接写文档**（RS0 必须先与用户确认研究课题和范围，未确认不得产出课题定义文档）
- 未定义研究范围就收集信息（先锁定问题，再找答案）
- 仅凭单一来源下结论（至少交叉验证 2 个来源）
- 假设未经验证就写入报告（区分事实和推测）
- 忽略反面证据（如实报告不支持假设的证据）
- 研究报告缺少可执行建议（研究不能只描述，必须指向行动）
- 研究范围不断膨胀（"这个也顺便看看"——保持聚焦）
