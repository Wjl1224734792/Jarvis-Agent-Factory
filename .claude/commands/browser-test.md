# 浏览器自动化测试闭环

> 这是主动测试模式（编写用例 → 执行 → 修复）。如果你要修复一个已知 Bug 并需要浏览器复现，请使用 `/bug-fix` 命令。

立即执行以下步骤：

## 步骤 0：加载技能

```
Skill("behavioral-guidelines")
Skill("browser-use")
```

## 步骤 1：确认测试范围

如果用户未提供测试范围，先询问并确认：
- **目标 URL**：要测试的页面地址
- **功能范围**：要验证的具体功能（登录、注册、表单提交、页面渲染等）
- **关键用户路径**：核心操作流程
- **已知风险点**：最近修改过的部分、历史上出过 Bug 的区域

## 步骤 2：编写测试用例清单

输出到 `docs/testing/YYYY-MM-DD-<topic>-browser-test-cases.md`。

每条用例包含：
- 用例编号（TC-001 起）
- 前置条件
- 操作步骤（用 browser-use CLI 可执行的明确指令）
- 预期结果
- 验证方式（截图/状态检查/文本匹配）
- 优先级（P0 阻塞 / P1 重要 / P2 次要）

## 步骤 3：逐条执行测试

按优先级从高到低逐条执行。每条用例：

```
1. browser-use open <url>          # 导航到测试页面
2. browser-use state               # 获取元素索引
3. browser-use click <index>       # 执行交互
4. browser-use input <index> "text" # 输入内容
5. browser-use screenshot          # 截图留证
```

执行规则：
- 每条用例执行前后截图
- 失败立即记录，截图保存失败状态
- 前置条件不满足则标记"跳过"，写明原因
- 遇到 browser-use 错误先 `browser-use close` 清理再重试

## 步骤 4：汇总测试报告

输出到 `docs/testing/YYYY-MM-DD-<topic>-browser-test-report.md`。

包含：
- 测试概览（通过/失败/跳过/通过率）
- 每条用例详细结果（含截图路径）
- 失败用例根因分析
- 所有截图引用

## 步骤 5：闭环——失败驱动修复

### 如果全部通过 → 闭环完成，输出最终报告。

### 如果存在失败 → 启动修复闭环：

1. **输出 Browser Test Findings**（失败用例 + 截图证据 + 修复建议）
2. **触发修复**：将 findings 提交给 `/review-fix` 闭环或调用对应实现 agent
3. **代码质量验证**：修复后必须先通过 Lint + Type-check + Build（三项全部通过才能继续）
4. **重测验证**：修复完成后，仅重跑失败用例（不重跑全部）
5. **更新报告**：标注修复状态（已修复/已验证/仍失败）
5. **全部通过后** → 闭环完成

```
测试 ──全部通过──→ ✅ 闭环完成
  │
  └──存在失败──→ Browser Test Findings
                    │
                    └──→ /review-fix（审查修复）──→ 重测失败用例
                                                      │
                                                      ├── 通过 ──→ ✅ 闭环完成
                                                      └── 仍失败 ──→ 再次修复（最多 2 轮）
```

**最多 2 轮修复-重测循环**，第 3 轮仍失败则标记为 BLOCKED 并上报。

## 红线

- 不写用例直接操作浏览器（缺少可追溯的测试计划）
- 测试失败不截图（缺少证据）
- 跳过修复闭环（失败用例不驱动修复，测试失去意义）
- 在浏览器中执行破坏性操作（删除生产数据、发起真实支付等）
