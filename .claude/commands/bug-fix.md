# Bug 修复闭环（浏览器复现 → 修复 → 验证）

立即执行以下步骤：

## 步骤 0：加载技能

```
Skill("behavioral-guidelines")
Skill("browser-use")
```

## 步骤 1：收集 Bug 信息

向用户确认以下信息（如未提供）：
- **Bug 描述**：什么问题？预期行为 vs 实际行为
- **影响页面/URL**：哪个页面出问题？
- **复现步骤**：具体操作流程（1→2→3）
- **环境信息**：浏览器、设备、登录状态
- **严重程度**：P0 阻塞 / P1 功能受损 / P2 轻微

输出 Bug Report 摘要，等待用户确认。

## 步骤 2：浏览器复现——捕获证据

使用 browser-use CLI 严格按复现步骤操作：

```
# 1. 导航到目标页面
browser-use open <URL>

# 2. 获取页面状态
browser-use state

# 3. 逐步执行复现操作
browser-use click <index>
browser-use input <index> "<text>"
browser-use scroll down

# 4. 在异常状态截图
browser-use screenshot

# 5. 检查控制台错误
# browser-use console  # 如可用
```

**关键操作：**
- 每步操作后 `browser-use state` 检查页面变化
- **异常发生时立即截图**（错误提示、空白页、布局错乱、数据不显示等）
- 尝试至少 1 个变体操作（如不同的输入值、不同的操作顺序），确认 Bug 的触发边界
- 记录完整的浏览器操作日志

**复现成功 → 进入步骤 3**
**复现失败（Bug 未出现）→ 回问用户补充复现条件（特定数据、权限、浏览器版本等），最多 2 轮**

## 步骤 3：定位根因

基于复现证据，追踪代码根因：

1. **从页面反查代码**：根据出问题的页面元素、交互行为，定位前端组件文件
2. **追踪数据流**：检查状态管理、API 调用、数据处理逻辑
3. **检查边界条件**：空值、未定义、异常数据、竞态条件

输出根因分析：

```
## Root Cause Analysis
- 故障文件：<文件路径>:<行号>
- 故障类型：逻辑错误 / 状态异常 / API 数据问题 / 样式 Bug / 兼容性
- 直接原因：<一句话>
- 影响范围：<受影响的组件/页面/功能>
- 修复方案：<具体的代码修改方案>
```

## 步骤 4：修复代码

按最小改动原则修复：

- 只改必须改的文件
- 遵循现有代码风格
- 不引入新依赖或重构无关代码
- 修复后立即自查：改动是否正确、是否影响其他功能

## 步骤 5：代码质量验证——Lint + Type-check + Build

修复完成后，必须执行代码质量检查：

```
1. Lint: npm run lint（或 eslint/ruff/golangci-lint 等）  — 必须 0 error
2. Type-check: tsc --noEmit（或 mypy/cargo check 等）    — 必须 0 error
3. Build: npm run build（或 cargo build/go build 等）     — 必须成功
```

**三项全部通过** → 进入步骤 6（浏览器验证）
**任一失败** → 修复导致失败的代码 → 重新执行失败的检查项

## 步骤 6：浏览器验证——确认修复

用**完全相同的复现步骤**重新执行浏览器操作：

```
1. browser-use open <URL>
2. browser-use state
3. 重复步骤 2 的复现操作
4. browser-use screenshot  # 对比修复前后
5. 验证 Bug 不再出现
```

**验证通过标准：**
- 原异常操作现在产生预期结果 ✓
- 截图对比：修复前（异常）→ 修复后（正常）✓
- 未引入新问题（相关功能抽查通过）✓

**验证通过 → 进入步骤 7**
**验证失败 → 回到步骤 3，重新分析（最多 2 轮回退）**

## 步骤 7：关闭 Bug

输出 `docs/bug-fix/YYYY-MM-DD-<bug-title>-bug-fix-report.md`：

```markdown
# Bug 修复报告

## Bug 信息
- 描述、严重度、影响页面

## 复现证据
- 复现步骤、修复前截图、控制台错误

## 根因分析
- 故障文件:行号、故障类型、直接原因

## 修复内容
- 修改文件、变更摘要

## 验证证据
- 相同步骤复测结果、修复后截图、修复前后对比

## 回归风险
- 低 / 中 / 高
- 建议补充的测试用例
```

## 闭环图示

```
Bug Report ──→ 浏览器复现 ──→ 截图/证据 ──→ 定位根因
                                                  │
                                                  ▼
                                             修复代码
                                                  │
                                                  ▼
                              Lint + Type-check + Build
                               │                │
                          三项通过          任一失败
                               │                │
                               ▼                ▼
                         浏览器验证         回到修复代码
                         （相同步骤复测）
                          │           │
                     Bug 不再出现  Bug 仍存在
                          │           │
                          ▼           ▼
                      ✅ 关闭     回到定位根因
                                 （最多 2 轮回退）
```

**最多 2 轮回退**，第 3 轮仍失败则标记为 BLOCKED，不修改代码，输出已有证据和分析，请求人工介入。

## 红线

- 不复现就直接改代码（没有证据的修复 = 猜测）
- 复现成功不截图（丢失关键证据）
- 不定位根因直接打补丁（治标不治本）
- 修改代码后不用浏览器验证（无法确认修复生效）
- 修复范围超出 Bug 本身（夹带无关改动）
- 在浏览器中执行破坏性操作（删除数据、发起支付等）
