# 通用编程规范与指南

## 1. 开发环境配置

- **语言**：项目文档、注释及沟通强制使用**中文**。
- **终端**：Windows 系统下推荐使用 **PowerShell**。
- **系统**：主要开发与运行环境为 **Windows**,生产运行环境为 **Linux**。

## 2. 代码规范

### 2.1 注释规范 (JSDoc/TSDoc) 📝

- **原则**：拒绝冗余，关注关键逻辑与边界条件。
- **格式**：JS 项目使用 JSDoc，TS 项目使用 TSDoc。必须包含 `@param`, `@returns`, `@throws`。

### 2.2 嵌套层级限制 🎯

- **限制**：推荐 1-3 层，最多 4 层，禁止超过 4 层。
- **优化手段**：卫语句、函数提取。

### 2.3 数组操作规范 🔄

- **禁止（改变原数组）**：`push`, `pop`, `shift`, `splice`, `sort`, `reverse`（空数组初始化除外）。
- **推荐（返回新数组/不改变原数组）**：`map`, `filter`, `reduce`, `find`, `some`, `every`, ES2023+ (`toSpliced`, `toSorted`, `with`)，扩展运算符。

### 2.4 ES6 模块化 📦

- **导出/导入**：推荐命名导出；功能单一模块可使用默认导出。
- **路径**：使用别名路径（如 `@/services`），避免深层相对路径。
- **原则**：高内聚，严禁循环依赖。

### 2.5 逻辑与设计 🏗️

- **设计原则**：SOLID、DRY、KISS。
- **条件优化**：3 个以上分支使用对象映射/Map；强制 `===`；使用 `??` 和 `?.`。
- **箭头函数**：禁止在对象方法和类方法中使用。
- **Promise**：善用 `Promise.all`/`allSettled`；避免无意义的 `return await`。

### 2.6 数据库外键规范 🗄️

- **原则**：禁止数据库物理外键约束，通过 ORM 定义逻辑外键。
- **理由**：性能、扩展性、迁移维护、错误处理。
- **实践**：ORM 中定义关联但禁用外键约束（如 `createForeignKeyConstraints: false`）；应用层保证一致性。

### 2.7 Tailwind CSS 使用规范 🎨

- **约束**：只能使用内联类名，禁止在 CSS 文件中通过 `@apply` 组合工具类。
- **理由**：可维护性、重构安全、构建性能、团队协作一致。
- **✅ 正确**：直接在 `className`/`class` 中写完整工具类（如 `px-4 py-2 bg-blue-500`）。
- **❌ 错误**：在 `.css` 文件中使用 `@apply` 定义自定义类。
- **例外**：需团队评审并注释说明（如第三方组件库强制要求单一类名）。

## 3. 异常与质量保证 ✅

### 3.1 例外处理

- **允许场景**：算法实现（递归）、经测试证明的性能关键代码。
- **要求**：必须注释说明原因，并经团队评审。

### 3.2 强制检查项 (Commit 前)

1. **层级**：嵌套不超过 4 层。
2. **模块**：无循环依赖，路径清晰。
3. **语法**：多分支用 Map，判空用 `?.`，全等用 `===`。
4. **异步**：Promise 使用正确，包含错误处理。
5. **数据库**：无物理外键约束，ORM 层禁用外键。
6. **样式**：无 `@apply` 指令，所有 Tailwind 类均为内联类名。

## 4. 沟通风格 💬

### 4.1 交互原则

- **友好表达**：适当使用 Emoji (🎉, 👍, ⚠️)。
- **清晰对比**：优先展示 ❌ 错误示例 vs ✅ 正确示例。
- **正向反馈**：肯定正确，提示风险。

### 4.2 解答与交流

- **结构化响应**：问题分析 -> 解决方案 -> 最佳实践 -> 注意事项。
- **鼓励提问**：耐心解释复杂概念。
- **实用导向**：提供可落地代码，避免过度抽象。
