# 通用编程规范与指南

## 1. 开发环境配置

- **语言**：项目文档、注释及沟通强制使用**中文**。
- **终端**：Windows 系统下推荐使用 **PowerShell**。
- **系统**：主要开发与运行环境为 **Windows**。

## 2. 代码规范

### 2.1 注释规范 (JSDoc/TSDoc) 📝

- **原则**：拒绝冗余，关注关键逻辑与边界条件。
- **格式**：JS 项目使用 JSDoc，TS 项目使用 TSDoc。必须包含 `@param`, `@returns`, `@throws`。

### 2.2 嵌套层级限制 🎯

- **限制**：推荐 1-3 层，最多 4 层，禁止超过 4 层。
- **优化手段**：
  - **卫语句 (Guard Clauses)**：优先处理异常/边界并返回，减少 `else`。
  - **函数提取**：将复杂条件或逻辑抽离为独立函数。

### 2.3 数组操作规范 🔄

- **禁止 (改变原数组)**：`push`, `pop`, `shift`, `splice`, `sort`, `reverse` (空数组初始化除外)。
- **推荐 (返回新数组/不改变原数组)**：
  - **迭代/转换**：`map`, `filter`, `reduce`, `find`, `some`, `every`。
  - **ES2023+**：`toSpliced`, `toSorted`, `with`。
  - **扩展运算符**：使用 `[...arr]` 或 `{...obj}` 进行复制、合并、更新。

### 2.4 ES6 模块化 📦

- **导出/导入**：推荐**命名导出** (`export const`)；功能单一模块可使用默认导出。
- **路径**：使用清晰的别名路径 (如 `@/services`)，避免深层相对路径 (`../../../`)。
- **原则**：按功能高内聚划分，**严禁循环依赖**。

### 2.5 逻辑与设计 🏗️

- **设计原则**：遵循 SOLID、DRY (不重复)、KISS (保持简单)。
- **条件优化**：
  - 3 个以上分支使用 **对象映射/Map** 替代 switch/else-if。
  - 强制使用 `===` 严禁 `==`。
  - 使用 `??` (空值合并) 和 `?.` (可选链) 简化判空。
- **箭头函数**：**禁止**在对象方法和类方法中使用 (因 `this` 绑定问题)。
- **Promise**：善用 `Promise.all`/`allSettled` 并行处理；避免无意义的 `return await`。

### 2.6 数据库外键规范 🗄️

- **原则**：禁止在数据库层面创建物理外键约束，应通过 ORM 层面定义逻辑外键关系。
- **理由**：
  - **性能**：物理外键在插入、更新、删除时会产生额外的约束检查，影响写入性能；在高并发分布式系统中，外键约束可能导致锁竞争和死锁。
  - **扩展性**：微服务架构中，数据库可能分库分表，物理外键难以跨库；逻辑外键更灵活，支持服务解耦。
  - **迁移与维护**：物理外键增加了数据库迁移的复杂度（如需要按依赖顺序导入数据）；逻辑外键使得数据迁移、回滚更加容易。
  - **错误处理**：物理外键引发的数据库错误难以在应用层优雅捕获和处理；逻辑外键可以在应用层通过事务、业务规则保证一致性，并提供友好的错误提示。
- **实践**：
  - 在 ORM（如 TypeORM、Sequelize、Prisma、Mongoose 等）中定义模型关联（如 `@ManyToOne`、`belongsTo`、`hasMany`），但不生成数据库外键约束（通常 ORM 提供选项禁用外键约束）。
  - 数据一致性由应用层通过事务、业务逻辑或最终一致性机制保证。
  - 在需要进行关联查询时，使用 ORM 的关联方法或手动 JOIN，但不依赖数据库的级联操作（如 `ON DELETE CASCADE`）。
  - 对于需要级联删除的场景，应在应用层显式处理（如使用事务），避免数据库隐式操作。
- **示例**：

**❌ 错误示例（在数据库中创建物理外键）**：
```sql
ALTER TABLE orders ADD CONSTRAINT fk_user FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE;
```
或通过 ORM 迁移文件显式添加外键约束。

**✅ 正确示例（在 ORM 层面定义逻辑外键）**：
```typescript
// TypeORM 示例：不生成物理外键
@Entity()
export class User {
  @PrimaryGeneratedColumn()
  id: number;

  @OneToMany(() => Order, order => order.user)
  orders: Order[];
}

@Entity()
export class Order {
  @PrimaryGeneratedColumn()
  id: number;

  @Column()
  userId: number; // 仅作为普通列，无物理外键

  @ManyToOne(() => User, user => user.orders, { createForeignKeyConstraints: false }) // 禁用外键约束
  user: User;
}
```

- **注意事项**：
  - 尽管没有物理外键，仍需确保数据完整性。可以通过数据库唯一约束、应用层事务、乐观锁等机制来保证。
  - 定期进行数据一致性校验，确保逻辑外键指向的记录存在。
  - 在 ORM 查询时，利用关联关系进行连接查询仍然可行，ORM 会基于逻辑外键生成 JOIN 语句。

## 3. 异常与质量保证 ✅

### 3.1 例外处理

- **允许场景**：算法实现 (递归)、经测试证明的性能关键代码。
- **要求**：必须添加详细注释说明原因，并经团队评审。

### 3.2 强制检查项 (Commit 前)

1.  **层级**：嵌套不超过 4 层。
2.  **模块**：无循环依赖，路径清晰。
3.  **语法**：多分支用 Map，判空用 `?.`，全等用 `===`。
4.  **异步**：Promise 使用正确，包含错误处理。
5.  **数据库**：无物理外键约束，所有关联在 ORM 层定义（`createForeignKeyConstraints: false` 或等效配置）。

## 4. 沟通风格 💬

### 4.1 交互原则

- **友好表达**：适当使用 Emoji (🎉, 👍, ⚠️) 增加亲和力。
- **清晰对比**：讨论问题时，优先展示 **"❌ 错误示例 vs ✅ 正确示例"**。
- **正向反馈**：对正确的理解给予肯定；对风险方案明确提示。

### 4.2 解答与交流

- **结构化响应**：问题分析 -> 解决方案 -> 最佳实践 -> 注意事项。
- **鼓励提问**：营造安全氛围，对复杂概念耐心解释。
- **实用导向**：提供可落地的代码，不做过度抽象。
