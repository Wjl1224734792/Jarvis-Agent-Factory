## 目的
通过提出关于系统、用户、工作流和优先级的自适应问题，引导产品经理创建用户故事地图——然后生成包含主干（活动）、用户任务和发布切片的二维地图。用此方法从扁平积压列表转向可视化的故事地图，传达整体图景、发现缺失功能，并实现有意义的发布规划——避免故事脱离整体系统叙述的"无上下文碎屑"。

这不是积压生成器——它是一种可视化沟通框架，按用户工作流（水平）和优先级（垂直）组织工作。

## 核心概念

### 什么是用户故事地图？

故事地图（Jeff Patton）以**两个维度**组织用户故事：

**水平轴（从左到右）：** 以叙述/工作流顺序排列的活动——是你向他人解释系统时的顺序

**垂直轴（从上到下）：** 每个活动内的优先级，最必要的任务在顶部

**结构：**
```
主干（顶部横跨的活动）
↓
用户任务（按优先级垂直向下排列）
↓
详细/验收标准（在底部）
```

### 核心原则

**主干：** 必要活动构成系统的结构核心——它们之间不相互排优先级；它们是叙述流。

**行走骨架：** 所有活动中优先级最高的任务构成最小可行产品——最小的端到端功能。

**肋骨：** 支撑任务在每个活动下垂直向下排列，通过位置表示优先级。

**从左到右、从上到下的构建策略：** 逐步构建所有主要功能，而非在开始另一个功能之前完整完成一个功能。

### 为什么这有效
- **可视化沟通：** 故事地图作为信息辐射器持续展示，保持对整体图景的关注
- **叙述结构：** 按用户工作流组织，而非技术架构
- **发布规划：** 水平切片揭示 MVP 和增量发布
- **差距发现：** 揭示扁平积压列表所掩盖的缺失功能

### 反模式（本技能不做什么）
- **不是甘特图：** 故事地图展示优先级，而非时间估算
- **不是技术架构：** 地图遵循用户工作流，而非系统分层（UI → API → DB）
- **不是项目计划：** 它是发现和沟通工具，不是进度表

### 何时使用
- 启动新产品或重要功能
- 重构现有积压列表（从扁平列表转向可视化地图）
- 在范围和优先级上对齐利益相关者
- 规划 MVP 或增量发布

### 何时不使用
- 单功能项目（故事地图过于复杂）
- 积压列表已被充分理解并排好优先级
- 技术重构工作（没有用户工作流可以映射）

## 应用

此交互技能提出**最多5个自适应问题**，每个步骤提供**3-4个列举选项**。

使用 `template.md` 获取引导议程和输出检查清单。

交互模式：当你需要逐步推进的流程，包含决策点的编号建议和常规问题的快捷选择选项时，搭配使用 `skills/workshop-facilitation/SKILL.md`。如果用户要求一次性输出，跳过多次引导流程。

### 步骤0：收集上下文（提问之前）

**智能体建议：**

创建故事地图前，先收集上下文：

**产品/功能上下文：**
- 你正在映射什么系统或功能？
- 产品概念、PRD 草稿或现有积压列表
- 网站文案、定位材料或用户流程
- 现有用户故事（如果从扁平积压列表转换中）

**用户上下文：**
- 目标画像或用户细分
- 用户研究、访谈或旅程地图
- Jobs-to-be-Done 或问题陈述

**你可以直接粘贴这些内容，或简要描述系统。**

---

### 问题1：定义范围

**智能体提问：**
"你在映射什么？（范围是什么？）"

**提供4个列举选项：**

1. **整个产品** —— "从发现到完成的完整端到端系统"（常见于新产品或全量重写）
2. **重要功能区域** —— "更大产品内的特定工作流（如'新手引导''结账''报表'）"（常见于功能发布）
3. **用户旅程** —— "特定的用户目标或 Jobs-to-be-Done（如'雇佣承包商''报税'）"（常见于 JTBD 驱动的映射）
4. **重新设计/重构** —— "正在重建或简化的现有产品/功能"（常见于遗留系统现代化）

**或描述你的具体范围。**

**用户回复：** [选择或自定义]

---

### 问题2：识别用户/画像

**智能体提问：**
"这个地图的主要用户是谁？（列出画像或用户细分。）"

**提供4个列举选项：**

1. **单一画像** —— "一种主要用户类型（如'小企业主'）"（简化映射，适合 MVP）
2. **多个画像，共享工作流** —— "不同用户类型，相同的核心活动（如'买家'和'卖家'都浏览列表）"（常见于交易平台）
3. **多个画像，不同工作流** —— "不同用户类型，有不同工作流（如'管理员' vs. '最终用户'）"（需要单独地图或泳道）
4. **组织内的角色** —— "不同工作职能（如'PM''设计师''工程师'）"（常见于内部工具）

**或描述你的用户。**

**自适应：** 使用步骤0中提供的上下文中的画像（proto-persona、JTBD 等）

**用户回复：** [选择或自定义]

---

### 问题3：生成主干（活动）

**智能体说：**
"让我们构建主干——用户为实现其目标而执行的活动叙述流。"

**智能体基于范围（Q1）和用户（Q2）生成5-8个活动**，按从左到右的工作流顺序排列。

**示例（如果范围 = "电商结账"）：**

```
主干活动（从左到右）：

1. 浏览商品
2. 加入购物车
3. 查看购物车
4. 输入配送信息
5. 输入支付信息
6. 确认订单
7. 接收确认
```

**智能体提问：**
"这个主干是否捕获了完整的工作流？我们应该添加、删除还是重新排序活动？"

**用户回复：** [确认、修改或添加自定义活动]

---

### 问题4：生成用户任务（在每个活动下）

**智能体说：**
"现在让我们在每个活动下添加用户任务，按优先级组织（顶部 = 必须有，底部 = 锦上添花）。"

**智能体为每个活动生成3-5个用户任务**，按优先级垂直排列。

**示例（对于活动2："加入购物车"）：**

```
加入购物车（活动）
├─ 添加单个商品到购物车（必须有，行走骨架）
├─ 调整数量（必须有）
├─ 一次性添加多个商品（应该有）
├─ 稍后保存商品（锦上添花）
└─ 添加礼品包装（锦上添花）
```

**智能体对所有主干活动重复此操作**，展示完整地图。

**智能体提问：**
"这捕获了关键任务吗？优先级是否正确（顶部 = MVP，底部 = 后续发布）？"

**用户回复：** [确认、修改或添加自定义任务]

---

### 问题5：识别发布切片（行走骨架 + 增量）

**智能体说：**
"让我们通过在地图上绘制水平线来定义发布切片。"

**智能体生成3个发布切片：**

**发布1（行走骨架）：** 所有活动中优先级最高的任务——最小端到端功能

**发布2（下一增量）：** 第二优先级任务，增强核心工作流

**发布3（打磨/扩展）：** 第三优先级任务（锦上添花、边界情况、优化）

**示例：**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
发布1（行走骨架）：
- 浏览商品（基础列表视图）
- 添加单个商品到购物车
- 查看购物车（订单项 + 合计）
- 输入配送信息（姓名、地址）
- 输入支付信息（仅信用卡）
- 确认订单（基础确认）
- 接收邮件确认
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
发布2（增强）：
- 商品筛选/搜索
- 调整购物车数量
- 稍后保存
- 多种配送选项
- 多种支付方式
- 订单追踪链接
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
发布3（打磨）：
- 商品推荐
- 游客结账
- 礼品包装
- 促销码
- 高级支付选项
- 购买后调研
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**智能体提问：**
"这些发布切片合理吗？我们需要调整范围或优先级吗？"

**用户回复：** [确认或修改]

## 示例

### 示例1：好的故事地图（电商结账）

**Q1回复：** "重要功能区域 —— 电商结账工作流"

**Q2回复：** "单一画像 —— 在线购物者"

**Q3 - 生成的主干：**
```

```

**Q4 - 生成的用户任务：**

```
浏览商品
├─ 查看商品列表（R1）
├─ 搜索/筛选（R2）
└─ 商品推荐（R3）

加入购物车
├─ 添加单个商品（R1）
├─ 调整数量（R2）
└─ 稍后保存（R3）

查看购物车
├─ 查看订单项 + 合计（R1）
├─ 应用促销码（R2）
└─ 估算运费（R3）

[...等等...]
```

**Q5 - 发布切片：**
- **发布1：** 行走骨架——基础流程，无额外内容
- **发布2：** 搜索、数量调整、促销码
- **发布3：** 推荐、游客结账、礼品选项

**为什么这有效：**
- 主干遵循用户叙述（而非技术分层）
- 行走骨架交付端到端价值
- 增量发布在不破坏核心流程的情况下增加复杂度

---

### 示例2：差的故事地图（技术分层）

**主干（错误）：**
```
UI层 → API层 → 数据库层 → 部署
```

**为什么这是失败的：**
- 不以用户为中心（用户不关心技术架构）
- 无法增量交付端到端价值
- 伪装成故事地图的瀑布式思维

**修复：**
- 按用户工作流映射："注册 → 配置设置 → 邀请团队 → 启动项目"
- 每个发布交付完整工作流，而非单一层

## 常见错误

### 错误1：伪装成故事地图的扁平积压列表
**症状：** 故事地图只是一个垂直列表，没有水平叙述

**后果：** 失去沟通优势；仍然是"无上下文碎屑"

**修复：** 强制水平结构——活动横跨顶部，任务垂直向下

---

### 错误2：以技术架构作为主干
**症状：** 主干 = "前端 → 后端 → 数据库"

**后果：** 不以用户为中心，无法增量交付价值

**修复：** 主干应遵循用户工作流，而非系统分层

---

### 错误3：功能完整的瀑布式
**症状：** 发布1 = "完整构建活动1"，发布2 = "完整构建活动2"

**后果：** 在所有活动完成之前没有端到端价值

**修复：** 行走骨架 = 横跨所有活动的薄切片，逐步增强

---

### 错误4：过早过详细
**症状：** 试图一次性映射每个边界情况和验收标准

**后果：** 分析瘫痪，丢失整体图景

**修复：** 从主干 + 高层任务开始，后续再优化

---

### 错误5：地图藏在工具里
**症状：** 故事地图存在于 Jira/Miro 中，从未展示出来

**后果：** 失去信息辐射器的价值

**修复：** 物理打印/张贴地图；让团队每天可见
