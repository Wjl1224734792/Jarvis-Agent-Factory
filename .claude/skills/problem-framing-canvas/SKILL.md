## 目的
通过跨越三个阶段提出结构化问题，引导产品经理完成 MITRE 问题框架画布流程：内省（审视你自己的假设和偏见）、外观（理解谁经历这个问题以及谁没有）、重构（将洞察综合为可操作的问题陈述和"我们如何能"问题）。用于确保你在跳入解决方案之前解决正确的问题——避免确认偏误、被忽视的利益相关者和解决方案优先思维。

这不是解决方案头脑风暴——它是一个问题框架工具，拓宽视角、挑战假设，并产生清晰、公平驱动的问题陈述。

## 核心概念

### 什么是 MITRE 问题框架画布？

问题框架画布（MITRE 创新工具包，v3）是一个结构化框架，帮助团队在提出解决方案之前全面探索问题空间。它分为**三个区域**：

1. **内省** —— 审视你自己的假设、偏见，以及你可能如何成为问题的一部分
2. **外观** —— 理解谁经历这个问题，谁从中受益，以及谁被排除在外
3. **重构** —— 将洞察综合为清晰、可操作的问题陈述和"我们如何能"问题

### 画布结构

```
┌─────────────────────────────────────────────────────────────────┐
│ 内省                                                             │
│ - 问题是什么？（症状）                                            │
│ - 我们为什么还没解决它？（新、难、低优先级等）                       │
│ - 我们如何成为问题的一部分？（假设、偏见）                          │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│ 外观                                                             │
│ - 谁经历这个问题？何时/何地/后果？                                 │
│ - 还有谁有这个问题？谁没有这个问题？                               │
│ - 谁被排除在外？                                                 │
│ - 当问题存在/不存在时谁受益？                                     │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│ 重构                                                             │
│ - 换一种说法，问题是：[重述]                                     │
│ - 我们如何能 [行动] 以达成 [目标]？                               │
└─────────────────────────────────────────────────────────────────┘
```

### 为什么这有效
- **拓宽视角：** 迫使你超越自己的假设
- **公平驱动：** 以边缘化声音为中心，问"谁被排除在外？"
- **挑战偏见：** 在框架问题之前要求明确审视假设
- **可操作的输出：** 产生可供解决方案探索使用的 HMW 陈述

### 反模式（这不是什么）
- **不是解决方案头脑风暴：** 画布框架问题；解决方案稍后出现
- **不是功能需求列表：** 聚焦根本问题，而非表面症状
- **不是单人练习：** 需要多元视角来挑战群体思维

### 何时使用
- 开始新计划的发现工作
- 重构现有问题（怀疑你在解决错误的东西）
- 在构建解决方案之前挑战假设
- 就问题定义对齐跨职能团队

### 何时不使用
- 当问题已经被充分理解且验证时
- 对于战术性 Bug 修复或技术债务（不需要深度框架）
- 当利益相关者已经承诺了某个解决方案（先解决共识问题）

## 应用

使用 `template.md` 获取完整的填写结构。

此交互式技能遵循**三阶段流程**，在每个阶段提出**自适应问题**。

---

### 步骤 0：收集上下文（提问前）

**智能体建议：**

在框架你的问题之前，让我们收集上下文：

**问题上下文：**
- 初步的问题陈述或利益相关者需求
- 你观察到的症状（支持工单、流失数据、用户投诉）
- 现有研究（用户访谈、调查、分析）
- 你对问题所做的假设

**利益相关者上下文：**
- 谁受此问题影响？（用户、客户、内部团队）
- 谁要求解决这个问题？（高管、销售、客户）
- 谁可能被忽视了？

**你可以直接粘贴这些内容，或简要描述问题。**

## 阶段 1：内省

**目标：** 审视你自己的假设、偏见，以及你可能如何成为问题的一部分。

### 问题 1：问题是什么？（描述症状）

**智能体提问：**
"根据你目前的理解，问题是什么？描述症状。"

**提供 4 个编号选项：**

1. **客户痛点** —— "客户在 [特定任务/结果] 上遇到困难"（例如，"客户找不到他们需要的功能"）
2. **业务指标问题** —— "我们看到 [指标下降]"（例如，"上季度流失率增加了 15%"）
3. **利益相关者需求** —— "利益相关者说我们需要 [功能/变更]"（例如，"销售团队说我们需要更好的报告"）
4. **观察到的行为** —— "我们注意到 [模式/趋势]"（例如，"用户在第 3 步放弃上手引导"）

**或描述你的问题/症状。**

**用户回复：** [选择或自定义]

**智能体提取：**
- **问题（初始框架）：** [来自用户的描述]

### 问题 2：我们为什么还没解决它？

**智能体提问：**
"为什么这个问题还没有被解决？"

**提供 6 个编号选项（可多选）：**

1. **它是新的** —— "问题最近才出现"
2. **它很难** —— "技术上复杂或资源密集"
3. **它优先级低** —— "其他计划优先"
4. **缺乏资源** —— "预算、人员或时间不够"
5. **缺乏权限** —— "不能做出决定或获得支持"
6. **系统性不公平** —— "问题不成比例地影响边缘群体，被忽视"

**或描述你自己的原因。**

**用户回复：** [选择或自定义]

### 问题 3：我们如何成为问题的一部分？（假设与偏见）

**智能体提问：**
"你（或你的团队）可能如何成为问题的一部分？你带来了什么假设或偏见？"

**提供 4 个编号选项：**

1. **假设我们知道客户想要什么** —— "我们没有与真实用户验证"（确认偏误）
2. **为我们自己优化，而非用户** —— "构建对我们容易的，而非对他们有价值的"（内部偏误）
3. **忽视特定用户细分** —— "聚焦多数用户，忽略边缘案例或边缘群体"（幸存者偏误）
4. **解决方案优先思维** —— "在理解根本问题之前就跳到'我们需要 [功能 X]'"（过早收敛）

**或描述你的具体假设/偏见。**

### 问题 6：谁被排除在外？谁受益？

**智能体提问：**
"到目前为止谁被排除在对话之外？当这个问题存在或不存在时谁受益？"

**智能体提示：**
- **谁被排除在外：** 边缘化声音、边缘案例、被忽视的利益相关者
- **问题存在时谁受益：** 谁从现状中获得收益？

## 常见陷阱

### 陷阱 1：跳过"内省"（假设你是中立的）
**症状：** 团队直接跳到"外观"而没有审视偏见

**后果：** 群体思维持续，假设未被挑战

**修复：** 强制对假设和偏见进行明确讨论（Q2-Q3）

---

### 陷阱 2：忽略"谁受益"问题
**症状：** 画布完成但未探索谁从问题存在中受益

**后果：** 错过了政治动态、对变革的抵制

**修复：** 始终坚持问"如果这个问题被解决谁受损？"（Q6）

---

### 陷阱 3：通用的问题陈述
**症状：** 重构后的问题模糊（"改善用户体验"）

**后果：** HMW 陈述不可操作

**修复：** 使问题具体（谁、什么、何时、后果、根本原因）

---

### 陷阱 4：HMW 陈述太狭窄
**症状：** "我们如何能添加一个移动应用？"

**后果：** 将解决方案空间限制为一个想法

**修复：** 保持 HMW 宽泛："我们如何能使移动优先用户在任何设备上访问核心工作流？"

---

### 陷阱 5：独自练习（没有多元视角）
**症状：** PM 独自填写画布

**后果：** 偏见持续，边缘化声音仍然被排除

**修复：** 与跨职能团队 + 客户输入一起主持画布工作坊
