## 目的

引导产品经理创建**Jeff Gothelf的精益UX画布（v2）**——一个单页引导工具，将工作围绕**需要解决的业务问题**而非**需要实现的解决方案**来框定。用此来对齐跨职能团队围绕核心假设、制定可测试的假设，并通过暴露理解差距（问题、用户、价值以及解决方案为何应该有效）确保每个迭代都发生学习。

这不是路线图或功能列表——它是一个**"保险单"**，将假设转化为实验，在承诺完整开发之前。画布将对话从**输出**转向**结果**，确保团队构建正确的东西，而不仅仅是正确构建东西。

## 核心概念

### 什么是精益UX画布？

**精益UX画布（v2）**是一个结构化的单页模板，旨在帮助团队将工作围绕业务问题而非解决方案来框定。它对齐跨职能团队在：
- 存在什么问题（以及为什么现在重要）
- 什么可衡量的结果表示成功
- 我们为谁解决
- 我们在做什么假设
- 我们首先需要学习什么
- 什么实验将测试这些假设

**起源：** 由Jeff Gothelf创建，*Lean UX*（O'Reilly, 2013）的作者。版本2发布以改善业务 vs. 用户结果的清晰度。

**关键洞察：** 画布像一张**保险单**——它在构建之前暴露理解差距，确保你不会在错误的东西上浪费迭代。

---

### 画布结构（8个框）

**布局（3列 × 3行）：**

```
┌─────────────────────┬──────────────┬───────────────────────┐
│ 1. 业务问题         │              │ 2. 业务结果           │
│                     │              │                       │
├─────────────────────┤ 5. 解决方案  ├───────────────────────┤
│ 3. 用户             │  (跨行1-2的  │ 4. 用户结果           │
│                     │   高框)      │    与收益            │
├─────────────────────┤              ├───────────────────────┤
│ 6. 假设             │──────────────┤ 8. 学习下一件最重要   │
│                     │ 7. 首先      │    事物的最少工作量    │
│                     │    学习      │                       │
└─────────────────────┴──────────────┴───────────────────────┘
```

**8个框（按此顺序填写）：**

1. **业务问题** — 世界上发生了什么变化，创造了一个值得解决的问题？
2. **业务结果** — 什么可衡量的行为变化表示成功？
3. **用户** — 你应该首先聚焦哪个/哪些角色？
4. **用户结果与收益** — 用户为什么寻求这个？他们获得什么收益？
5. **解决方案** — 什么功能/举措可能解决问题并满足用户需求？
6. **假设** — 结合框2-5的可测试假设（If/Then格式）
7. **首先学习的最重要事物是什么？** — 当前最具风险的单一假设
8. **学习下一件事的最少工作量是什么？** — 验证/证伪该假设的最小实验

---

### 为什么这有效

**问题优先，而非解决方案优先：**
从"世界上发生了什么变化？"开始，而非"我们应该构建X。"这防止了解决方案驱动的思维。

**假设驱动：**
在构建之前明确假设。每个学科都揭示其风险（技术可行性、用户价值、业务可行性）。

**以实验为中心：**
在投入资源之前测试假设。小实验胜过大赌注。

**跨职能对齐：**
共享的画布创造共同语言。每个人都看到相同的理解差距。

---

### 关键区别（避免混淆）

**框2（业务结果）vs. 框4（用户结果）：**
- **框2：** 可衡量的**行为变化**（留存率、网站停留时间、平均订单价值）
- **框4：** **目标、收益、情感、同理心**（省钱、升职、与家人共度时光）

框2是指标。框4是人性。

**解决方案（框5）是假设，而非承诺：**
列出候选解决方案（功能、政策，甚至业务模式转变）。你不是承诺构建所有——你是在探索解决方案空间。

**假设（框6）是可测试的：**
使用模板："我们认为[业务结果]将被实现，如果[用户]通过[解决方案]获得了[收益]。"每个假设聚焦**一个**解决方案。

---

### 反模式（这不是什么）

- **不是功能列表：** 解决方案是要测试的想法，而非待办事项
- **不是项目计划：** 画布框定学习，而非交付时间线
- **不是战略的替代品：** 画布执行战略；它不创造战略
- **不是一次性练习：** 随学习重新访问；更新假设

---

### 何时使用

✅ **在以下情况使用：**
- 开始一个新的产品举措或功能
- 重新框定一个现有项目（怀疑你在构建错误的东西）
- 对齐跨职能团队在假设和实验上
- 规划发现迭代或MVP
- 利益相关者是解决方案驱动的（"我们需要构建X"），你需要暴露假设

❌ **在以下情况不要使用：**
- 问题和解决方案已经验证（转向执行）
- 战术Bug修复或技术债务（无需学习）
- 利益相关者已承诺解决方案无论证据如何（先解决对齐问题）

### 问题1：业务问题（框1）

**Agent询问：**

**业务有什么需要你解决的问题？**

描述：
- **当前状态：** 业务今天如何交付价值？
- **发生了什么变化：** 市场转变、竞争威胁、客户行为变化、新交付渠道
- **为什么这很重要：** 为什么当前情况不满足期望？

**好的示例：**
- "我们的结账转化率在移动流量超越桌面后下降了15%。我们的结账流程并非为移动端设计，且竞品有一键结账。"

**坏的示例（太模糊）：**
- "我们需要增加收入"（没有关于什么变化的情境）
- "用户想要更多功能"（没有陈述业务问题）

3. **我不确定业务问题是什么** — [Agent建议先使用 `skills/problem-statement/SKILL.md` 或 `skills/problem-framing-canvas/SKILL.md`]

**用户响应：** [选择或描述]

**Agent验证：** 这描述了什么**变化**以及为什么它创造了问题？如果没有，问澄清问题。

### 问题2：业务结果（框2）

**Agent询问：**

**你将如何知道你解决了业务问题？你将衡量什么？**

聚焦**可衡量的行为变化**（欢迎领先指标）。问自己："如果解决方案有效，人们会有什么不同的做法？"

**业务结果示例：**
- 将移动结账转化率从45%提高到60%
- 将企业接入时间从3周减少到3天
- 将平均订单价值从$50提高到$75
- 将客户支持工单减少30%
- 将免费转付费转化率从5%提高到10%

**重要：** 这是**框2（行为变化）**，不是框4（用户收益/同理心）。指标在这里。情感在框4。

**Agent提供3个选项：**

1. **我心中有具体的指标** — [陈述你的业务结果]
2. **帮我定义可衡量的结果** — [Agent基于业务问题建议指标]
3. **我只有滞后指标（收入、利润）** — [Agent帮助识别预测这些结果的领先指标]

**用户响应：** [选择或描述]

**Agent验证：** 这些是可衡量的吗？可观察的吗？它们表示行为变化（不仅仅是"增加收入"）吗？

### 问题3：用户（框3）

**Agent询问：**

**你应该首先聚焦哪些类型（即角色）的用户和客户？**

考虑：
- 谁**购买**它？
- 谁**使用**它？
- 谁**配置**它？
- 谁**管理**它？

**为什么这重要：** 团队倾向于在这里走捷径（"每个人"）。画布想要一个**共享的用户愿景**——而且不总是"客户"。

**示例：**
- "专业服务业（顾问、会计师、律师）中的SMB业主（1-10名员工）"
- "为500+员工配置SSO的企业IT管理员"
- "每周订购外卖3次以上的移动优先千禧一代（25-35岁）"

**Agent提供3个选项：**

1. **我已经有角色了** — [引用 `skills/proto-persona/SKILL.md` 或粘贴角色]
2. **帮我识别目标用户** — [Agent询问：谁最经历业务问题？谁最可能采用？谁最容易触达？]
3. **我需要先创建角色** — [Agent建议使用 `skills/proto-persona/SKILL.md` 组件技能]

**用户响应：** [选择或描述]

**Agent验证：** 这足够具体到能想象一个真实的人吗？还是太宽泛（"所有用户"）？

### 问题4：用户结果与收益（框4）

**Agent询问：**

**为什么你的用户会寻求你的产品或服务？他们会获得什么收益？我们可以观察到什么行为变化告诉我们他们达成了目标？**

聚焦**目标、收益、情感、同理心**——不是指标（那些在框2）。

**用户结果与收益示例：**
- 每周节省10小时手工数据录入时间（更多时间陪伴家人）
- 通过更快速交付项目获得晋升
- 避免在朋友面前结账失败的尴尬
- 对无需致电支持就能配置企业软件感到自信

**为什么这重要：** 这是**同理心框**。它关乎人性动机，而不仅仅是行为变化。

**Agent提供3个选项：**

1. **我知道用户想要达成什么** — [描述用户收益]
2. **帮我识别用户收益** — [Agent提示：他们雇佣这个产品来完成什么任务？他们在避免什么痛苦？他们在寻求什么收获？]
3. **我需要先研究这个** — [Agent建议 `skills/jobs-to-be-done/SKILL.md` 或 `skills/discovery-interview-prep/SKILL.md`]

**用户响应：** [选择或描述]

**Agent验证：** 这解释了用户**为什么**在乎（不仅仅是他们会做什么）吗？

### 问题5：解决方案（框5）

**Agent询问：**

**我们可以制作什么来同时解决我们的业务问题并满足客户需求？**

列出可能有效的**功能、举措、政策、系统，甚至业务模式转变**。鼓励广泛的解决方案空间：大的/小的、创新的、"奇怪的"和非技术解决方案。

**示例：**
- 一键移动结账（Apple Pay、Google Pay）
- 自助接入向导（无需人工配置）
- AI驱动的推荐引擎
- 礼宾接入（高接触、人工——在自动化之前测试）
- 改变定价模式（基于用量而非固定费率）

**重要：** 这些是**假设**，而非承诺。你在探索选项，而不是承诺构建所有。

**Agent提供3个选项：**

1. **我有解决方案想法** — [列出你的候选解决方案]
2. **帮我头脑风暴解决方案** — [Agent基于问题、结果和用户建议解决方案]
3. **我心中只有一个解决方案** — [Agent挑战："还有什么可以解决这个问题？如果那个解决方案不行呢？"]

**用户响应：** [选择或描述]

**Agent验证：** 你至少有3个候选解决方案吗？（更多选项 = 更好的假设，稍后）

### 问题6：假设（框6）

**Agent询问：**

现在让我们通过结合框2-5的假设来创建可测试的假设。

**规则：**
- 每个假设聚焦**一个**解决方案（来自框5）

### 问题7：我们首先需要学习的最重要事物是什么？（框7）

**Agent询问：**

对于框6中的每个假设，识别其**最具风险的假设**。然后确定**当前最具风险的那个**。

**风险类型：**
- **价值风险：** 用户会真正使用这个吗？他们在乎吗？
- **可用性风险：** 用户能弄清楚如何使用吗？
- **可行性风险：** 我们在技术上能构建这个吗？
- **生存性风险：** 这会实现业务结果吗？

**提示：** 早期，将风险聚焦于**价值**而非可行性（大部分时间）。不要构建用户不想要的东西，即使在技术上是可行的。

**Agent提供：**

基于你的假设，以下是最具风险的假设：

1. **[假设1风险]** — 例如"用户会在没有看到明细费用的情况下信任一键结账"
2. **[假设2风险]** — 例如"自助接入会将设置时间减少到<3天"
3. **[假设3风险]** — 例如"AI推荐会将平均订单价值提高50%"

**哪一个是当前最具风险的？**

**选项：**
1. **风险1** — [选择并解释原因]
2. **风险2** — [选择并解释原因]
3. **风险3** — [选择并解释原因]
4. **我不确定哪个最具风险** — [Agent帮助优先级排序：哪个假设如果错误，会扼杀这一举措？]

**用户响应：** [选择]

**Agent记录：** 这是我们将首先测试的假设。

### 问题8：我们需要做的最少工作来学习下一件最重要的事是什么？（框8）

**Agent询问：**

设计一个实验来验证或证伪最具风险的假设（来自框7），**以你能做到的最快速度**。

**实验类型示例：**
- **客户访谈** — 5-10次访谈来测试价值假设
- **落地页** — 假门测试来衡量兴趣
- **礼宾/手动原型** — 在自动化之前的高接触、手动版本
- **绿野仙踪** — 假装功能存在（幕后由人类操作）
- **烟雾测试** — 宣布功能，衡量注册数

**提醒：** 实验设计是一门艺术。画布推动你去做——期望在实验设计上单独迭代。

**Agent提供：**

基于你最具风险的假设，以下是实验选项：

1. **客户访谈** — 与10位用户交谈，询问他们是否会使用[解决方案]，观察他们当前的变通方法
2. **落地页测试** — 创建描述[解决方案]的落地页，衡量注册率
3. **礼宾测试** — 向5位用户手动交付[收益]，观察他们是否达成[结果]
4. **绿野仙踪原型** — 用幕后的人模拟[解决方案]，看用户是否完成工作流
5. **设计自定义实验** — [描述你的实验]

**你会运行哪个实验？**

## 常见陷阱

### 1. **从解决方案而非问题开始**
**失败模式：** 框1说"我们需要构建X"而不是描述什么变化了。

**后果：** 你构建别人已经决定好的解决方案，而没有验证问题是否存在。

**修复：** 问："世界上发生了什么变化？为什么这是现在的问题（vs. 6个月前）？"

---

### 2. **模糊的业务结果**
**失败模式：** 框2说"增加收入"或"让用户满意"。

**后果：** 没有办法衡量成功；无法判断实验是否有效。

**修复：** 定义可衡量的行为变化。"将平均订单价值从$50提高到$75"或"将支持工单减少30%。"

---

### 3. **太宽泛的用户细分**
**失败模式：** 框3说"所有用户"或"每个人"。

**后果：** 无法设计有针对性的实验；在没有会采用的角色的情况下浪费时间。

**修复：** 先选一个角色开始。以后可以扩展。

---

### 4. **混淆框2和框4**
**失败模式：** 将情感放在框2中，将指标放在框4中（反之亦然）。

**后果：** 假设对齐错误；成功标准不清晰。

**修复：** **框2 = 行为变化（指标）。** **框4 = 目标、收益、情感（同理心）。**

---

### 5. **框5中只有一个解决方案**
**失败模式：** 列出一个功能因为利益相关者已经决定了。

**后果：** 没有对替代方案的探索；无法测试哪个解决方案最好。

**修复：** 强制自己列出3+个解决方案。问："还有什么可以解决这个问题？"

---

### 6. **跳过实验（框8）**
**失败模式：** "我们直接构建它看看会发生什么。"

**后果：** 浪费数周/数月构建错误的东西。

**修复：** 先设计最小的实验。如果你想不到，使用 `skills/pol-probe-advisor/SKILL.md` 选择一个验证方法。
