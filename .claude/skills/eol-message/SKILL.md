## 目的
撰写清晰、有同理心的产品生命周期结束（EOL）消息，传达产品或功能的停用决定，解释理由，说明客户影响，提供过渡支持，并定位替代解决方案。用此来在困难过渡期间维护客户信任，通过展现关怀和提供明确的前进路径来降低流失。

这不是一个通用的关停公告——它是以客户为中心的沟通，承认损失的同时将变化框定为进步。

## 核心概念

### EOL消息框架
有效的EOL消息在诚实地说明变化与对客户影响展现同理心之间取得平衡。它包含：

1. **公司情境：** 你是谁以及你对客户的承诺
2. **公告：** 什么正在停用以及什么在替代它
3. **理由：** 为什么这个决定对客户有益（而不仅仅是对业务）
4. **当前产品情境：** 该产品曾是什么以及它为谁服务
5. **客户影响：** 这对用户有何影响（承认中断）
6. **过渡解决方案：** 替代方案是什么以及它如何改进旧方案
7. **支持措施：** 你将如何帮助客户迁移
8. **时间线：** 关键日期和里程碑
9. **行动号召：** 后续步骤和联系信息

### 为什么这有效
- **同理心优先：** 在合理化决定之前先承认客户中断
- **清晰度：** 关于什么在变化以及何时变化没有歧义
- **以支持为中心：** 表明你不会在过渡期间抛弃客户
- **面向未来：** 将变化框定为进步，而非损失

### 反模式（这不是什么）
- **不是简略的关停通知：** "我们将停用产品X。再见。"
- **不是以业务为中心：** 不要以"这降低了我们的成本"开头
- **不是模糊的：** "很快"不是一个时间线
- **不是防御性的：** 不要责怪客户（"低使用率迫使我们关停"）

### 何时使用
- 停用产品、功能或服务
- 将客户从旧版平台迁移到新平台
- 关停被收购目标的产品
- 弃用某个技术栈或API

### 何时不使用
- 对于小的功能调整（不要对细微变更过度沟通）
- 在你制定过渡计划之前（在你知道如何支持客户*之后*再沟通）
- 如果你暗中希望客户不会注意到（要保持透明）

## 应用

使用 `template.md` 获取完整填写结构。

### 步骤1：收集情境信息
在起草之前，确保你已准备好：
- **正在停用的产品：** 具体什么将终止？
- **替代解决方案：** 什么在替代它（如果有的话）？
- **时间线：** 关键日期（公告、功能冻结、关停、数据导出截止日期）
- **客户影响：** 多少用户受影响？哪些工作流被中断？
- **支持计划：** 迁移支持、培训、折扣、数据导出工具
- **理由：** 为什么会发生？（技术过时、战略转变、整合等）

**如果缺少情境信息：** 在有完整的过渡计划之前不要发送消息。客户会问"我现在该做什么？"——你必须有答案。

---

### 步骤2：起草产品过渡叙述

#### 公司情境
确立你是谁以及你的承诺：

```markdown
### 产品过渡叙述

**我们是：** [描述公司及其与正在逐步淘汰产品的关系]
- [关于公司对客户承诺的关键点]
- [关于公司产品演进的关键点]
- [关于公司未来愿景的关键点]
```

**示例：**
```markdown
**我们是：** Acme Workflows，一家为50,000家小企业服务的工作流自动化平台
- 我们致力于帮助你节省时间，专注于重要的事情
- 我们基于你的反馈和技术进步不断演进我们的产品
- 我们正在构建一个自动化触手可及、强大且简单的未来
```

---

#### 公告
清晰直接：

```markdown
**公告：**
- [一句话清晰陈述产品的EOL并介绍其替代方案]
```

**示例：**
- "我们将在2026年12月31日停用Acme Workflows Classic，并将所有客户迁移到Acme Workflows Pro。"

---

#### 理由（以客户利益为中心）
解释这*为什么*对客户有益：

```markdown
**因为：**
- [理由1：例如，技术进步]
- [理由2：例如，性能提升]
- [理由3：例如，更好地与客户需求对齐]

**这意味着对你而言：**
- [从客户角度描述影响和收益]
```

**示例：**
```markdown
**因为：**
- Acme Workflows Classic运行在过时的基础设施上，限制了性能和可扩展性
- Acme Workflows Pro建立在现代技术之上，实现更快的自动化、更好的集成和实时协作
- 整合到一个平台使我们能够将100%的工程资源投入到你已请求的功能上

**这意味着对你而言：**
- 更快的自动化执行（3倍速度提升）
- 50+个与您已使用工具的新集成
- 获取新功能，如实时协作和移动应用
```

---

### 步骤3：提供当前产品情境

承认什么是正在失去的：

```markdown
### 当前产品情境

**我们的产品** [正在停用的产品名称]
- **是一个** [对产品及其主要功能的简要描述]
- **已为** [目标客户/用户] **服务了** [时长或时间段]
- **通过提供** [产品提供的关键收益或解决方案]
```

**示例：**
```markdown
**我们的产品** Acme Workflows Classic
- **是一个** 帮助小企业消除重复任务的工作流自动化工具
- **已为** 超过20,000位客户 **服务了** 8年
- **通过提供** 可靠、简单的自动化而无需技术专业知识
```

---

### 步骤4：承认客户影响

诚实地说明中断：

```markdown
### 客户影响

**我们理解这可能通过以下方式影响你：**
- [对客户运营或流程的潜在影响1]
- [对客户运营或流程的潜在影响2]
- [对客户运营或流程的潜在影响3（如适用）]
```

**示例：**
```markdown
**我们理解这可能通过以下方式影响你：**
- 需要时间将工作流从Classic迁移到Pro
- 学习新功能和界面变化
- 如果你自定义了工作流，需要更新集成或API连接
```

---

### 步骤5：呈现过渡解决方案

使用定位陈述格式（参考 `skills/positioning-statement/SKILL.md`）：

```markdown
### 过渡解决方案

**为** [受EOL影响的目标客户/用户]
- **当前使用** [正在逐步淘汰的产品名称]
- [替代产品名称]
- **是一个** [替代产品类别定义]
- **能够** [对用户的收益陈述，聚焦连续性和改进]

### 差异化和连续性

- **像** [正在逐步淘汰的产品]一样，
- [替代产品名称]
- **提供** [替代产品如何保持旧产品的关键收益]
- **同时还提供** [新的收益或改进]
```

**示例：**
```markdown
### 过渡解决方案

**为** 小企业主
- **当前使用** Acme Workflows Classic
- Acme Workflows Pro
- **是一个** 下一代工作流自动化平台
- **能够** 保持所有你喜爱的简单性和可靠性，同时增加3倍更快的性能、50+个新集成和实时协作

### 差异化和连续性

- **像** Acme Workflows Classic一样，
- Acme Workflows Pro
- **提供** 无需编码即可轻松构建自动化、可靠的执行和简单的定价
- **同时还提供** 3倍更快的工作流、移动应用访问、实时团队协作，以及与Slack、Asana和Notion等工具的集成
```

---

### 步骤6：概述支持措施和时间线

#### 支持措施

```markdown
### 支持与后续步骤

**为确保平稳过渡，我们将：**
- [支持措施1，例如，"为所有客户提供1对1迁移协助"]
- [支持措施2，例如，"自动迁移你的工作流（经你批准）"]
- [支持措施3，例如，"为现有客户提供Acme Workflows Pro 3个月折扣"]
```

---

#### 时间线

```markdown
### 时间线

- [关键日期1及相关里程碑，例如，"2026年3月1日：迁移工具可用"]
- [关键日期2及相关里程碑，例如，"2026年9月1日：Acme Workflows Classic变为只读"]
- [关键日期3及相关里程碑，例如，"2026年12月31日：Acme Workflows Classic完全停用，数据导出截止日期"]
```

**质量检查：**
- **充足的准备时间：** 客户需要时间来规划（通常6-12个月）
- **明确的里程碑：** 何时功能冻结？何时关停？
- **数据导出截止日期：** 他们何时失去对数据的访问权限？

---

### 步骤7：提供明确的后续步骤

```markdown
### 行动号召

- [客户需要执行的明确后续步骤，例如，"登录你的账户启动迁移向导"]
- [有问题或需要协助的联系信息，例如，"联系我们的支持团队 support@acme.com 或拨打 1-800-ACME-HELP"]
```

## 常见陷阱

### 陷阱1：以业务为中心的理由
**症状：** "我们正在停用产品X以降低成本并整合我们的产品组合。"

**后果：** 客户感觉自己是商业决策中的附带损害。

**修复：** 以客户收益来框定理由："我们正在整合到产品Y，以便能够将100%的资源投入到你已请求的功能中。"

---

### 陷阱2：模糊的时间线
**症状：** "产品X将很快被停用。"

**后果：** 客户无法规划。焦虑和流失增加。

**修复：** 提供具体日期："3月1日：迁移工具可用。12月31日：完全关停。"

---

### 陷阱3：没有支持计划
**症状：** "你需要迁移到产品Y。祝你好运！"

**后果：** 客户感到被抛弃。高流失风险。

**修复：** 提供迁移支持："1对1协助、自动迁移工具、3个月折扣、培训资源。"

---

### 陷阱4：忽视客户影响
**症状：** 消息从公告直接跳到"这是新产品！"

**后果：** 客户感觉他们的关切未被承认。

**修复：** 明确承认影响："我们理解这需要时间来迁移和学习新功能。"

---

### 陷阱5：简略或防御性语气
**症状：** "由于低使用率，我们正在关停产品X。"

**后果：** 听起来像是在责怪客户。

**修复：** 要有同理心和前瞻性："我们正在整合以聚焦自动化的未来。"
