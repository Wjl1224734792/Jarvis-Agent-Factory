项目概览

Jarvis Agent Factory（贾维斯智能体工厂）是一套面向 Claude Code 平台的多智能体配置工程。它将软件开发从"手动编码"提升为"编排驱动"——通过 39 条指令、71 个专业智能体、34 个技能和 15 条流水线，覆盖从需求澄清到发布上线的完整软件生命周期。


核心思路：FSM 硬约束闸门系统（Gate A→E）强制每个阶段完成必要产出后才能推进，确保工程严谨性；同时 /auto 智能路由自动检测任务类型并跳过无关闸门，兼顾日常效率。


安装方式：npm i -g jarvis-agent-factory → jarvis init -y → Claude Code 重启后引擎自动拉起。零原生依赖（Node 22.5+ 内置 node:sqlite），安装秒级完成。

核心概念

📋

指令 Command

用户交互入口，以 / 开头（如 /jarvis、/auto）。每条指令对应一条流水线，定义从需求到交付的完整流程。共 39 条。

🤖

智能体 Agent

专业化 AI 角色，在流水线各阶段由编排者 spawn 执行。按职能分 6 类：实现(25)、测试(15)、审查(16)、规划(4)、架构(4)、支撑(7)。共 71 个。

💪

技能 Skill

可复用的行为指南和能力模块，由 Agent 在任务执行中加载。涵盖需求、规划、实现、质量、测试、安全、文档等 12 个领域。共 34 个。

🏭

流水线 Pipeline

预定义的 Gate 序列，每种开发场景对应一条。full（13门完整）和 lite（自适应跳过）是两条核心流水线，另有 13 条专业流水线。共 15 条。

🚪

闸门 Gate

FSM 硬约束检查点，当前 Gate 通过后才能推进到下一 Gate。引擎通过 gate_check 和 gate_enforce 强制执行操作权限表，禁止跳过或回退。

⚙

引擎 Engine

MCP stdio 自动拉起的编排引擎（端口 3456），内置 SQLite WAL 模式、SSE 实时推送、FSM Gate 系统。负责会话管理、Gate 执行、Agent 调度和文档产物追踪。

指令体系 — 39 条指令，8 个分类

每条指令是一条完整的开发流水线入口。输入 /指令名 任务描述 即可启动。引擎自动推进 Gate 序列，在对应阶段 spawn 专业 Agent 执行任务。不确定用什么指令？直接用 /auto。

一、编排入口（2条）

日常默认入口：/auto — 自动检测 12 种任务类型 → 路由最优流水线 → 智能跳过无关 Gate → 按复杂度分配 Team/Subagent。99% 的情况用它就够了。
/jarvis
全流程严格模式 — 13 Gate 全部强制执行，适合中大型功能开发。从需求到发布的完整生命周期。
Gate A需求澄清：分析用户输入，产出 REQ-XXX 需求文档
Gate B-DDD领域驱动分析：识别聚合/实体/值对象/领域服务
Gate B-BDD行为驱动场景：Gherkin Given/When/Then 验收场景
Gate B-TDD测试驱动分解：TASK-XXX 映射 REQ-XXX，Red→Green→Refactor
Gate B1架构评审：spawn 对应架构师，产出架构评审报告
Gate C执行规划：planner 产出 parallel_batches 并行任务分组
Gate C-impl并行实现：Team 模式 spawn 多个实现 Agent 并行编码
Gate C1质量门：Lint + Type-check + Build + Deps Audit 全部通过
Gate C1.5视觉验证：截图对比，确保 UI 与设计一致（条件性）
Gate C2测试验证：Team 模式 spawn 测试 Agent，最多 5 次重试
Gate D评审：Team 模式 spawn 审查 Agent，产出分级审查报告

/auto
智能路由模式 — 自动检测 12 种任务类型（新功能/Bug/重构/热修复/审查/调研/调试/简化/改进/迁移/发布/小修改），路由最优流水线，智能跳过无关 Gate。按复杂度（轻/中/重）分配 Team 或 Subagent。
Step 1任务类型检测：分析输入，匹配 12 种已知类型
Step 2流水线路由：选择 full / refactor / hotfix / research / debug / simplify / improve / migrate / release 流水线
Step 3Gate 跳过：检测无关 Gate（如纯后端跳过 C1.5），调用 gate_jump
Step 4复杂度评估：轻→Subagent / 中→Team 2-3 / 重→Team 4-6
Step 5执行 + 质量门 + 测试 + 评审：按流水线 Gate 推进
关键 Agent：按路由结果动态选择

三、需求与设计（5条）

/ask
需求探询 — 4 模式自适应：Interview（模糊→Socratic 追问澄清）、Direct（清晰→直接分析）、Consensus（ralplan 式 3 阶段共识：Planner→Architect→Critic）、Review（Critic 评估现有计划）。含 5 维度评分矩阵（BV/EF/RS/UI/DC）。
Gate K0模式选择：分析输入清晰度，选择 Interview/Direct/Consensus/Review
Gate K1信息收集：需求背景、业务价值、功能范围、用户场景
Gate K2多维度分析：BV(业务价值) / EF(工程可行) / RS(风险安全) / UI(用户体验) / DC(交付信心) 评分

Gate RS0课题定义：明确研究范围、目标和成功标准
Gate RS1信息收集：多源数据采集（文档/社区/案例/基准测试）
Gate RS2多维分析：技术成熟度/生态/性能/安全性/维护成本
Gate RS3假设验证：证据矩阵 + 竞争假设评估
Gate RS4研究报告：结论 + 推荐方案 + 实施路线图

/debug
调试诊断 — 5 Gate 交互式诊断：收集信息→最小复现→调试会话→交互诊断→报告+修复方案。

Gate T2证据收集：为每个假设收集支持/反对证据
Gate T3贝叶斯更新：用证据更新假设概率，置信度>70% 接受方案
Gate T4解决方案：基于最可能假设的修复方案
/evaluate
技术评估 — 4 Gate：定义标准+权重→生成原型→收集指标→多维度报告+推荐方案。适合技术选型。
Gate E0评估标准：定义维度+权重（性能/生态/学习成本/维护性）
Gate E1原型生成：为每个候选方案生成概念验证原型
Gate E2指标收集：运行基准测试，收集量化数据
Gate E3报告：多维雷达图 + 加权评分 + 推荐方案
/simplify
代码简化 — 4 Gate 五桶分析：死代码/重复/过度抽象/AI痕迹/复杂性。并行简化→回归验证→报告。失败自动回滚。
Gate S0五桶分析：死代码→重复代码→过度抽象→AI痕迹→高复杂度，分类标记
Gate S1并行简化：5 个维度并行处理，每步安全检查和回归验证
Gate S2回归验证：全部测试通过+覆盖率不降，失败自动回滚
Gate S3报告：简化前后对比（行数/圈复杂度/重复率）
/improve
度量驱动迭代改进 — 5 Gate + 迭代循环：量化目标→研究→计划→执行+benchmark→评估→迭代/停止。支持 IM4→IM1 循环直到达标。
Gate IM0量化目标：定义目标指标+baseline测量+成功阈值
Gate IM1研究：分析瓶颈，生成优化假设
Gate IM2计划：选择最佳假设，制定实施计划
Gate IM3执行：实施优化 + 运行 benchmark 测量
Gate IM4评估：达标→停止；未达标→回 Gate IM1 继续迭代

五、审查与修复（3条）

/review
只读代码审查 — 不修改文件。spawn 多个审查 Agent 并行收集 findings（每条含文件/行号/依据/严重度），产出审查报告+严重度分级。
Step 1范围确定：选定审查文件和关注维度
Step 2并行审查：spawn 审查 Agent 并行扫描（安全/性能/风格/架构）
Step 3发现汇总：合并 findings + 去重 + 严重度分级（P0/P1/P2/P3）
Step 4报告输出：审查报告（只读，不修改代码）
关键 Agent：frontend-review-expert / backend-review-expert / security-review-expert / perf-review-expert / project-review-expert / diff-review-expert / change-review-expert / qa-review-expert
/review-fix
审查修复优化闭环 — ①初审(只读)→②修复规划→③执行修复→④验证(质量重检)→⑤复审。最多 2 轮闭环。
Step 1初审：spawn 审查 Agent 并行审查（同 /review）
Step 2修复规划：remediation-planner 制定修复方案，按严重度排序
Step 3执行修复：spawn 对应实现 Agent 逐项修复
Step 4质量重检：Lint+Type-check+Build+测试 全部通过
Step 5复审：重新审查，≤2轮。仍失败→ABORT
关键 Agent：review-fix-optimize / remediation-planner / review-only / 实现类+审查类 Agent
/bug-fix
Bug 修复闭环 — 浏览器复现→根因定位→修复→浏览器验证。涉及前端/交互类 Bug 的 7 步闭环，集成 Playwright MCP。
Step 1Bug 描述：收集复现步骤+期望行为+实际行为
Step 2浏览器复现：Playwright 打开页面，按步骤操作确认 Bug
Step 3根因定位：代码追踪→假设验证→确定根因
Step 4修复实现：最小化代码修改
Step 5浏览器验证：Playwright 重新操作确认修复
Step 6回归测试：确保无新问题引入
Step 7修复报告：根因+修复+验证结果
关键 Agent：remediation-expert / browser-test-expert

流水线架构

每条指令对应一条流水线，每条流水线由一组 Gate 组成。Gate 是 FSM 硬约束检查点——前一个 Gate 未通过，无法推进到下一个。引擎通过 MCP 工具 gate_check 和 gate_enforce 在每次 Agent spawn 后自动检查 Gate 状态，禁止跳过或回退。

使用推荐

💡 新手入门路径：
/auto 需求描述 → 自动路由 + 引导 → /review 审查产出 → /review-fix 修复问题 → /release 发布
不需要记住所有指令，99% 的情况 /auto 就够了。

💻 日常开发（默认）

使用 /auto，自动检测任务类型→路由最优流水线→跳过无关Gate→按复杂度分配Agent。最省心。

🏗 中大型功能

使用 /jarvis，13 Gate 完整生命周期。DDD→BDD→TDD→架构→规划→并行实现→视觉验证→测试→评审→发布，质量保证最强。

🔧 重构代码

使用 /refactor，R1-R5 安全网保护。建立基线测试→执行重构→行为漂移检测→失败自动回滚。

🚨 紧急修复线上故障

使用 /hotfix，H0-H3 紧急协议。影响声明+审批→最小化修复→快速验证+回滚预案→事后审计。

🧹 代码质量清理

使用 /simplify，五桶分析（死代码/重复/过度抽象/AI痕迹/复杂性）。失败自动回滚，不怕改坏。

🔍 排查 Bug

简单问题用 /debug（交互式诊断），复杂根因用 /trace（贝叶斯因果推理，多假设竞争+证据更新）。

⚡ 性能优化

使用 /improve，度量驱动迭代。量化目标→研究→计划→执行+benchmark→评估→不达标继续迭代。

📊 技术选型

快速选型用 /evaluate（原型+量化指标+雷达图），深度调研用 /research（5阶段分析+竞争假设验证）。

❓ 需求不清晰

使用 /ask（Interview 模式），Socratic 追问→K1探索空间→K2五维评分矩阵→K3产出分析报告+路由建议。

📦 框架升级

使用 /migrate，M1-M4 迁移流程。定义规则→应用迁移→编译验证→自动修复Lint≤2轮。

📝 代码审查

使用 /review（只读审查）→ /review-fix（修复闭环）。先审查后修复再复审，最多2轮。

📤 发布上线

个人/小团队用 /release（质量门→版本递增→CI自动发布），团队协作用 /publish（含 PR+审查+合并流程）。

🎯 前端开发

使用 /frontend，C1.5 视觉验证强制。spawn frontend-dev/ui/state-expert 并行实现，frontend-architect 架构评审。

🌐 后端开发

使用 /backend，跳过 C1.5 视觉验证。spawn backend-dev/api/logic/data-expert 并行实现，database-architect 数据模型评审。

📱 移动端/跨端开发

使用平台对应指令（/android /ios /flutter /expo /react-native /taro），每个平台有专属 dev/ui/state/test/review 全套 Agent。

🧪 测试补齐

按需选择：/test-unit（单元）/test-integration（集成）/test-e2e（端到端）/test-perf（性能）/test-security（安全）。前端用 /browser-test 文档驱动测试。

快速开始

3 步开始：

1. npm i -g jarvis-agent-factory — 安装 CLI（零原生依赖，Node 22.5+ 内置 node:sqlite）

2. jarvis init -y — 一键部署配置 + MCP + 钩子到当前项目

3. 重启 Claude Code → 引擎自动拉起 → 输入 /auto 你的任务 开始使用


Web 面板：jarvis web → http://localhost:3457 查看流水线看板、会话管理、Agent 配置、归档记录。

远程面板：每次 GitHub Release 附带单 HTML 文件，下载即可打开，无需本地安装。
