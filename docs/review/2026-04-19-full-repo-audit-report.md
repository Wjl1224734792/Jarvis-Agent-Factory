# 2026-04-19 全仓审查报告

- `apps/web`
- `apps/admin`
- `apps/server`
- `packages/config`
- `packages/db`
- `packages/http-client`
- `packages/schemas`

- 相关根脚本、根配置和文档

## 审查方法

- 按架构、算法/数据流、性能/扩展性、测试/交付四个维度并行只读审查
- 以仓库内代码、脚本和文档路径作为主要证据来源
- 无法仅凭仓库证实的内容标记为 `inferred`

## 总体结论

- 结论：有条件通过，但不宜直接进入无计划的大规模优化
- 说明：
  - 仓库的整体工程护栏和 monorepo 骨架是成立的
  - 当前最主要的问题不是“完全失控”，而是关键边界已经开始漂移，少数热点文件、共享 client、共享 seed 与横切业务规则正在成为系统性风险放大器

## 已确认的 P0 问题

1. `API client / 契约` 边界裂开，共享 client 与 Web/Admin 本地包装层并存，漂移风险高。
2. 多个核心 service/repo 已经成为仓库级热点文件，边界装不下真实职责。
3. 品牌申请状态流存在 `hidden` 契约断裂，运行时可能直接落库失败。
4. 榜单条目把“评分写入”和“评论写入”绑定在一起，会持续制造脏数据并污染排序。
5. 多个 server 测试文件在 `beforeEach` 使用整库 `reset + seed`，验证成本和基础设施敏感度高。

## 已确认的 P1 问题

1. `social.service` 已演变成跨领域枢纽服务。
2. `admin-search` 将数据结果与管理端导航路径硬编码耦合。
3. 推荐流和站内搜索都存在排序/分页语义不稳的问题。
4. 投稿域状态机和落库字段语义存在漂移。
5. 日志链路存在写入串行与读取放大问题。
6. 前端大页面、大入口和重型 vendor chunk 已达到应主动治理的程度。
7. `packages/db` 缺少独立测试层，E2E 更像联调而非发布态验收。

## 正向观察

- `packages -> apps` 反向依赖未在静态检索中发现。
- 根级 `lint/typecheck/test/build` 闭环完整。
- 前端网络调用大体收口在 `lib` 层，未看到大面积页面内散落 `fetch`。

## 当前阶段不处理

- 不在本轮直接改业务代码。
- 不在本轮直接改共享契约、根脚本或跨 app 边界。
- 不在证据不足时强行给出线上性能结论。

## 建议处理原则

- 先修会制造脏数据、状态断裂和共享漂移的 P0。
- 再做热点文件和共享边界治理。
- 最后再推进更细粒度的性能优化和体验优化。
