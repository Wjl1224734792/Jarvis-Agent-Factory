## 2. 任务概览
本轮只做飞行器库主链路，不扩展到评分、发布、榜单、消息。

目标是先打通一条可用的垂直链路：
`共享合同与数据底座 -> /models 列表与筛选 -> 机型详情页 -> 后台分类/机型最小闭环`

已知边界：
- 现有 `apps/server` 只承载认证、用户与会话。
- 现有 `apps/web`、`apps/admin` 都已经有基础壳。
- `packages/db`、Drizzle、migration、seed、auth 持久化已完成并验证。
- 飞行器库会新增一组新的共享合约和业务模块，不能塞进 auth 模块里。

## 3. 建议的最小垂直切片
### 切片 A：共享合同与读链路
先把飞行器库的“数据长什么样、接口怎么说、列表/详情怎么查”定死。
- 产出：分类、品牌、机型、参数、筛选条件、返回结构、排序与重置规则。
- 价值：后续前端、后台和 server 不再各自猜字段。

### 切片 B：前台 `/models` 完整浏览链路
先做可浏览、可筛选、可进详情的主链路。
- 产出：机型列表、类型筛选、品牌筛选/索引、动力类型筛选、不限重置、详情页、核心/详细参数展示。
- 价值：先让“飞行器库能用”，再补后台维护能力。

### 切片 C：后台最小闭环
先把分类和机型的最小管理能力做出来。
- 产出：分类增删改查/启停/排序，机型增删改查/发布状态/关联分类与品牌。
- 价值：保证前台内容可持续维护。

## 4. 任务分解列表
| 任务 ID | 名称 | 类型 | 优先级 | 完成标准 |
|---|---|---|---|---|
| T1 | 飞行器库共享合同与数据底座收敛 | DDD + TDD | P0 | `packages/schemas`、`packages/http-client`、`packages/db`、`packages/shared` 的飞行器库字段、路由常量、请求/响应结构、迁移/seed 口径统一；品牌、分类、机型、动力类型、详情参数的命名和主键/slug 规则固定 |
| T2 | server 飞行器库只读模块 | DDD + TDD | P0 | `GET /models`、`GET /models/:slug`、分类/品牌/动力筛选查询可用；列表分页或聚合规则、排序规则、重置语义、品牌索引返回口径固定；server 不暴露数据库实现细节 |
| T3 | web `/models` 列表与筛选页 | 直接开发 | P0 | `/models` 路由可访问；列表、类型筛选、品牌筛选/索引、动力筛选、不限重置可操作；页面能接入真实 API 并保持响应式布局 |
| T4 | web 机型详情页 | 直接开发 | P0 | `/models/:slug` 可访问；核心参数、详细参数、基础介绍、相关操作入口可展示；从列表或首页可跳转进详情 |
| T5 | 后台分类管理最小闭环 | DDD + TDD | P1 | 后台可完成分类的新增、编辑、启停、排序、列表查询；分类 slug 与品牌/机型关联规则稳定；不会破坏前台筛选结果 |
| T6 | 后台机型管理最小闭环 | DDD + TDD | P0 | 后台可完成机型新增、编辑、发布/下架、关联分类/品牌/动力类型、参数录入与列表查询；前台详情页所需字段可由后台维护 |
| T7 | 前后台飞行器库请求层统一 | TDD + 直接开发 | P0 | `packages/http-client` 提供飞行器库读写接口封装；web/admin 只通过统一请求层访问 server，不直连内部实现；错误结构统一 |
| T8 | 迭代交付说明与验收口径补齐 | 直接开发 | P1 | 启动方式、环境变量、接口入口、筛选规则、后台操作路径写清；planner 可直接据此拆执行任务 |

## 5. DDD 分类
### 必须 DDD
- T1 飞行器库共享合同与数据底座收敛
- T2 server 飞行器库只读模块
- T5 后台分类管理最小闭环
- T6 后台机型管理最小闭环

原因：
- 分类、品牌、机型、动力类型、参数展示是一个稳定领域。
- 筛选、索引、详情参数、后台维护之间有明显一致性约束。
- 分类与品牌关系、机型与分类/品牌关系会影响多个页面和接口。

### 可以不走 DDD
- T3 web `/models` 列表与筛选页
- T4 web 机型详情页
- T7 前后台飞行器库请求层统一
- T8 迭代交付说明与验收口径补齐

## 7. 依赖顺序
### 推荐主顺序
1. T1 先收敛共享合同与数据底座。
2. T2 再落 server 只读模块。
3. T3、T4 可并行推进前台浏览链路。
4. T5、T6 可并行推进后台最小闭环，但不要同时改同一份共享合同。
5. T7 放在 T1 之后、T2 之后都可以，但应在前后端联调前收口。
6. T8 最后补交付说明。

### 可以并行的窗口
- T3 与 T4：同属 web，只要路由和接口口径稳定，可以并行，但避免同时改公共 layout 或路由常量。
- T5 与 T6：同属后台业务，但共享分类/机型合约，建议只在合同稳定后并行。

## 8. 前后端边界
### 共享区
- `packages/db`
- `packages/schemas`
- `packages/http-client`
- `packages/shared`

### server 责任
- 只负责飞行器库的查询、筛选、详情、后台 CRUD、权限校验和错误结构。
- 不承载页面渲染，不暴露数据库细节给前端。

### web 责任
- 只负责 `/models` 浏览体验、筛选交互、详情展示和页面跳转。
- 不直接访问数据库，不绕过 `http-client`。

### admin 责任
- 只负责分类和机型的最小维护闭环。
- 不复用 web 的页面逻辑，不直接读写数据库。

## 9. 必须主代理单线程收敛的共享区
以下区域不要并行抢改，必须由主代理单线程收敛：

- `packages/db/src/schema.ts`
- `packages/db/src/seed.ts`
- `packages/schemas/src/*`
- `packages/http-client/src/index.ts`
- `packages/shared/src/index.ts`
- `apps/server/src/app.ts`
- `apps/server/src/modules/*` 中新增的飞行器库模块
- `apps/server/src/routes/*` 中新增的飞行器库路由

原因：
- 这些地方一旦字段名、路由名、返回结构改错，会同步影响 web、admin、server 三端。
- 机型、品牌、分类、动力类型的命名一旦不收敛，后面每个页面都会重复返工。

## 10. 风险任务
- T1 是最高风险共享区，任何字段名、slug、排序、参数结构的偏差都会同步破坏前后端。
- T2 涉及筛选语义和详情返回，容易把查询规则写成“页面需要什么就返回什么”，后续难以维护。
- T5/T6 涉及分类与机型的关联一致性，最容易出现“后台能改，前台查不到”。
- T7 若把错误映射做重，会把请求层变成第二套业务逻辑。
- seed 与既有测试数据如果不统一，前台品牌索引和筛选顺序会漂。

## 11. 文件所有权和共享路径提醒
下面路径属于共享高风险区，改动前要先锁定单一负责人：

- `packages/db`
- `packages/schemas`
- `packages/http-client`
- `packages/shared`
- `apps/server/src/modules/*`
- `apps/server/src/routes/*`
- `apps/web/src/routes/*`
- `apps/admin/src/features/*`

额外提醒：
- 当前任务文档位于 `docs/tasks`，这是共享路径，后续如果再拆子任务，不要多人同时改同一份任务文档。
- 本轮不要让 web、admin、server 三端各自私改飞行器库字段，必须先由共享合同收敛。

## 12. 推荐交付顺序
1. 先交 T1，锁定合同、字段与数据底座。
2. 再交 T2，把 server 读链路固定。
3. 接着交 T3、T4，先让用户能看模型。
4. 然后交 T5、T6，补上后台维护闭环。
5. 最后补 T7、T8，统一请求层和交付说明。
