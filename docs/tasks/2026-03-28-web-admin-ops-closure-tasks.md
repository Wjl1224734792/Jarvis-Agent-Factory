# 2026-03-28 Web/Admin 运营闭环与紧凑化任务拆分

## 1. 需求文档路径
- 无单独落盘需求文档；以上游主会话已完成的需求对齐、拟定计划与当前会话结论为准。
- 如后续需要补文档，建议另行落盘到 `docs/requirements/2026-03-28-web-admin-ops-requirements.md`，但本次 `planner` 可直接以本任务文档执行。

## 2. 任务概览
- 本轮目标分三段：用户端紧凑化与榜单评分收敛、设置页手机号换绑闭环、管理端运营工作台与审核发布闭环。
- 高风险共享面集中在 `packages/shared`、`packages/schemas`、`packages/http-client`、`packages/db`、`apps/server`，这些路径不要并行抢改。
- 本轮默认以“最小可交付闭环”为准，不扩展到无关重构。

## 3. 任务分解列表
| 任务 ID | 名称 | 类型 | 优先级 | test_strategy | 完成标准 |
|---|---|---|---|---|---|
| SC1 | 手机号换绑与审核开关契约 | 共享契约 | P0 | `tdd` | 新增手机号换绑请求/确认契约、图形码/短信码复用契约、后台站点审核开关契约；前后端可按同一 schema 解析。 |
| SC2 | 用户资料与手机号脱敏读模 | 共享契约 | P0 | `tdd` | 当前手机号仅以后四位脱敏展示所需的读模字段与辅助类型准备完毕，不暴露完整号码到设置页展示层。 |
| BE1 | 手机号换绑后端闭环 | 后端 | P0 | `tdd` | `apps/server` 支持新手机号验证码申请、确认换绑、唯一性校验与资料刷新；换绑后前台再次读取到新手机号。 |
| BE2 | 帖子审核开关与发布状态规则 | 后端 | P0 | `tdd` | 站点审核开关可控制普通用户文章/动态是否直发；开启时保持审核，关闭时自动发布；管理员发布官方文章始终可直发。 |
| BE3 | 官方文章发布与官方标签透出 | 后端 | P0 | `tdd` | 管理员可发布官方文章，首页/详情可识别官方作者或官方内容并透出标签；接口契约与列表/详情读模一致。 |
| BE4 | 飞行器投稿审核入口接线 | 后端 | P1 | `tdd` | 后台已有飞行器投稿审核能力在服务端保持可用，前端可按状态查询与审核，不破坏现有投稿流。 |
| FE1 | 顶栏用户入口改为个人中心入口 | 前端(web) | P0 | `test_after` | 顶部头像+用户名区域整块可点击并进入个人中心；移动端与桌面端都保留可访问性，退出按钮独立保留。 |
| FE2 | 榜单评分展示收敛 | 前端(web) | P0 | `test_after` | 榜单卡片本体不再显示星星和评分；榜单条目/详情中的评分数字统一放大并使用评分蓝，榜单详情条目将星星与分数移动到右侧，评分在星星上方。 |
| FE3 | 飞行器库紧凑布局 | 前端(web) | P1 | `test_after` | 分类/品牌/动力筛选占位显著缩小，首屏更多空间让给机型卡片；筛选仍可用、搜索仍可用、结果区不受破坏。 |
| FE4 | 设置页重排与手机号换绑入口 | 前端(web) | P0 | `test_after` | 设置页改为紧凑布局，不暴露同步后端文案；手机号区域只显示脱敏号码与换绑入口，提交后能刷新当前资料展示。 |
| FE5 | 首页官方标签展示 | 前端(web) | P1 | `test_after` | 首页文章流中官方内容显示官方标签，且不影响普通内容卡片。 |
| FE6 | 管理后台首页重做与图表 | 前端(admin) | P0 | `manual_only` | 后台首页具备清晰的操作入口、核心指标与至少一块趋势图/图表，管理员能一眼看懂去哪里处理审核、发布与投稿。 |
| FE7 | 管理后台官方文章发布与审核开关界面 | 前端(admin) | P0 | `manual_only` | 后台可直接发布官方文章，可开关帖子审核功能，并能进入飞行器投稿审核入口；界面文案面向 C 端/运营，不暴露“同步后端”等实现词。 |
| TEST1 | 用户端与后台闭环回归测试 | 测试 | P0 | `tdd` | 形成最小闭环回归集，覆盖首页、飞友圈、飞行器库、榜单、个人中心、消息、设置、登录、后台审核、官方发布与相关联动。 |

## 6. 风险任务
- `SC1`、`BE1`：涉及手机号唯一性、验证码复用、会话刷新，属于高风险契约与身份状态变更。
- `BE2`：审核开关是全局行为开关，必须避免影响管理员发布与飞行器投稿规则。
- `BE3`、`FE5`：官方标签透出要和作者角色/内容来源保持一致，避免前后端展示口径不一致。
- `FE6`、`FE7`：管理端视觉与信息架构改动较大，优先保证操作路径清晰，再做细节美化。

## 7. 文件所有权和共享路径提醒
- 共享路径：`packages/shared/src/index.ts`、`packages/schemas/src/*`、`packages/http-client/src/index.ts`、`packages/db/src/schema.ts`、`apps/server/src/modules/*`
- 前端专属路径：`apps/web/src/routes/*`、`apps/web/src/features/auth/*`、`apps/admin/src/features/*`、`apps/admin/src/components/admin-ui.tsx`
- 风险提示：涉及共享契约、路由前缀、数据库 schema 的任务不要并行抢改同一文件；`planner` 需要按依赖串行排期。

## 8. 推荐交付顺序
1. `SC1`、`SC2` 先定契约和读模。
2. `BE1`、`BE2`、`BE3`、`BE4` 串行完成后端闭环。
3. `FE4`、`FE1`、`FE2`、`FE3`、`FE5` 完成用户端体验收敛。
4. `FE6`、`FE7` 完成后台运营入口与发布界面。
5. `TEST1` 最后补齐端到端回归，确认闭环。
