## 2. 任务概览
- 本轮目标是把 web 端视觉密度、个人资料持久化、评分模型去除、榜单分类调整和 admin 路由兜底收敛成一组可串行交付的任务。
- 最小改动原则下，先收敛共享契约和数据库，再做后端实现，最后做前端页面和布局调整。
- 涉及 `packages/db`、`packages/schemas`、`apps/server`、`apps/web`、`apps/admin` 的共享路径，必须明确所有权，避免并行抢改同一文件。

## 4. DDD 分类

### 需要 DDD
- DB2 机型评论评分模型清理
- DB3 榜单评分模型清理
- SC2 机型评论契约去评分
- SC3 榜单契约去评分并改分类
- BE2 机型评论后端去评分
- BE3 榜单后端去评分并改分类输出

原因：这几项都在改核心业务规则、状态与聚合数据边界，不只是展示层改动。

### 不需要 DDD
- DB1、SC1、BE1
- FE1、FE2、FE3、FE4、FE5、FE6、FE7

原因：这些主要是布局、资料表单、路由兜底或简单持久化闭环，不涉及复杂领域规则编排。

## 7. 风险任务

- DB1/DB2/DB3 是高风险共享 schema 变更，任何回退都要依赖迁移和 seed 的可重复性。
- SC2/SC3 会影响 `apps/server`、`apps/web`、`apps/admin` 三端契约，若有残留评分字段，构建和运行时都会碎。
- BE1 不能只改前端草稿，必须真写回后端并保证其他使用 `userSummary` 的页面不被破坏。
- FE2 的 skeleton 如果和真实列数算法不一致，页面会继续出现跳动和错位。
- FE7 需要同时处理 admin 根路由和兜底错误体验，不能只补一个 `/` 跳转。
