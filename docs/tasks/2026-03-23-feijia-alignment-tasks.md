## 2. 任务概览

本轮目标不是继续扩写 PRD，而是把当前仓库收敛成一个可交付、可验证的 MVP 完整闭环。现有仓库已经有 `apps/web`、`apps/admin`、`apps/server`、`packages/shared`、`packages/schemas`、`packages/http-client`、`packages/db` 的基础骨架，因此任务优先级应该围绕共享契约收口、服务端领域收敛、前端体验完成度三条主线展开。

当前对齐范围内，真正要完成的是用户端浏览、机型点评、帖子互动、基础通知、后台治理和桌面端视觉完成度。榜单、私信、心愿单、认证账号、账号注销等不纳入本轮必交。

## 4. 共享契约优先顺序

1. `packages/schemas` 先收口身份与会话契约，再收口机型/点评契约，最后收口帖子/互动/通知/治理契约。
2. `packages/shared` 先固化路由常量和公共枚举，再补充纯工具函数。
3. `packages/db` 先落核心事实表和唯一约束，再补治理相关字段和索引。
4. `packages/http-client` 必须在契约稳定后再封装，否则容易把错误结构和字段映射写成第二套业务规则。
5. `apps/server/src/app.ts` 只在契约稳定后再挂载/调整新模块，避免路由常量和服务端实现同时漂移。

## 5. DDD 分类

### 必须 DDD

- P2 数据模型、迁移与种子收口
- S1 服务端认证与身份域
- S2 服务端机型与点评域
- S3 服务端帖子与基础互动域
- S4 服务端后台治理域

原因：

- 这些任务都涉及明确的领域规则、状态转换、权限判断或唯一性约束。
- 它们跨前后端共享事实模型，必须由统一领域约束驱动，而不是 UI 侧各自解释。
- 一旦拆成松散 CRUD，后面会在审核、互动一致性和权限上回补成本。

### 不强制 DDD

- P1 共享契约与路由常量收口
- P3 统一请求层与错误映射
- W1 Web 视觉系统与壳层升级
- W2 Web 核心页面闭环
- W3 Web 响应式与状态体验收口
- A1 Admin 壳层与概览升级
- A2 Admin 管理页面闭环
- A3 Admin 状态与审核体验收口

- 这些任务一旦字段名、返回结构或状态机偏差，前后端会同时回归。

## 7. 风险任务

- P1 是最高风险共享区，任何字段名、枚举值、分页参数或错误结构不一致，都会同时影响 `web`、`admin`、`server`。
- P2 涉及唯一约束、级联删除和治理字段，最容易在迁移和索引上出问题。
- S1 会影响整站身份态，必须先稳定再扩展其他需要登录的页面。
- S2/S3/S4 会共享部分事实模型和计数逻辑，不能各自写一套“看起来能用”的规则。
- P3 如果写成第二套业务层，会直接破坏共享契约层的边界。
- W1/W2 不能先硬编码服务端返回结构，否则后续接真实接口会重构页面状态模型。
- A2 不能绕过 `packages/http-client` 直连实现细节，否则后台和前台会出现不同治理口径。

## 8. 文件所有权和共享路径提醒

以下区域必须单线程收敛，不能并行抢改：

- `packages/shared/src/index.ts`
- `packages/schemas/src/*`
- `packages/db/src/schema.ts`
- `packages/db/src/seed.ts`
- `packages/db/drizzle/*`
- `packages/http-client/src/index.ts`
- `apps/server/src/app.ts`
- `apps/server/src/modules/*`
- `apps/web/src/app.tsx`
- `apps/web/src/features/auth/*`
- `apps/web/src/routes/*`
- `apps/web/src/store/*`
- `apps/admin/src/app.tsx`
- `apps/admin/src/features/*`
- `docs/tasks/*`

边界提醒：

- `packages/shared` 只放路由常量、公共枚举和纯工具。
- `packages/schemas` 只放请求/响应契约和校验，不放实现逻辑。
- `packages/db` 只放表结构、迁移、种子和仓储基础能力。
- `packages/http-client` 只放请求封装和错误映射，不写业务判断。
- `apps/server` 统一承载领域规则、权限、读写聚合和治理接口。
- `apps/web` 只负责展示、交互和状态同步，不直连服务端内部模块。
- `apps/admin` 只负责治理界面，不绕过请求层。

## 9. 推荐交付顺序

1. 先做 `P1`，锁定全局契约和路由常量。
2. 再做 `P2`，把数据模型、迁移和唯一约束收口。
3. 接着做 `P3`，统一前端请求入口和错误映射。
4. 然后并行推进 `S1`、`S2`、`S3`、`S4`，前提是契约已稳定。
5. 在服务端契约稳定后，并行推进 `W1` 和 `A1` 的壳层与视觉升级。
6. 再推进 `W2` 和 `A2` 的真实业务闭环。
7. 最后补 `W3` 和 `A3` 的响应式、分页、加载态与审核态收口。

## 10. 推荐的下一步

把这份拆解交给 planner，先锁定：

- `P1 -> P2 -> P3` 的共享契约串行顺序
- `S1/S2/S3/S4` 是否按领域并行推进
- `W1/A1` 是否允许先做静态视觉壳层
- 每个任务的最小验证命令和回归点
