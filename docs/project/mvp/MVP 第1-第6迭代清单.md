# MVP 第1迭代：项目基础骨架

**目标：** 把整个系统搭成“可持续接功能”的基础盘。

这一轮只需要形成基础工程闭环：

- `web / admin / server` 三端骨架可运行
- 共享契约、共享请求层、共享 UI 基础能力就位
- 基础路由壳、布局壳、全局状态壳就位
- 数据库连接、迁移、种子、对象存储适配基础打通
- OpenAPI、环境变量校验、统一错误结构建立

这一步不是业务功能，但必须先做，因为你的工程约束已经明确：前后端共享类型必须来自统一 schema，服务端按业务域拆模块，前端也必须走统一请求层和组件基线。

**这一轮结束标志：**
系统能启动、能请求、能鉴权预留、能迁移数据库、能给后续功能稳定落位。

------

# MVP 第2迭代：账号与身份体系

**目标：** 先解决“用户怎么进入系统”。

建议这一轮完成：

- 登录 / 注册
- 手机号 + 验证码登录主链路
- 图形验证码防刷
- 基础用户资料
- 登录态持久化
- 顶部导航用户入口
- 退出登录
- 个人中心最小可用入口

因为 PRD 已经把登录注册列为基础模块，而且后面的评分、收藏、发帖、关注、消息都依赖身份体系。

**这一轮结束标志：**
用户可以稳定登录，系统能识别身份，后续所有“需要登录”的动作都能接上。

------

# MVP 第3迭代：飞行器库主链路

**目标：** 让平台先具备“专业数据库”价值。

建议这一轮完成：

- 飞行器库列表
- 飞行器类型分类
- 品牌维度筛选
- 动力类型筛选
- 机型详情页
- 参数展示
- 基础搜索入口
- 后台最小机型管理闭环
- 分类管理最小闭环

因为你的产品核心定位不是泛社区，而是“飞行器数据库 + 用户口碑社区”。机型库必须尽早上线，否则平台没有内容锚点。PRD 对 `/models` 的筛选和详情要求也很明确。

**这一轮结束标志：**
用户已经能查机型、看参数、按类型和品牌筛选，后台也能维护基础机型数据。

------

# MVP 第4迭代：评分与点评体系

**目标：** 把“数据库”升级成“数据库 + 真实口碑”。

建议这一轮完成：

- 机型评分
- 机型点评
- 单用户对单机型唯一点评记录
- 点评列表
- 综合评分展示
- 快速评分入口
- 点评和评分的基础审核 / 风控处理
- 后台对点评内容的最小治理能力

PRD 明确把“真实用户评分与点评”定义为核心场景，而且综合评分、快速评分、唯一点评记录都已经写清楚了。

**这一轮结束标志：**
每个机型不只是参数页，而是已经有用户评分、点评和可展示的综合口碑。

------

# MVP 第5迭代：首页信息流

**目标：** 让已有内容开始流动起来，形成主站入口。

建议这一轮完成：

- 首页 `/home`
- 推荐 / 最新 等基础 Tab
- 内容流卡片
- 机型相关内容在首页的承接展示
- 基础热度排序
- 基础全局搜索入口接入首页
- 首页发布入口露出

PRD 里首页信息流是主入口，并且推荐策略允许前期先走“热度 + 时间衰减”的简化方案，所以非常适合放在 MVP 中期。

**这一轮结束标志：**
首页不再只是导航落点，而是一个可消费内容、可跳转核心功能的主站入口。

------

# MVP 第6迭代：发帖与评论主链路

**目标：** 让社区真正开始生产内容。

建议这一轮完成：

- 发帖子
- 图文内容发布
- 帖子详情页
- 评论
- 回复
- 删除自己的内容
- 举报入口
- 发布后的审核状态流转
- 后台内容审核最小闭环
- 审核结果通知入口预留

PRD 已明确发布功能、帖子类型、评论、审核、举报、后台内容审核等能力；同时非功能要求里也强调 UGC 要做敏感词和图片审核。

**这一轮结束标志：**
用户可以发布、查看、讨论内容，平台初步具备社区活性。

------

# 这 6 轮之后，MVP 实际形成了什么

做完第 1 到第 6 轮，平台已经具备最核心的 MVP 价值：

- 用户能注册登录
- 用户能查飞行器
- 用户能给机型评分点评
- 用户能在首页看到内容流
- 用户能发帖和评论
- 后台能维护机型与审核基础内容

也就是说，这时产品已经完成了从 **“工具型数据库”** 到 **“带社区互动的垂直平台”** 的最小闭环。

------

# 第7轮以后再接什么最顺

你后面最自然的顺序就是：

**第7轮：点赞 / 收藏 / 分享 / 举报完善**
**第8轮：榜单体系**
**第9轮：关注与用户主页**
**第10轮：消息系统**
**第11轮：后台完善**
**第12轮：性能、安全、异步一致性增强**
