# MVP Roadmap（第1-第6迭代）

## 1. 文档目标

本文档用于定义飞加项目在 MVP 阶段前 6 个迭代的里程碑顺序、阶段目标、完成定义与主要风险，作为产品、设计、研发与 Agent 协作时的统一阶段路线图。

本文档不展开到接口级、页面级、任务级实现细节，只定义：
- 这一轮为什么做
- 这一轮做完算什么状态
- 这一轮对下一轮提供什么基础
- 这一轮最需要警惕的风险是什么

---

## 2. 迭代总原则

MVP 阶段按以下价值链推进，而不是按页面并行平铺开发：

**基础骨架 → 账号身份 → 飞行器库 → 评分点评 → 首页信息流 → 发帖评论**

这样安排的原因是：
1. 先把工程底座和契约层搭稳，避免后续反复返工
2. 先建立用户身份，再承接一切需要登录的行为
3. 先把飞行器库做成平台核心资产，再叠加口碑价值
4. 先让首页能承接内容，再做社区生产和讨论
5. 让 MVP 形成“可登录、可查机型、可看口碑、可发内容、可讨论”的最小闭环

---

## 3. 迭代里程碑总览

| 迭代 | 主题 | 一句话目标 |
|---|---|---|
| 第1迭代 | 基础骨架 | 把 web / admin / server 和共享包底座搭稳 |
| 第2迭代 | 账号与身份体系 | 解决用户进入系统并形成身份态 |
| 第3迭代 | 飞行器库主链路 | 让平台先具备专业数据库价值 |
| 第4迭代 | 评分与点评体系 | 把机型库升级为“参数 + 真实口碑” |
| 第5迭代 | 首页信息流 | 让主站成为内容消费入口 |
| 第6迭代 | 发帖与评论主链路 | 跑通社区内容生产与讨论闭环 |

---

## 4. 第1迭代：基础骨架

### 4.1 目标
把 `web / admin / server` 和共享包底座搭稳，让后续功能都能挂在统一工程基线上。

### 4.2 输入
进入本轮前，已经明确：
- 使用 Bun Workspace
- 使用 React 技术体系
- 使用 Hono、Zod、Drizzle、PostgreSQL、Redis、MinIO/S3 抽象
- 使用 monorepo 目录边界
- `apps/mobiles` 仅占位，不纳入当前开发

### 4.3 输出
本轮结束后应沉淀：
- `apps/web`、`apps/admin`、`apps/server` 可独立启动
- `apps/mobiles` 完成工程占位
- `packages/config`
- `packages/shared`
- `packages/schemas`
- `packages/ui`
- `packages/db`
- `packages/storage`
- `packages/http-client`
- OpenAPI 基础能力
- 统一错误结构
- `.env.example`
- 数据库迁移与 seed
- Docker 本地开发环境链路可用

### 4.4 完成定义
满足以下条件即视为完成：
- 三端都能启动
- 共享包边界已固定
- 前后端统一契约已可用
- 本地数据库、缓存、对象存储可跑通
- 后续业务功能开发不需要再回头大改基础结构

### 4.5 对下一轮的依赖价值
为账号与身份体系提供：
- 统一请求入口
- 统一鉴权接入点
- 统一 schema 契约
- 可落库的数据环境
- 可复用应用壳

### 4.6 风险提醒
这一轮最大风险不是“功能少”，而是如果契约层、目录边界、存储抽象和服务边界没锁住，后面每一轮都会反复返工。

---

## 5. 第2迭代：账号与身份体系

### 5.1 目标
先解决“用户怎么进入系统并形成身份态”。

### 5.2 输入
需要第1轮已经具备：
- 应用壳
- 统一请求层
- 服务端鉴权入口
- 数据库与缓存基础能力
- 统一 schema 契约

### 5.3 输出
本轮结束后应沉淀：
- 手机号 + 短信验证码登录/注册主链路
- 图形验证码防刷能力
- Token / 会话保持
- 基础用户资料
- 顶部导航登录入口
- 退出登录链路
- 受保护接口的身份识别能力
- 登录注册与 Token 的基础测试

### 5.4 完成定义
满足以下条件即视为完成：
- 用户可以稳定登录/注册
- 刷新页面后身份不丢失
- 未登录与已登录状态显示正确
- 服务端可识别受保护请求身份
- 后续所有“需登录动作”都可以直接接入当前身份体系

### 5.5 对下一轮的依赖价值
为飞行器库中的用户行为入口提供挂载点：
- 收藏
- 想买
- 写点评
- 发布内容
- 进入个人中心

### 5.6 风险提醒
这一轮最大风险是把登录只做成单独页面，而没有做成系统级身份能力，导致后面的评分、发帖、收藏、审核都要重新补鉴权。

---

## 6. 第3迭代：飞行器库主链路

### 6.1 目标
让平台先具备“专业飞行器数据库”的核心价值。

### 6.2 输入
需要第2轮已经具备：
- 稳定身份体系
- 统一接口契约
- 数据表与基础后台能力
- 可维护的分类与机型数据结构

### 6.3 输出
本轮结束后应沉淀：
- `/models` 路由
- 飞行器列表
- 飞行器类型筛选
- 品牌筛选 / 品牌索引
- 动力类型筛选
- “不限”重置筛选
- 机型详情页
- 核心参数展示
- 详细参数展示
- 机型详情页中的行为入口位
- 后台机型管理最小闭环
- 后台分类管理最小闭环

### 6.4 完成定义
满足以下条件即视为完成：
- 用户可以浏览机型列表
- 用户可以按类型、品牌、动力类型筛选
- 用户可以进入详情页查看参数
- 后台可以维护最基础的机型与分类数据
- 平台已经可以被当作“飞行器数据库工具”使用

### 6.5 对下一轮的依赖价值
为评分与点评体系提供明确承载对象：
- 所有评分都对应具体机型
- 所有点评都对应具体机型
- 综合评分展示有固定详情页承接

### 6.6 风险提醒
这一轮最大风险是只做展示页不做数据维护闭环，那后面的评分点评会缺少稳定对象，后台也无法持续供数。

---

## 7. 第4迭代：评分与点评体系

### 7.1 目标
把机型库从“参数库”升级为“参数 + 真实口碑”的核心资产。

### 7.2 输入
需要第3轮已经具备：
- 机型列表与详情页
- 用户身份体系
- 服务端写操作能力
- 后台最小治理能力

### 7.3 输出
本轮结束后应沉淀：
- 用户对机型评分
- 用户对机型写点评
- 单用户单机型唯一点评记录
- 机型详情页点评列表
- 综合评分展示
- 快速评分入口
- 后台异步计算或最终一致性处理
- 后台对点评内容的最小治理能力
- 评分一致性相关测试

### 7.4 完成定义
满足以下条件即视为完成：
- 用户可以对机型留下真实可见的评分和点评
- 同一用户不会对同一机型生成多条点评
- 综合评分可被稳定展示
- 评分与计数在并发下不会明显错乱
- 机型详情页已经成为“参数 + 口碑”页面

### 7.5 对下一轮的依赖价值
为首页信息流提供高价值内容源：
- 机型口碑内容
- 评分热度信号
- 热门点评内容
- 后续榜单数据基础

### 7.6 风险提醒
这一轮最大风险是评分、计数和综合分更新一致性处理不到位，后续首页、榜单、推荐都会被脏数据放大。

---

## 8. 第5迭代：首页信息流

### 8.1 目标
让主站从“能查数据”变成“能持续消费内容”的入口。

### 8.2 输入
需要第4轮已经具备：
- 账号体系
- 飞行器库
- 评分点评内容
- 基础热度信号
- 可承接的详情页与跳转路径

### 8.3 输出
本轮结束后应沉淀：
- `/home` 路由
- 推荐 Tab
- 最新 Tab
- 基础内容卡片流
- 首页发布入口
- 基础排序逻辑
- 首页承接机型相关内容与口碑内容的能力

### 8.4 完成定义
满足以下条件即视为完成：
- 首页已经不是空壳
- 用户进入首页后可以浏览内容
- 用户可以从首页进入详情页
- 首页可以为后续帖子分发提供入口
- 即使推荐不够智能，也已经形成基础内容流

### 8.5 对下一轮的依赖价值
为发帖与评论提供：
- 分发入口
- 曝光入口
- 内容消费入口
- 发布入口的主站位置

### 8.6 风险提醒
这一轮最大风险是过早追求复杂推荐算法，反而忽略首页最核心的问题其实是“先有内容承接、先有分发入口”。

---

## 9. 第6迭代：发帖与评论主链路

### 9.1 目标
把社区的内容生产和讨论能力真正跑通。

### 9.2 输入
需要第5轮已经具备：
- 用户身份体系
- 首页主入口
- 统一上传/存储能力
- 服务端审核流入口
- 后台内容治理边界

### 9.3 输出
本轮结束后应沉淀：
- 发帖能力
- 图文内容承载
- 视频内容最小承载
- 帖子详情页
- 评论
- 回复
- 删除自己的内容
- 举报入口
- 基础敏感词处理
- 审核状态流转
- 后台内容审核最小闭环
- 上传签名与审核状态流转相关测试

### 9.4 完成定义
满足以下条件即视为完成：
- 用户可以发布内容
- 用户可以查看帖子详情
- 用户可以评论和回复
- 用户可以删除自己的内容并发起举报
- 后台可以完成最基础内容审核
- 平台已经形成最小社区生产与讨论闭环

### 9.5 对后续迭代的依赖价值
为第7轮以后提供真实社区基础：
- 点赞
- 收藏
- 分享
- 榜单扩展
- 关注关系
- 消息系统
- 用户主页
- 创作者成长能力

### 9.6 风险提醒
这一轮最大风险是上传、审核、敏感词、状态流转没有形成完整链路，导致内容虽然“能发”，却无法稳定上线与治理。

---

## 10. 前6轮完成后的 MVP 定义

完成第1-第6轮后，MVP 应达到以下状态：

- 用户可以登录进入系统
- 用户可以浏览飞行器库并查看参数
- 用户可以查看真实评分与点评
- 首页已经成为内容消费入口
- 用户可以发布帖子并参与评论讨论
- 后台能够进行最基础的数据维护和内容审核

这意味着产品已经形成了：

**专业数据库 + 用户口碑 + 基础社区互动**

的最小闭环。

---

## 11. 阶段门判断语句

为了便于排期和阶段评审，每轮收口时只问一句：

- 第1轮：底座是否已经稳定到后续不再频繁返工？
- 第2轮：用户是否已经可以稳定登录并形成身份态？
- 第3轮：平台是否已经可以被当作飞行器数据库使用？
- 第4轮：机型页面是否已经具备真实评分与点评口碑？
- 第5轮：首页是否已经成为可消费内容的主入口？
- 第6轮：用户是否已经可以发帖、评论，后台是否能做基础审核？

只要每轮都能回答“是”，就可以进入下一轮。

---

## 12. 第7轮以后建议顺序

在 MVP 前 6 轮完成后，建议后续顺序为：

1. 点赞 / 收藏 / 分享 / 举报完善
2. 榜单体系
3. 关注与用户主页
4. 消息系统
5. 后台完善
6. 性能、安全与一致性增强

原因是这些能力都更适合建立在已有内容、已有评分、已有首页分发和已有社区生产基础之上。
