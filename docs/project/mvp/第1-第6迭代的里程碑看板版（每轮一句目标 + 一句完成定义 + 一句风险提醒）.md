整体顺序仍然遵循你这个项目的核心价值链：**先底座，再身份，再机型库，再口碑，再信息流，再社区生产**。

| 迭代                          | 一句目标                                                     | 一句完成定义                                                 | 一句风险提醒                                                 |
| ----------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **第1迭代：基础骨架**         | 把 `web / admin / server` 和共享包底座搭稳，让后续功能都能挂在统一工程基线上。 | 三端可启动，`schemas / db / storage / http-client / ui / config` 基础结构可用，OpenAPI、Zod 校验、统一错误结构、迁移与本地容器链路跑通，就算完成。 | 这一轮最大风险不是功能少，而是如果前后端契约、存储抽象、目录边界没锁住，后面每一轮都会反复返工。 |
| **第2迭代：账号与身份体系**   | 先解决“用户怎么进入系统并形成身份态”。                       | 用户能通过手机号加短信验证码登录/注册，登录态可持久化，受保护接口能识别身份，导航中的登录/退出链路正常，就算完成。 | 这一轮最大风险是把登录做成单页功能而不是系统能力，导致后面的评分、发帖、收藏、审核都要重新补鉴权。 |
| **第3迭代：飞行器库主链路**   | 让平台先具备“专业飞行器数据库”的核心价值。                   | 用户能进入 `/models`，完成类型、品牌、动力类型筛选，查看机型详情与参数，后台还能维护机型和分类数据，就算完成。 | 这一轮最大风险是只做展示页不做数据维护闭环，那后面的评分点评会缺少稳定承载对象，后台也无法持续供数。 |
| **第4迭代：评分与点评体系**   | 把机型库从“参数库”升级成“参数 + 真实口碑”的核心资产。        | 用户能对机型评分和点评，单用户单机型只有一条记录，机型详情页能展示综合评分和点评列表，后台能做最小治理，就算完成。 | 这一轮最大风险是评分和计数一致性处理不到位，因为 PRD 已明确点赞、评分、评论存在异步计数与最终一致性要求。 |
| **第5迭代：首页信息流**       | 让主站从“能查数据”变成“能持续消费内容”的入口。               | `/home` 可用，推荐/最新等基础 Tab 可用，内容卡片流能加载和跳转，首页有明确发布入口，就算完成。 | 这一轮最大风险是过早追求复杂推荐算法，反而忽略了首页首先要解决的是“有内容承接、有入口分发”。 |
| **第6迭代：发帖与评论主链路** | 把社区的内容生产和讨论能力真正跑通。                         | 用户能发布图文/视频帖子、查看帖子详情、进行评论回复和举报，后台能完成最小审核闭环，就算完成。 | 这一轮最大风险是 UGC 审核、敏感词、上传签名和状态流转没打通，因为这些在工程约束里都属于关键链路，不是上线后再慢慢补的边角项。 |

再压缩成一句看板总览，就是：

**第1轮搭底座， 第2轮建身份， 第3轮做机型库， 第4轮补评分点评， 第5轮起首页流， 第6轮通发帖评论。** 这样做完后，你的 MVP 就已经具备了 **登录、查机型、看口碑、逛首页、发内容、做讨论** 的主闭环。
