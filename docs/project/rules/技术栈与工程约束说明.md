# 技术栈与工程约束说明

## 1. 目标

本文档用于统一本项目的基础技术栈、工程约束、目录边界、共享包职责与开发协作规范，作为后续人工开发、AI Agent 执行、代码审查与项目初始化的共同基线。

本项目定位为：

* PC 端优先的 Web 应用
* 兼顾移动端体验
* 包含用户端、管理后台、后端服务
* 当前阶段预留移动端工程位，但不实现独立移动端应用

## 3. 核心工程原则

### 3.1 单一技术基线原则

* 全项目必须基于 Bun Workspace 管理
* 所有应用共享统一 TypeScript、Lint、格式化与环境变量校验规则
* 前后端共享类型时，必须优先从共享 Schema 派生，不允许多份手写重复定义
* 所有新增能力默认遵循本文档，不允许局部自行引入另一套风格

### 3.2 样式原则

* 必须使用 Tailwind CSS
* 必须使用内联类名
* 禁止以大规模独立 CSS 文件、BEM、Sass Module 作为主要样式组织方式
* 允许少量全局样式，仅用于 reset、主题变量、第三方兼容修正
* 页面样式、组件样式、状态样式优先通过 class 组合函数与 variant 组织

### 3.3 前端状态原则

* Zustand 只存放跨页面共享状态、身份态、全局 UI 状态
* 列表数据、详情数据、分页数据、缓存数据优先由 TanStack Query 管理
* 页面内短生命周期状态优先本地管理
* 禁止把所有接口返回数据都堆进 Zustand

### 3.4 后端架构原则

* 后端采用模块化单体
* 按业务域拆模块，而不是按 controller/service/utils 随意平铺
* 路由层只负责协议适配与调用协调
* 业务规则放在 service
* 数据访问放在 repository
* 涉及权限、幂等、消息投递、审核状态流转的逻辑必须可测试

### 3.5 存储原则

* 所有对象存储必须经统一 S3 适配层访问
* 业务代码禁止直接依赖具体云厂商 SDK
* 上传、删除、签名 URL、读取元数据等能力必须统一封装
* 开发环境默认 MinIO，生产环境兼容 OSS / COS / KODO

## 4. 本地开发环境原则

保留当前根目录 `docker/` 目录，用于维护开发环境依赖。

开发环境基础能力统一走容器：

* PostgreSQL
* Redis
* MinIO

约束如下：

* 不要求开发者手工安装数据库、缓存、对象存储
* 所有端口、账号、密码、Bucket、连接串通过 `.env` 管理
* 必须提供 `.env.example`
* 必须提供数据库迁移脚本与测试种子脚本
* 当前项目保持 `docker/` 根目录设计，不额外搬迁目录结构

这个约束与现有文档里“本地依赖统一容器化”的基线一致。

## 5. 数据层约束

### 5.1 PostgreSQL

PostgreSQL 负责：

* 主业务数据库
* 发布 / 订阅
* 轻量消息队列

约束：

* 核心业务数据以 PostgreSQL 为主
* 轻量异步任务优先基于 PostgreSQL 队列模型
* 真正高吞吐场景再评估独立 MQ，不提前引入复杂度

### 5.2 Redis

Redis 负责：

* 缓存
* 限流
* 热点计数
* 短时状态标记
* 分布式锁（谨慎使用）

### 5.3 Drizzle ORM

* 必须作为主要 ORM
* 必须使用连接池
* Schema、迁移、查询、仓储代码分层管理
* 非必要不得在业务代码中散落原始 SQL

## 6. 接口与契约约束

* 所有请求参数、响应体、环境变量必须优先用 Zod 校验
* 所有外部接口必须补齐 Swagger / OpenAPI
* Web、Admin、Mobiles 使用的接口契约必须来自统一定义
* 错误返回结构必须统一
* 分页、列表查询、排序参数保持统一命名风格

## 7. 测试约束

### 7.1 必测

* 登录注册与 Token 流程
* 权限判断
* 审核状态流转
* 评分、点赞、收藏等数据一致性逻辑
* 文件上传与签名地址流程
* 队列消费、失败重试、幂等处理
* 关键 Zod 校验逻辑

### 7.2 可后补

* 纯展示组件
* 简单静态页面
* 低风险样式调整

## 8. 移动端预留约束

根据 PRD，本项目必须兼顾移动端体验，因此工程上要预留移动端位置；但当前阶段不实现独立移动端应用，只保留工程位与共享契约位。

约束如下：

* 预留 `apps/mobiles`
* 当前不纳入正式开发范围
* 当前不纳入 CI 必测矩阵
* 当前不建立独立业务实现
* 仅预留目录、README、占位配置与共享依赖入口

未来若启动移动端，优先延续 React 技术体系，建议采用：

* Ionic React + Capacitor

原因：

* 与 React / Zustand / React Router / Zod 共享认知一致
* 可以最大化复用 API 契约、状态管理方式与一部分 UI 设计语言
* 不会过早把团队拉进双技术栈维护
