# 2026-04-17 Web 与 Server 审查优化需求说明

## 需求摘要

- 使用 `agent-orchestration` 与 `spawn` 多代理流程，对 `apps/web` 与 `apps/server` 进行分阶段代码审查与优化。
- 执行顺序固定为：先审查 `apps/web`，确认问题清单后完成前端最小优化；再审查 `apps/server`，确认问题清单后完成后端最小优化。
- 审查维度覆盖：性能、代码质量、模块结构、可维护性、重复逻辑、功能逻辑正确性、明显的运行风险与可验证性。
- 仅做与本轮审查直接相关的最小正确改动，不做无关重构，不扩展到 `apps/admin` 或仓库外应用。

## 范围内

- `apps/web`
- `apps/server`
- 与修复直接相关的共享包或测试文件，但仅限用户请求涉及的问题链路
- `docs/requirements`、`docs/tasks`、`docs/plans`、`docs/review` 中与本轮编排直接相关的文档

## 范围外

- `apps/admin` 的专项审查与优化
- 数据库 schema、迁移、seed 语义调整，除非某个高优问题无法绕开且需单独升级为计划补丁
- 无关的 UI 重设计、技术栈迁移或大规模目录重构
- 微信小程序、Flutter App、`apps/mobiles`

## 成功标准

- 输出一份前端审查结论，明确列出已确认问题、影响、建议和本轮处理范围。
- 完成前端最小优化，并说明哪些问题已修复、哪些问题仅记录未处理以及原因。
- 输出一份后端审查结论，明确列出已确认问题、影响、建议和本轮处理范围。
- 完成后端最小优化，并说明哪些问题已修复、哪些问题仅记录未处理以及原因。
- 至少运行与本轮改动直接相关的验证；如环境限制导致无法完成默认全量验证，需要明确记录原因。

## 约束

- 优先沿用现有应用分层与共享包，不在应用层重复定义共享结构。
- 共享协议、环境变量、数据库语义若需变更，必须先检查相关包和文档，并在必要时升级为计划补丁。
- 审查结论要优先关注真实风险、行为回归和验证缺口，不输出空泛建议。
- 默认收尾考虑：`bun run lint`、`bun run typecheck`、`bun run test`、`bun run build`。

## 已对齐假设

- 用户允许直接落地 `apps/web` 与 `apps/server` 的最小优化代码，并执行收尾验证。
- 本轮以仓库当前状态为准，不回滚历史实现，不处理与前后端审查无直接关系的问题。
