# 七牛云 Kodo 对象存储接入需求

日期：2026-04-21

适用范围：`apps/server`、`packages/shared`、`packages/schemas`、`packages/http-client`、`packages/db`、根目录文档与环境变量说明

## 1. 背景

- 主会话已明确需求：在当前仓库中接入七牛云 Kodo 对象存储，并先按编排流程完成任务拆解，不直接编写业务代码。
- 本轮输出必须覆盖：仓库审查、接口与契约评估、服务端实现、配置与环境变量、文档、测试与验证。
- 任务设计阶段不得自行扩展未经确认的产品范围；如发现需求外分支，只能记录为开放项或回退主会话澄清。

## 2. 目标

- 审查现有上传与对象存储架构，确认 Kodo 接入是在现有能力上补齐，还是需要调整共享契约与服务端实现。
- 输出一套可直接交给 `planner` 的任务拆解，明确依赖关系、测试策略和共享路径所有权。
- 确保后续实现阶段覆盖服务端上传初始化、直传、完成校验、读取地址生成，以及与配置、文档、验证闭环相关的改动。

## 3. 已确认范围

- 仓库现状审查：上传模块、存储 provider、共享 schema、HTTP client、运行时 seed、README、`.env.example`、现有测试覆盖。
- 接口与契约评估：`packages/schemas`、`packages/shared`、`packages/http-client` 与 `apps/server` 之间的上传契约与边界。
- 服务端实现范围：对象存储 provider 适配、上传链路校验、文件读取 URL 生成、服务端错误分支与回退行为。
- 配置范围：`STORAGE_*` 相关环境变量、默认值、云厂商说明、开发与生产差异。
- 文档范围：README、环境变量说明、接入说明、验证与排障说明。
- 测试与验证范围：单元测试、服务端测试、必要的真实环境或准真实环境校验策略。

## 4. 非目标

- 本轮不编写业务实现代码。
- 不默认扩展新的前端产品交互；除非接口契约评估证明现有前端上传调用无法复用，否则不主动拆出前端页面改造。
- 不默认重构现有 `files` 业务模型；仅在审查后确认元数据不足以支撑 Kodo 时，才升级为变更任务。
- 不把日志系统、短信系统或其它云厂商能力纳入本轮实现范围。

## 5. 成功标准

- 形成正式任务文档，能直接交给 `planner` 继续编排。
- 任务文档中明确哪些任务需要 DDD，哪些必须 TDD，哪些可以直接开发。
- 任务文档中明确共享路径、单一责任方、依赖关系、风险任务和推荐交付顺序。
- 任务文档能覆盖 Kodo 接入的服务端、配置、文档与验证闭环，不遗漏高风险共享契约。

## 6. 开放决策

- Kodo 接入是否只依赖当前的 S3 兼容能力，还是需要补 Kodo 专属兼容处理。
- 现有 `files` 表与上传响应是否已经足够，还是需要增加新的元数据或错误信息。
- 运行时 seed、测试数据、公开访问 URL 与私有签名读取策略是否需要因 Kodo 调整。
- 是否需要补充前端或 HTTP client 侧的特殊错误提示与回退逻辑。
