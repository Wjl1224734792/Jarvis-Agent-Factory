# Kodo 对象存储与七牛审核落地需求

日期：2026-04-21

适用范围：`apps/server`、`apps/admin`、`packages/schemas`、`packages/http-client`、`packages/shared`、`packages/db`、根目录配置与文档

说明：本需求文档覆盖并收敛此前仅针对 Kodo 对象存储接入的临时需求。后续任务拆解、执行规划与实现以本文为准。

## 1. 背景

- 仓库当前已经存在统一上传链路、`files` 数据表、后台审核页和站点审核开关。
- 对象存储目前处于多 provider 历史兼容状态，但本轮目标是正式收口为两种对象存储方式：
  - `minio`：仅用于本地测试 / 本地开发
  - `kodo`：用于开发环境与生产环境
- 内容审核能力本轮统一收口为七牛云：
  - 文本审核：调用七牛文本审核 API
  - 媒体审核：使用七牛对 Kodo 对象的审核能力，结合回调落库
- 后台现有“自动通过 / 人工审核”表述需要调整为“AI 审核 / 人工审核”模式，并且审核结果、审核记录、回调状态必须能在后台追踪。

## 2. 总目标

- 将对象存储 provider 收口为 `minio | kodo`，移除 `oss / cos / qiniu alias` 的产品级入口与文档承诺。
- 将 Kodo 实现固定为七牛官方 SDK 路线，不再通过 AWS S3 兼容实现 Kodo。
- 打通七牛审核能力：
  - 文本内容在服务端调用七牛文本审核 API
  - 图片 / 视频等对象内容走七牛审核能力并接入回调
- 将后台审核开关语义改为：
  - 开启：AI 审核
  - 关闭：后台人工审核
- 将审核过程、审核结果、失败状态、人工复核结果持久化，并在后台可见、可追踪、可复核。

## 3. 范围

### 3.1 对象存储

- 统一保留 `minio` 与 `kodo` 两种 provider。
- `minio` 继续服务本地开发与测试环境。
- `kodo` 作为开发 / 生产统一对象存储实现。
- Kodo 仅使用七牛官方 `qiniu` SDK 及对应上传 / 下载 / 查询能力。
- 与对象存储有关的上传初始化、直传、完成确认、文件读地址生成逻辑纳入本轮。

### 3.2 审核能力

- 文本审核纳入以下域：
  - 文章正文 / 标题
  - 动态正文
  - 评论正文
  - 评测正文
  - 机型投稿 / 品牌申请 / 榜单 / 评分对象等当前受审核内容中的文本字段
- 媒体审核纳入以下对象：
  - 头像图片
  - 帖子图片 / 视频
  - 排行榜封面 / 条目图片
  - 投稿图片 / 视频
  - 举报图片等上传到对象存储的媒体
- 图片 / 视频审核以七牛审核结果为准，不再引入其它云厂商审核源。

### 3.3 后台管理端

- 保留现有审核入口与审核页，不推翻现有后台 IA。
- 将审核卡片、设置页和说明文案从“自动通过 / 人工审核”调整为“AI 审核 / 人工审核”。
- 后台需要可查看：
  - 当前审核模式
  - 当前对象或内容的审核状态
  - 最近一次 AI 审核结果
  - 七牛回调结果或错误
  - 人工复核操作记录

### 3.5 用户通知

- 审核结果需要通过系统消息通知到用户。
- 至少覆盖以下最终结果：
  - AI 审核通过
  - AI 审核拒绝
  - 人工审核通过
  - 人工审核拒绝
- 通知内容需要能让用户理解：
  - 是哪一类内容被审核
  - 当前结果是什么
  - 是否需要用户修改 / 重新提交

### 3.4 数据与追踪

- 审核结果必须持久化，不能只靠运行时日志。
- 后台需要可追踪审核全链路：
  - 何时提交审核
  - 审核来源（文本 / 图片 / 视频）
  - 审核模式（AI / manual）
  - 审核状态（待审 / 通过 / 拒绝 / 回调失败 / 审核异常 / 待人工复核）
  - 七牛返回的关键结果与原始 payload 摘要
  - 最终人工处理结果与操作者

## 4. 功能需求

### R1. 对象存储 provider 收口

- 环境变量、provider 枚举、文档说明、测试覆盖、运行时 seed 行为必须统一为 `minio | kodo`。
- `kodo` 视为唯一正式的七牛对象存储 provider 名称。
- `minio` 作为本地开发 / 测试专用，文档中必须明确其定位。

### R2. Kodo 上传链路

- 上传初始化由服务端生成受限上传凭证。
- 客户端继续采用“拿凭证后直传对象存储，再调用 complete”的三段式流程。
- complete 阶段必须由服务端回查对象元数据并校验大小 / 类型 / 存在性。
- Kodo 的实现细节不得再依赖 S3 presigned PUT 逻辑。

### R3. 文本审核

- 服务端在内容创建 / 更新的关键写入节点接入七牛文本审核。
- 当对应域配置为 AI 审核时：
  - 文本进入 AI 审核流程
  - 根据审核结果决定内容是直接通过、转人工、还是拒绝
- 当对应域配置为人工审核时：
  - 不触发 AI 审核
  - 内容进入后台人工审核队列

### R4. 媒体审核

- 上传成功的媒体在 AI 审核模式下必须进入七牛审核流程。
- 媒体审核结果通过服务端回调或服务端确认机制落库。
- 在审核结果未通过前，媒体不得被当作“可公开展示资源”对外消费。
- 在人工审核模式下，上传完成的媒体进入人工审核队列。

### R5. 审核模式替换

- 当前各业务线已有的 moderation 开关语义需要改造：
  - 不再表示“是否人工审核”
  - 改为“是否启用 AI 审核”
- 关闭 AI 审核时的默认行为是进入后台人工审核，不是自动通过。
- 不允许保留“关闭审核 = 自动上线 / 自动通过”的旧逻辑。

### R6. 后台追踪与人工复核

- 后台应能看到 AI 审核与人工审核的统一状态。
- 对 AI 未通过、AI 不确定、AI 回调异常的对象，应支持人工复核。
- 管理员操作必须留下记录，便于审计和追踪。

### R7. 七牛回调接入

- 服务端需要提供用于接收七牛审核结果的回调入口。
- 回调结果需验证来源、解析 payload、更新审核记录与业务状态。
- 对本地开发无法直接接收外网回调的场景，文档中要给出真实环境或 tunnel 验证说明。

### R8. 审核结果通知用户

- 当内容审核进入最终状态时，系统需要向内容作者发送站内系统消息。
- 通知触发点应覆盖 AI 审核与人工审核两条路径。
- 通知内容需与后台最终状态保持一致，避免出现后台已拒绝、用户端仍无反馈的情况。
- 通知记录应能和审核记录关联，便于后台追踪。

## 5. 状态与数据要求

### 5.1 审核状态

- 本轮需要明确并统一“上传成功”与“审核通过可展示”是两件不同的事。
- 至少要支持以下语义：
  - 已上传待审
  - AI 审核中
  - AI 审核通过
  - AI 审核拒绝
  - AI 审核异常 / 回调失败
  - 待人工复核
  - 人工审核通过
  - 人工审核拒绝

### 5.2 审核记录

- 审核记录应至少包含：
  - 业务域
  - 业务对象 ID
  - 审核对象类型（text / image / video / mixed）
  - provider（固定为 qiniu）
  - mode（ai / manual）
  - 状态
  - 七牛 taskId / jobId / requestId 等可追踪标识（如有）
  - 原始结果摘要
  - 审核时间
  - 最终人工处理信息

## 6. 兼容性要求

- 现有后台审核页、审核待办统计、总览图表不能因这轮改造失效。
- 已有内容状态与审核页路由尽量保持兼容。
- 前端页面应继续通过共享 `http-client` 调用上传链路，不在页面层散改上传协议。

## 7. 非目标

- 本轮不引入新的第三方审核服务。
- 本轮不重做整套后台信息架构。
- 本轮不把所有审核结果写入日志文件后再反查，必须有业务级可追踪记录。
- 本轮不把 MinIO 升格为生产候选对象存储。

## 8. 风险与实现边界

- 七牛媒体审核回调需要公网可达地址；本地开发环境的完整回调验收可能需要 tunnel 或开发环境联调。
- 文本审核与媒体审核的返回模型不同，不能假设使用同一数据结构硬合并。
- 站点设置、后台 UI、审核待办统计、审核详情追踪属于共享边界，必须统一设计后再实现。
- 审核消息通知与现有站内通知系统相连，需核对是否复用既有通知域模型还是新增审核专用消息类型。

## 9. 成功标准

- 产品级对象存储只剩 `minio | kodo` 两种方式。
- 文本审核与媒体审核都落到七牛云能力上。
- AI 审核关闭后，内容进入后台人工审核而不是自动通过。
- 后台可追踪审核过程、审核结果和人工复核结果。
- 用户能收到审核最终结果的系统消息通知。
- `lint`、`typecheck`、`test`、`build` 能通过，且关键审核链路有对应测试。

## 10. 默认假设

- 本轮允许对数据库 schema、共享 schema、后台设置项与后台审核页做必要调整。
- 本轮优先追求“链路闭环可运行”，不要求一次性补齐所有运营化细节。
- 七牛测试环境凭据仅用于本地 / 测试验证，生产凭据后续替换。
