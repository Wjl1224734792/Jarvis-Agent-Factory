# 2026-04-19 全仓审查报告与优化路线图需求

## 需求摘要

- 使用 `agent-orchestration` 与 `spawn` 多代理流程，对仓库进行一次全仓技术审查。
- 本轮第一阶段目标是先输出“全仓审查报告”和“优化路线图”，不要求在同一阶段大规模直接改代码。
- 审查维度至少覆盖：
  - 架构与模块边界
  - 算法与数据流
  - 性能与扩展性
  - 工程质量与可维护性
  - 测试、验证与交付风险
- 在审查报告与路线图被确认后，再进入下一轮按路线图实施优化。

## 范围内

- `apps/web`
- `apps/admin`
- `apps/server`
- `packages/config`
- `packages/db`
- `packages/http-client`
- `packages/schemas`
- `packages/shared`
- 与全仓审查直接相关的工程脚本、根配置与文档

## 范围外

- 仓库外的小程序 / App / 其他独立代码库
- 没有明确证据支撑的大规模重写
- 在本轮报告阶段直接引入高风险架构迁移

## 阶段目标

### 阶段 1：全仓审查报告与路线图

- 盘点仓库当前真实状态，而不是泛化建议。
- 输出结构化审查报告，明确问题、影响、证据、优先级和建议方向。
- 输出分阶段优化路线图，说明哪些可以快速落地，哪些需要专项治理。

### 阶段 2：按路线图实施优化

- 仅在阶段 1 产物确认后启动。
- 按优先级逐步落地，不做与路线图无关的顺手重构。

## 审查维度

### 1. 架构

- monorepo 依赖方向是否清晰
- `apps/*` 与 `packages/*` 职责边界是否稳定
- 共享契约、路由常量、DB 层、应用层是否有职责穿透
- 模块拆分是否存在明显耦合、重复、热点文件失控

### 2. 算法

- 推荐 / 搜索 / 排序 / 聚合逻辑是否可解释、可维护、可扩展
- 关键业务规则是否集中，还是散落在 route / page / util 中
- 当前实现是否存在明显的错误激励、冷启动问题、重复曝光或无效计算

### 3. 性能

- 关键接口是否存在串行查询、N+1、候选集过窄或过宽、同步 I/O 等问题
- 前端是否存在明显的大包、重复请求、无效渲染、阻塞交互
- 测试、seed、迁移与日志链路是否存在拖慢开发与验证的基础设施问题

### 4. 工程质量

- 代码风格、注释、契约清晰度
- 测试覆盖与失败模式
- 文档与真实实现是否一致
- 是否存在“默认全局可变状态”“隐式耦合”“坏味道热点文件”

## 成功标准

- 输出一份全仓审查报告，至少包含：
  - 审查范围
  - 审查方法
  - 架构问题清单
  - 算法问题清单
  - 性能问题清单
  - 工程质量/测试/交付问题清单
  - 每条问题的严重度、影响面、证据路径、建议处理方式
- 输出一份优化路线图，至少包含：
  - 优先级分层（P0/P1/P2 或等价分层）
  - 分阶段目标
  - 推荐的执行顺序
  - 哪些任务适合并行，哪些任务必须串行
  - 哪些任务可能触发 plan patch / contract change request
- 本轮报告阶段优先追求“全、准、可执行”，不追求一次性全部修完。

## 约束

- 所有结论都应尽量附仓库内证据路径。
- 不输出空泛“建议优化架构/建议提高性能”式结论，必须尽量落到模块、文件、链路或模式。
- 审查时优先识别真实阻塞点、系统性问题和高杠杆改进点，而不是低价值风格挑刺。
- 如果某个问题需要依赖外部事实或线上指标，而仓库内无法证明，要明确标注“推断”。

## 已对齐假设

- 用户本轮优先要“报告与路线图”，不是立刻大规模编码。
- 用户后续会基于本轮路线图继续推进优化。
- 用户希望覆盖全仓，而不是只审查某一个 app。
