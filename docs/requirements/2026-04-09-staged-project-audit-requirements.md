# 分阶段项目审查与修复需求说明

## 需求摘要

- 按顺序对 `apps/web`、`apps/admin`、`apps/server` 做项目审查，每一阶段都需要先输出审查报告，再完成低风险高收益优化、测试验证、提交并推送到远程仓库，随后才能进入下一阶段。
- 审查维度覆盖代码健壮性、性能优化、架构边界、一致性与可维护性。
- 在三阶段完成后，补做 OpenAPI 文档完整性与时效性检查，必要时更新，同时优化根目录 `package.json` 中与数据库种子、测试数据注入相关的脚本分层。
- 需要使用浏览器自动化做至少一轮真实页面验证；用户已提供 `agent-browser`、Playwright 浏览器目录和测试缓存目录，可直接使用，无需再向用户确认。

## 目标与成功标准

- `web`、`admin`、`server` 三个阶段均产出独立审查结论，且至少修复一批高置信度问题。
- 每一阶段结束前完成该阶段相关测试与必要的浏览器/接口验证，并记录结果。
- 每一阶段结束后完成一次 git 提交并推送远程分支。
- OpenAPI 文档入口、内容与后端实际实现保持一致，若发现缺口则补齐。
- 根目录脚本清晰区分开发环境 seed、测试环境大批量测试数据脚本，以及“清空后再注入”的 reset 流程，避免混用。

## 范围内

- `apps/web`
- `apps/admin`
- `apps/server`
- 与上述变更直接相关的 `packages/shared`、`packages/http-client`、`packages/schemas`
- 根目录 `package.json`
- 与 OpenAPI、脚本说明直接相关的 `README.md`、`.env.example`（仅在确有环境变量或脚本使用说明变更时）
- 审查/实现/评审文档

## 范围外

- 新业务功能开发
- `apps/mobiles`、微信小程序、独立 App 工程
- 与本轮问题无关的大规模重构
- 无法在当前仓库内独立验证的外部基础设施改造

## 关键模块 / 功能列表

- Web：入口路由、认证链路、页面级懒加载、发布链路、富文本、分享、数据请求与缓存策略
- Admin：后台路由与权限边界、首屏与图表负载、表单和列表稳定性
- Server：应用入口、错误处理、服务/路由边界、OpenAPI 生成与文档一致性、脚本与数据流程
- 根脚本：`db:seed`、测试数据脚本、重置与批量注入流程

## 风险与开放问题

- 项目已存在昨日的审查与维护文档，本轮需避免重复修复同类问题，同时要确认是否仍有残留未闭环项。
- 浏览器自动化依赖本地服务、数据库和测试数据状态；若基础环境缺失，需要先最小化补齐启动条件。
- OpenAPI 更新可能触及后端文档组织结构，若发现需要较大拆分，应先以“确保最新与正确”为主，再决定是否继续做结构性整理。
- 根脚本调整会影响开发、测试和未来生产流程，必须确保命名语义清晰且 reset 行为不会误用。

## 已收敛结论

- 用户明确要求采用编排方式推进，并授权使用 `spawn` 与浏览器自动化。
- 用户明确要求不再等待确认，主会话可在合理假设下自主推进。
- 阶段顺序固定为：`web -> admin -> server -> openapi/scripts`。
- 每个阶段的最低交付顺序固定为：审查报告 -> 修复优化 -> 测试验证 -> 提交推送。
