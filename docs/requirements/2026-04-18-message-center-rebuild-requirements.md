# 2026-04-18 消息中心重建与主页联动需求说明

## 需求摘要

- 采用方案 C，对当前 `web` 端消息中心进行独立域重建，并同步改造 `server`、`packages/schemas`、`packages/http-client`、`packages/db` 以支撑新的消息模型。
- 新消息中心需要支持 4 个一级分类：
  - `点赞和收藏`
  - `新增关注`
  - `评论和@`
  - `系统消息`
- `系统消息` 必须能够承接内容发布与审核状态变化，例如文章、动态、机型投稿、榜单、品牌申请等内容的发布状态或审核结果。
- `web` 端消息页采用单列长列表布局，并使用成熟的虚拟列表技术实现高性能渲染。
- 在同一轮中对个人主页与他人主页进行功能审查与优化，补齐与新消息中心、关系状态、内容可见性、内容组织相关的缺漏。

## 已对齐假设

- 当前产品尚未上线生产环境，允许对通知数据结构、共享契约、数据库表结构和接口设计进行重构级调整。
- 本轮可以自主规划实施，不需要在执行过程中再次向用户征询方向选择。
- 允许以“新消息域优先”为原则，对现有通知表与返回结构做不兼容升级，只要同步完成前后端与测试迁移。
- 本轮目标是把消息中心与主页功能做成可交付状态，而不是仅输出设计稿或审查报告。

## 范围内

- `apps/web`
- `apps/server`
- `packages/schemas`
- `packages/http-client`
- `packages/db`
- 与本轮功能直接相关的测试、种子数据与文档

## 范围外

- `apps/admin` 专项功能扩展，除非为适配共享契约变更而做最小兼容修正
- 微信小程序、Flutter App、`apps/mobiles`
- 非消息中心/主页链路的无关视觉重设计
- 无关的技术栈迁移或项目级重构

## 目标与成功标准

- 建立一套新的消息域模型，能表达互动消息、关注消息、评论/@ 消息和系统消息。
- `server` 提供与新模型对应的通知接口，返回适合 `web` 端直接渲染的统一消息结构。
- `web` 端消息页完成重做：
  - 顶部可切换 4 个一级分类
  - 列表为单列虚拟长列表
  - 支持未读态、批量已读、分组统计、空态和错误态
  - 不同消息项支持正确跳转到帖子详情、评论上下文、用户主页或发布状态详情
- `系统消息` 能展示内容发布/审核状态，并可定位到对应内容或状态页。
- 个人主页与他人主页完成审查与优化：
  - 个人主页补齐与消息中心联动的信息入口与内容组织
  - 他人主页补齐关系状态、资料与内容可见性下的交互缺口
  - 明确并修复本轮审查中发现的真实功能缺漏
- 完成与本轮变更直接相关的测试与验证，并尽量通过仓库默认验证。

## 功能分解

### 1. 消息域重建

- 定义新的消息类型、消息分类、目标对象类型、系统消息载荷结构。
- 为消息项提供统一字段：
  - `id`
  - `category`
  - `type`
  - `isRead`
  - `createdAt`
  - `title`
  - `summary`
  - `target`
  - `actor`
  - `preview`
  - `metadata`
- 允许系统消息无 `actor`，但必须有明确的 `target` 与状态元信息。

### 2. Server 侧改造

- 调整数据库结构与 seed 数据以支持新消息域。
- 重构 `social` 模块或拆分新的消息服务，统一生产、查询和已读更新逻辑。
- 把内容发布/审核状态变化接入消息生产链路。
- 提供适合前端分类渲染和虚拟列表消费的通知接口。

### 3. Web 侧消息中心

- 重做 `notifications-page` 的布局与交互。
- 使用单列虚拟长列表技术，优先复用现有 `react-virtuoso` 基础设施。
- 支持：
  - 一级分类切换
  - 未读数统计
  - 批量已读
  - 单条已读
  - 按类型展示不同卡片样式
  - 跳转到详情页、主页或状态页

### 4. 个人主页优化

- 审查当前“我的内容 / 我的收藏 / 设置 / 消息入口”的组织方式。
- 优化与新消息中心的联动入口和信息层级。
- 审查个人主页当前功能缺漏并做最小必要修复。

### 5. 他人主页优化

- 审查当前资料展示、关系状态、内容可见性与关注交互的完整性。
- 优化内容流组织与空态/受限态表达。
- 修复本轮审查确认的真实功能缺漏。

## 技术与结构约束

- 运行时：Bun
- 测试框架：Vitest
- 服务端框架：Hono
- ORM：Drizzle
- 依赖方向保持：`apps -> packages -> 独立`
- 修改共享协议时必须同步检查：
  - `packages/schemas`
  - `packages/http-client`
  - `packages/shared`
  - `apps/server`
  - `apps/web`
  - `apps/admin`
- 修改数据库相关内容时必须检查 `packages/db` 与 `apps/server`，并同步考虑 seed / reset / migrate 语义

## 数据迁移与兼容性策略

- 由于尚未上线生产环境，本轮允许：
  - 重塑通知表结构
  - 重建种子数据
  - 调整通知接口契约
- 但仍要求：
  - 修改后保持本仓库开发、测试和构建可运行
  - 同步更新测试与种子数据
  - 避免残留双轨通知模型导致的长期维护负担

## 验证要求

- 阶段内必须执行与各自改动直接相关的测试。
- 收尾默认考虑：
  - `bun run lint`
  - `bun run typecheck`
  - `bun run test`
  - `bun run build`
- 如因改造过程需要补充新的测试入口或测试辅助方法，应一并纳入本轮。
