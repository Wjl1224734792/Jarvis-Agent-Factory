# Phase 2 优化：活动追踪 + 只读 Web Dashboard

> 日期：2026-05-08 | 前置版本：v3.23.0

## 背景

v3.23.0 发布后发现以下问题：

1. **心跳机制不可靠**：终端 MCP 连接正常，Web 页面却显示断开。原因为独立心跳定时器与 MCP 工具调用脱节。
2. **Web 面板设计过度**：Dashboard 有"推进"按钮，但实际 Gate 推进由 Agent 通过 MCP 工具自动完成，用户不需要手动点击推进。
3. **会话不可见**：断开连接的会话从列表中消失，用户无法查看历史状态。
4. **超时时间不合理**：30 分钟太短，个人工具不需要频繁过期。

## 需求

### REQ-001：活动追踪模式（替代独立心跳）

**用户原话**："用行业成熟方案来优化，因为这是个 MCP 个人用的只需要轻量不需要很复杂"

- 每次 MCP 工具调用自动更新会话活动时间（`touchSession`）
- 移除独立的 5 分钟自动心跳定时器
- 移除独立的 30 秒心跳保活定时器
- 只保留 5 分钟过期标记定时器（`markStaleSessions`）
- 超时时间设为 **2 小时**（7200 秒）

**已实现**：`db.js` touchSession、`server.js` resolveSid 模式

### REQ-002：Web Dashboard 只读化

**用户原话**："不需要再 web 点击推进的，因为 agent 会再满足条件后自己推进不用我在 web 点击推进"

- 移除 Gate 步骤卡片上的"推进"按钮
- Web 面板纯粹展示状态，不做主动操作
- Gate 推进只能通过 MCP 工具调用完成

### REQ-003：会话全量展示

**用户原话**："就算连接断了我也要看到状态啊我重新刷新就行了啊"

- `/api/status` 统计所有会话（含 active + inactive）
- `/api/sessions` 返回所有会话（已实现）
- 前端侧边栏显示所有会话，用状态标签区分 active/inactive
- inactive 会话显示但不是"已断开/消失"

### REQ-004：超时时间优化

- `SESSION_TIMEOUT` = 2 小时（7200 秒）
- 前端 `isOnline` 判定窗口同步为 2 小时
- 会话永远不会被删除，仅标记为 inactive

## 验收标准

1. MCP 工具调用 → Web 面板即时显示"活跃"状态 ✅
2. Web 面板无"推进"按钮，纯展示 ✅
3. 侧边栏显示所有会话（含 inactive）✅
4. 2 小时无活动后标记为 inactive，不删除 ✅
5. 代码简洁，无独立心跳机制 ✅
