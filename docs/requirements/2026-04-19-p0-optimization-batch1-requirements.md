# 2026-04-19 P0 优化第一批需求

## 需求摘要

基于全仓审查报告与优化路线图，先落地第一批可直接实施的 P0 问题修复：

1. 品牌申请状态机修复
2. 榜单条目评分/评论写入解耦
3. server 测试与 seed 基础设施降重

本轮不处理“共享 client / 契约边界收口”，因为该项已在路线图中被归类为 `needs_ddd_first`。

## 范围内

- `packages/schemas`
- `packages/db`
- `apps/server`
- 与本轮修复直接相关的测试和实现文档

## 范围外

- `apps/web` / `apps/admin` 的共享 client 收口
- 推荐/搜索排序重构
- 大规模热点文件拆分
- 无关的结构重写

## 目标与成功标准

- 品牌申请状态在 schema、route、service、DB 语义上完全一致，不再出现 `hidden` 断裂。
- 榜单“纯评分”不再隐式创建评论；更新评分不会持续堆评论；榜单排序不再被该脏数据链路污染。
- server 测试不再默认依赖整库 `reset + seedDatabase()` 才能跑通核心场景，至少把当前热点测试文件的初始化成本降下来。
- 根级 `bun run lint`、`bun run typecheck`、`bun run test`、`bun run build` 通过。

## 约束

- 品牌申请修复优先对齐到当前 DB 允许的状态语义，不额外扩展新状态。
- 榜单评分与评论写入必须按 TDD 推进。
- 测试/seed 降重优先做最小可验证收敛，不为了“彻底完美”引入另一套复杂测试框架。
- 如发现实现必须跨到共享 client / app 端契约，停止并回主会话登记 plan patch。

## 已对齐假设

- 本轮品牌申请状态以 `pending/approved/rejected` 为准。
- 榜单“评分”与“评论”是两个独立动作，允许有评分但没有评论。
- 测试基础设施优化优先覆盖当前最重、最常失败或最常用的 server 测试链路。
