# 登录验证码重发、短信错误修复与发布菜单中文化需求

## 需求摘要

处理三项用户可见问题：

1. 登录页面补充“重新发送验证码”逻辑
2. 修复 `/auth/sms/request` 返回 500 的问题
3. 将发布入口下拉菜单改为中文

同时在收尾阶段把模拟数据重新推到数据库，并完成全量验证、提交、推送。

## 目标与成功标准

- 登录页支持重新发送短信验证码，交互清晰，不会误导用户。
- `/auth/sms/request` 不再因为可预期业务错误返回 500。
- 发布菜单文案全部改为中文。
- 模拟数据成功推入数据库。
- `bun run check` 通过。
- 变更已提交并推送到当前分支。

## 范围内

- `apps/web` 登录页验证码交互
- `apps/server` 短信发送错误处理
- `apps/web` 发布菜单文案
- 数据初始化脚本执行
- 文档同步

## 范围外

- 全站验证码流程重构
- 小程序或 App 仓库改动
- 非当前发布入口页面的大范围中文化

## 关键模块

- `apps/web/src/features/auth/login-page.tsx`
- `apps/server/src/modules/auth/auth.service.ts`
- `apps/server/src/modules/auth/auth.route.ts`
- `apps/web/src/features/auth/web-layout.tsx`
- 相关数据库与种子脚本

## 风险与开放问题

- 重新发送验证码需要兼顾后端频率限制，不能制造误导。
- `/auth/sms/request` 的 500 很可能是业务错误未正确映射，需要优先修复后端 service 层抛错路径。
- 推模拟数据前后需要避免脏状态影响验证结论。

## 已收敛结论

- 这轮以最小正确改动为主。
- 重新发送验证码优先实现为“刷新图形验证码 + 再次请求短信验证码”的稳定流程。
