# 认证密码与内容来源声明需求

## 摘要

本批次补齐普通用户密码能力和内容来源声明能力。用户要求无需继续澄清，因此以下需求按现有产品形态和成熟安全实践收敛：服务端强制密码策略，密码登录必须通过一次性图形验证码，改密后撤销旧会话；文章和动态支持声明来源，并在 Web 与 Admin 的发布、编辑、审核、详情和列表中展示。

## 假设

- 密码复杂度按用户明确要求执行：至少 8 个字符，至少包含 1 个大写字母、1 个小写字母、1 个特殊符号；不额外强制数字。
- 短信登录保持现有链路，不破坏首次登录后完成资料的注册流程。
- 普通用户注册必须设置密码；App 注册继续使用同一完成注册契约，因此也携带密码字段。
- “密码登录需要图形验证码”同时适用于 Web 普通用户密码登录和 Admin 管理员密码登录。
- 修改密码要求当前密码校验，成功后撤销该用户所有既有会话，要求重新登录。
- 来源声明为可选字段，包含来源名称和可选来源链接；没有填写时响应为 `source: null`。

## REQ-001 普通用户注册设置密码

首次手机号短信验证后进入注册资料补全时，必须设置符合密码策略的新密码。服务端负责最终校验并以哈希形式落库，客户端只做即时提示。

验收标准：
- 注册完成请求缺少密码或密码不满足策略时被拒绝。
- 成功注册的普通用户 `password_hash` 不为空。
- 登录态和头像补充流程保持现有行为。

## REQ-002 Web 登录支持短信和密码两种方式

Web 登录页提供短信登录和密码登录。短信登录继续使用手机号、短信验证码；密码登录使用手机号、密码、图形验证码。

验收标准：
- 已注册且有密码的用户可通过手机号 + 密码 + 图形验证码登录。
- 未注册手机号使用密码登录不触发注册流程，只返回凭据错误。
- 短信登录仍可在手机号未注册时返回注册补全分支。

## REQ-003 密码登录图形验证码

所有密码登录请求必须携带图形验证码挑战 ID 和验证码。验证码由现有 captcha 仓储生成、校验并一次性消费。

验收标准：
- Web 密码登录缺少或错误图形验证码时返回 `INVALID_CAPTCHA`。
- Admin 密码登录缺少或错误图形验证码时返回 `INVALID_CAPTCHA`。
- 成功或失败的密码校验不复用已消费验证码。

## REQ-004 修改密码

普通用户在设置页可修改自己的密码。管理员改密沿用现有页面，但使用相同密码策略。

验收标准：
- 普通用户输入当前密码、新密码、确认新密码后可改密。
- 当前密码错误、弱密码、新旧密码相同时被拒绝。
- 成功改密后撤销旧会话并清理 Cookie。
- Admin 改密也执行相同强密码策略。

## REQ-005 文章和动态声明来源

用户发布或编辑文章、动态时可填写来源名称和可选来源链接。Admin 官方文章编辑同样支持来源声明。来源保存到帖子数据，并在列表和详情响应中统一输出。

验收标准：
- 创建、编辑文章或动态时可提交 `sourceLabel` 和 `sourceUrl`。
- 来源名称为空时来源整体为 `null`，不保存孤立链接。
- 来源链接如果填写必须是合法 URL。
- Web 文章详情、动态详情、文章/动态 feed 中显示来源。
- Admin 官方文章编辑、官方文章列表和内容审核视图中可填写或查看来源。

## 范围内

- `packages/schemas`、`packages/shared`、`packages/http-client`
- `packages/db` schema 与 migration
- `apps/server` auth/posts repo、service、route
- `apps/web` 登录、注册、设置、文章/动态发布、feed/detail 展示
- `apps/admin` 登录验证码、改密策略、官方文章来源、内容审核来源展示

## 范围外

- 不接入新的第三方短信、风控或验证码服务。
- 不新增找回密码、忘记密码、邮箱登录、OAuth 登录。
- 不改变 CORS、OpenAPI、上传、审核模式默认策略。
- 不恢复或新增移动端目录。

## Gate A

- 需求均有 `REQ-XXX` 标识。
- 跨包契约、DB 和 Web/Admin/server 影响范围已列出。
- 用户明确要求“不要问我”，因此关键假设由主会话收敛并留痕。
