# 2026-04-27 消息中心互动链路优化需求说明

## 需求摘要

- 本轮在既有消息中心重建与管理端消息中心基础上，专项检查并优化消息中心各类互动链路，确保用户端与管理端的消息读取、筛选、已读、批量已读、跳转与入口计数通畅可用。
- 范围覆盖 `apps/web`、`apps/admin`、`apps/server`、`packages/schemas`、`packages/http-client`、`packages/shared`，但目标是最小修通链路，不做消息模型、数据库结构或 UI 体系的大规模重构。
- 若排查发现共享契约、服务端响应、前端消费或路由跳转存在漂移，本轮按共享契约优先顺序修复，禁止在页面层堆临时兼容。

## 已收敛结论

- 用户已确认本轮同时纳入 `apps/web` 与 `apps/admin`。
- 本轮目标是修通现有消息中心互动链路，而不是新增实时推送、管理员私聊或新的消息业务域。
- `apps/admin` 使用既有 Ant Design 后台体系；`apps/web` 使用既有消息中心页面与功能组件。
- 若需要改 API 形状、路由常量、请求/响应类型，按根 `AGENTS.md` 的顺序检查 `packages/schemas`、`packages/http-client`、`packages/shared`、`apps/server`、`apps/web`、`apps/admin`。

## 目标与成功标准

- 用户端消息中心：
  - 分类切换、列表加载、未读态、单条已读、批量已读、空态、错误态可用。
  - 点击消息能跳转到对应帖子、评论上下文、用户主页、内容状态页或系统消息目标。
  - 消息入口未读数与页面列表状态保持一致。
- 管理端消息中心：
  - 消息中心、审核待办、首页摘要、壳层角标之间的计数口径一致。
  - 业务域、消息类型、已读状态筛选可用，不产生无效组合或死链。
  - 单条已读、批量已读后正确刷新消息列表、首页摘要与壳层角标。
  - 消息与待办能跳转到对应审核页，并携带可消费的筛选参数。
- 服务端与共享契约：
  - 用户端与管理端消费的消息 DTO、筛选参数、已读接口、跳转 target 与共享类型保持一致。
  - 对无法落地的历史消息或缺少目标的消息，前端有明确降级，不触发运行时异常。
- 验证：
  - 补齐或修复与本轮链路直接相关的测试。
  - 收尾执行根级 `bun run lint`、`bun run typecheck`、`bun run test`、`bun run build`，如有失败记录原因与影响。

## 范围内

- `apps/web` 消息中心、消息入口、通知状态、消息操作与相关测试。
- `apps/admin` 消息中心、审核待办、首页摘要、壳层消息角标、审核页跳转与相关测试。
- `apps/server` 消息查询、已读写回、管理端消息 / 待办聚合、跳转载荷生成。
- `packages/schemas`、`packages/http-client`、`packages/shared` 中与消息链路直接相关的契约、请求方法和共享路径 / target 常量。
- 必要的实现说明、评审文档与定向验证记录。

## 范围外

- 新增实时推送、WebSocket、SSE 或移动端推送。
- 管理员私聊、群聊或协作 IM。
- 新增数据库表或迁移，除非现有链路无法在不改库的情况下修通；若触发需先提交 plan patch。
- `apps/mobiles`、小程序、Flutter App。
- 无关页面的视觉重做、导航重构或业务流程重构。

## 关键链路清单

1. 用户端入口链路：登录态恢复 / 未读数加载 -> 首页或导航入口展示 -> 进入消息中心 -> 列表与分类状态一致。
2. 用户端操作链路：单条已读 / 批量已读 -> 缓存失效或本地状态更新 -> 未读数与列表刷新一致。
3. 用户端跳转链路：消息项 target -> `message-actions` 解析 -> 业务详情页、用户页或内容状态页。
4. 管理端入口链路：后台壳层角标 / 首页最近通知 / 首页待办 -> 消息中心或待办页。
5. 管理端筛选链路：URL query -> 筛选控件 -> http-client 请求参数 -> server 聚合查询 -> 表格或待办列表。
6. 管理端操作链路：单条已读 / 批量已读 -> query invalidation -> 消息中心、首页摘要、壳层角标同步刷新。
7. 管理端跳转链路：消息 target / navigation -> 审核页路径与 query -> 审核页筛选落点。

## 风险与开放问题

- 既有 Web 与 Admin 消息中心来自两个相邻迭代，可能存在共享契约字段、target 命名或缓存 key 漂移。
- 某些历史消息可能缺少可跳转目标；本轮应保证前端降级可用，并在服务端补齐可安全生成的 target。
- 若待办聚合依赖业务对象实时状态，而消息已读只修改消息状态，需要重点验证“待办独立于已读”的既定口径没有被前端操作误伤。
- 若排查发现必须新增数据库字段或迁移，本轮需先暂停实现并提交 plan patch。
