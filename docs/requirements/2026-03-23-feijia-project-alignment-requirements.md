# 飞加网项目需求对齐稿

日期：2026-03-23  
适用范围：当前仓库 `apps/web`、`apps/admin`、`apps/server` 及 `packages/*`

## 1. 需求摘要

本轮不是继续扩写 PRD，而是把现有仓库收敛成一套可交付、可验证、可继续迭代的 MVP。目标是让飞加网形成一个完整的社区主链路闭环：用户能登录、浏览首页信息流、浏览机型库、查看机型详情、提交点评、浏览帖子与评论、完成基础互动；管理员能登录后台并完成最小治理；前端要在 PC 优先的前提下做出清晰、克制但有质感的视觉效果。

当前仓库已经存在 `web`、`admin`、`server`、`shared`、`schemas`、`http-client`、`db` 的基础骨架，因此本轮应优先补齐真实业务链路和视觉完成度，而不是再拆新的抽象层。

## 2. 目标和成功标准

- 用户可以完成登录/退出，并在刷新后恢复身份。
- 游客可以浏览首页、机型库、机型详情和帖子详情的公开内容。
- 已登录用户可以发帖、上传图片、评论、回复、点赞、收藏、分享和关注。
- 已登录用户可以对机型提交点评，并在详情页看到综合评分和点评列表。
- 管理员可以登录后台，查看并治理帖子、评论、点评、机型、分类和品牌。
- 前端页面在桌面端有明确设计语言，在移动端保持可用而不是仅有挤压后的桌面布局。
- 本地开发、类型检查、测试和构建链路能跑通。

## 3. 范围

### 范围内

- `web` 主站：首页信息流、机型库、机型详情、帖子详情、登录/个人中心、通知入口。
- `admin` 后台：概览、分类管理、品牌管理、机型管理、点评治理、帖子治理、评论治理。
- `server`：认证、帖子、机型、点评、关注/通知、后台治理接口。
- `packages/schemas`：统一请求/响应契约与 Zod 校验。
- `packages/http-client`：统一前端请求入口。
- `packages/db`：机型、帖子、点评、用户、会话及治理相关数据结构。
- 图片上传、帖子互动、基础通知、基础审核流。
- 响应式布局和首页/详情页的完整视觉改造。

### 范围外

- 私信 / 实时聊天。
- 榜单的完整产品化实现。
- 自定义榜单创建与排序编辑。
- 心愿单的完整业务闭环。
- 账号注销、认证账号、厂商入驻、品牌官方回复。
- 复杂推荐算法、A/B 灰度、贝叶斯平滑、风控反作弊。
- 独立移动端应用。

### 可合理降级为静态演示或占位

- 榜单：可保留入口位或静态卡片，不要求本轮完成真实排行引擎。
- 私信：可保留消息中心页面入口，但不做实时会话。
- 心愿单：可保留按钮和态位，但不要求完整后台统计。
- 高级推荐：可用热度 + 时间衰减的简化排序，暂不接协同过滤。
- 认证账号、举报流的深度治理：可先做基础标识或基础提交，不做完整审核编排。

## 4. 模块 / 功能列表

- 认证模块：手机号登录、管理员登录、会话恢复、退出。
- 首页信息流：推荐、最新、关注等基础 tab。
- 机型库：分类、品牌、动力类型筛选，机型卡片列表，详情页。
- 点评体系：单用户单机型唯一点评、评分汇总、后台点评治理。
- 帖子体系：发布、详情、评论、回复、点赞、收藏、分享、删除、举报。
- 通知体系：未读提示、通知列表、基础已读状态。
- 后台治理：帖子、评论、点评、机型、分类、品牌的最小治理。
- 设计系统：统一色彩、卡片、排版、交互状态和移动端适配。

## 5. 关键业务流程

1. 游客进入首页，看到品牌感明确的首屏和信息流入口。
2. 用户登录后，前端恢复身份并展示可写操作。
3. 用户浏览机型库，按类型、品牌、动力类型筛选机型。
4. 用户进入机型详情页，查看核心参数、综合评分和点评列表，并提交点评。
5. 用户浏览帖子详情，进行点赞、收藏、分享、评论和回复。
6. 用户发布帖子后，内容进入正常前台链路，后台可治理异常内容。
7. 管理员登录后台，对帖子、评论、点评、机型和分类进行治理。

## 6. 关键对象

- `User`：普通用户或管理员，承载身份与基础资料。
- `Session`：会话记录，用于登录恢复与权限判断。
- `AircraftModel`：机型主数据，承载品牌、分类、动力类型和参数。
- `ModelReview`：机型点评，包含评分、文本、状态和时间。
- `Post`：帖子主体，包含标题、正文、图片和审核状态。
- `PostComment`：帖子评论与回复。
- `PostInteraction`：点赞、收藏、分享、关注等互动记录。
- `Notification`：通知和未读状态。
- `Category` / `Brand`：机型筛选维度。

## 7. 模块交互草案

- `apps/web` 通过 `packages/http-client` 调用 `server`，不直接依赖服务内部实现。
- `apps/admin` 通过 `packages/http-client` 调用后台接口，不直接接触数据库。
- `apps/server` 负责校验、业务编排、写入和读取，统一使用 `packages/schemas` 的契约。
- `packages/shared` 只放路由常量、通用常量和纯工具，不承载业务流程。
- `packages/db` 负责表结构、迁移、种子和仓储基础能力。
- 图片上传、互动计数、点评汇总等写操作优先集中到 `server`，前端只消费结果。

## 8. 与当前代码结构的映射

- `apps/web/src/app.tsx`、`apps/web/src/features/auth/web-layout.tsx`、`apps/web/src/routes/home-page.tsx`、`apps/web/src/routes/models-page.tsx`、`apps/web/src/routes/model-detail-page.tsx`、`apps/web/src/routes/post-detail-page.tsx` 已经覆盖了本轮最核心的用户端入口，应优先在这些页面上完成真实体验闭环。
- `apps/admin/src/app.tsx`、`apps/admin/src/features/auth/admin-shell.tsx`、`apps/admin/src/features/auth/admin-overview-page.tsx`、`apps/admin/src/features/models/*`、`apps/admin/src/features/posts/*`、`apps/admin/src/features/reviews/*` 已经覆盖后台主干，应作为治理能力落点。
- `apps/server/src/app.ts` 已挂载 `auth`、`posts`、`social`、`aircraft-models`、`reviews`、`brands`、`categories` 模块，说明服务端主线已成形，适合继续补齐缺失能力。
- `packages/shared/src/index.ts` 已集中管理路由和 API 常量，新增或调整接口应该继续延续这里的模式。
- `packages/schemas/src/*.ts` 已按业务域拆分契约，新增字段和响应结构应优先在这里收口。
- `packages/db/src/*` 已承载数据库初始化、迁移和种子，新的数据对象应优先落这里。
- 当前仓库中还未看到独立的 `packages/ui`、`packages/storage` 实现，因此本轮不应默认它们已经存在；如果需要统一 UI 封装或 S3 抽象，应作为显式新工作项，而不是隐式前提。

## 9. 风险 / 缺失信息 / 开放问题

- 榜单、私信、心愿单是否必须纳入本轮交付，需要最终确认；当前建议都不作为 MVP 必交项。
- `web` 首页当前实现是顶部导航 + 内容区，和 PRD 的“顶部导航 + 左侧边栏 + 主内容区”不完全一致，是否在本轮补成完整 PC 结构需要确认。
- 图片上传、通知、互动计数是否要求强一致，还是接受最终一致性窗口，需要明确。
- 机型点评的综合评分计算口径是否固定为 10 分制加权平均，需要确认。
- 认证账号、举报、账号注销是否只是入口位，还是要做完整治理链路，需要确认。
- 是否要在本轮建立 `packages/ui` 与 `packages/storage`，还是先保持 app 内实现，需要确认。

## 10. 推荐的下一步

先把上面的范围确认成一个稳定的 MVP 定义，再交给 `task_design` 拆分实现任务。若目标是尽快交付，我建议优先顺序是：

1. `web` 首页、机型库、机型详情、帖子详情和登录态闭环。
2. `server` 的真实写入链路和点评/互动/通知契约。
3. `admin` 的最小治理页面。
4. 最后统一做前端视觉升级和响应式收口。
