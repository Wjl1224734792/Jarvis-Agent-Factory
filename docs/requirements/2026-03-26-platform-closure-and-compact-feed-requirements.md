# 平台闭环与紧凑信息流需求收敛

日期：2026-03-26  
适用范围：`apps/web`、`apps/server`、`packages/db`、`packages/http-client`、`packages/shared`、`packages/schemas`

## 1. 需求摘要

本轮目标不是扩新业务，而是先把当前仓库收敛成一个可跑通、可种子初始化、可扩展存储/短信 provider、并且前端信息流更紧凑的闭环。

当前已知现状：
- PostgreSQL、Redis、MinIO 容器已存在，但项目仍需要迁移和种子初始化才能形成可用基线。
- `packages/db` 已有迁移与 seed 入口，但对象存储和短信 provider 仍偏向单实现或 mock。
- `apps/web` 已有首页、榜单、帖子详情、发布页、评论/互动骨架，但信息流密度、首页右栏内容和榜单卡片还不符合当前要求。

## 2. 目标与成功标准

- 本地环境可通过统一流程完成数据库迁移、种子初始化和基础服务接入。
- 种子数据可以覆盖 PostgreSQL，并把必要的示例媒体和演示状态推送到 MinIO、Redis。
- 存储层开始具备 S3 兼容抽象，至少可为 `COS`、`OSS`、`KODO`、`MinIO` 预留统一配置入口。
- 短信层开始具备 provider 抽象，至少可为阿里云和腾讯云预留统一配置入口。
- 前端页面整体信息流更紧凑，榜单页卡片更小，首页右侧出现指定的三个热门块。
- 评论、评分、点赞、收藏、浏览量形成完整闭环，不再只是展示或临时计算。
- 不新增个人中心和设置展示入口。

## 3. 范围内

- 数据库迁移、seed、示例数据生成与初始化链路。
- 存储 provider 抽象与环境变量配置。
- 短信 provider 抽象与环境变量配置。
- 前端首页、榜单页、详情页、发布页、帖子/榜单互动相关的展示收口。
- 评论、评分、点赞、收藏、浏览量的后端闭环与前端接入。

## 4. 范围外

- 个人中心与设置页的展示化落地。
- 复杂推荐算法、完整榜单产品化、实时聊天、心愿单。
- 多端独立重做或大规模 UI 重构。
- 非必要的重命名和清理。

## 5. 关键模块映射

- `packages/db`：迁移、seed、示例数据、初始化脚本。
- `packages/shared`：路由常量、公共枚举、provider 配置键名。
- `packages/schemas`：请求/响应 schema、provider 输入输出契约。
- `packages/http-client`：统一请求层与错误映射。
- `apps/server`：存储、短信、互动闭环、数据读写和校验。
- `apps/web`：信息流布局、页面密度、互动 UI、展示收口。

## 6. 风险与开放问题

- 现有 seed 已经包含较多业务对象，新增 seed 逻辑要避免和现有幂等规则冲突。
- Redis 目前更像运行时依赖，不宜直接当作唯一事实源，seed 只能写可重建示例数据。
- S3 兼容抽象一旦先做成“看起来能用”的接口，后续接 COS/OSS/KODO 会被第二套业务逻辑反噬。
- 浏览量如果继续停留在前端推算，后续互动闭环会继续失真，必须明确是否落库或通过缓存聚合。
- 首页右侧内容块和榜单小卡片都属于共享展示风格，后续不要在多个页面各写一套不同密度。

## 7. 推荐下一步

把本需求交给 `task_design` 和 `planner`，先锁定基础设施闭环、provider 抽象、互动闭环，再做前端紧凑化。
