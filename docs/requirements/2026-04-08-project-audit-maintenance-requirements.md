# 项目审查与维护需求说明

## 需求摘要

- 对当前仓库进行一次面向交付质量的系统审查，覆盖架构、性能、维护性、代码逻辑正确性、命名清晰度与规范一致性。
- 在审查过程中，对高置信度、低风险的问题直接修复，不等待额外确认。
- 为前端补充自动化测试用例，优先覆盖 `apps/web` 的关键用户链路与易回归逻辑。

## 目标与成功标准

- 输出按严重级别排序的审查结论，能够明确区分阻塞问题、建议修改项和参考性优化项。
- 修复至少一批高置信度问题，并提供可验证的回归结果。
- 新增前端自动化测试，覆盖本轮修复或高风险交互逻辑。
- 运行并记录 `bun run lint`、`bun run typecheck`、`bun run test`、`bun run build` 的结果。

## 范围内

- `apps/server` 的架构、数据流、错误处理、热点 service/repo、OpenAPI 组织方式。
- `apps/web` 的路由组织、鉴权流、状态管理、体积与懒加载策略、关键交互自动化测试。
- `apps/admin` 的路由、登录保护、构建与可维护性风险扫描。
- `packages/http-client`、`packages/schemas`、`packages/shared` 中影响前后端契约与维护性的共性问题。
- 构建配置、测试配置、数据库与脚本层面的高置信度工程问题。

## 范围外

- 新业务功能开发。
- 微信小程序、App 端独立工程。
- 大规模视觉重设计。
- 无法在当前仓库中独立验证的外部基础设施改造。

## 关键模块 / 功能列表

- 服务端应用入口、错误处理、OpenAPI 文档组织。
- Web 端登录保护、重定向、路由懒加载、构建产物体积。
- 管理端登录保护与后台路由组织。
- HTTP Client 的契约边界与可维护性。
- 自动化测试覆盖面与缺口。

## 风险与开放问题

- 代码热点文件体积较大，审查中可能发现需要分阶段处理的结构性问题，本轮只修高置信度且低风险的部分。
- 当前前端测试设施以 Vitest 为主，若缺少 DOM 测试依赖，则优先采用纯函数/路由辅助逻辑测试，不强行引入重型测试栈。
- 构建体积优化可能需要在收益与配置复杂度之间做平衡，本轮以“显著降低主包风险且不改变运行时行为”为标准。

## 已收敛结论

- 用户明确要求按编排流程处理，并允许在无需额外确认的前提下自主寻找并修复可优化项。
- 默认假设“前端自动化测试”优先指向 `apps/web`，`apps/admin` 仅做风险扫描与必要辅助修复，不作为本轮主测试面。
- 本轮优先级顺序为：正确性与回归风险 > 性能热点 > 可维护性与命名规范。
