# 2026-04-19 发布敏感词过滤与信息流推荐排序需求

## 需求摘要

在现有 `posts` 发布与 feed 查询链路上补齐两类能力：

1. 发布内容敏感词过滤
2. 信息流展示算法与推荐排序

当前仓库已经具备帖子发布、首页文章流、圈子动态流，以及一个非常初版的 `feed-recommendation.ts` 加权排序，但缺少完整的内容安全治理与分流策略。本轮需要在不引入无关重构的前提下，把“发布前过滤”和“按流展示”补成可验证、可演进的后端能力。

## 目标与成功标准

- 用户在发布文章、动态，或后台编辑官方文章时，标题与正文会经过统一的敏感词过滤。
- 命中敏感词时，请求被拒绝，并返回可读且稳定的错误信息；不允许脏内容落库后再依赖人工修复。
- 首页文章流与圈子动态流的 `recommended / latest / following` 三类 tab 都有明确、可解释的展示规则。
- `recommended` 不再只是静态按点赞/评论粗排，而是结合新鲜度、互动、关系、内容质量等信号做分类型排序。
- `latest` 与 `following` 保持稳定、可预测的时间序或关注序，不被推荐算法误伤。
- 本轮实现仅优先落在 `apps/server` 的发布与 feed 管线，避免把算法逻辑散落到前端。
- 输出足够的测试与文档，能说明哪些信号已接入、哪些仍是后续可扩展项。

## 范围内

- `apps/server/src/modules/posts/**`
- 与 posts feed / 发布相关的测试
- 必要的共享 schema 调整，仅限返回错误或展示排序结果确有需要时
- 推荐排序策略设计文档与实现文档

### 具体覆盖

- 用户创建帖子：文章、动态
- 用户编辑帖子：文章、动态
- 后台编辑官方文章
- 首页文章流：`recommended`、`latest`、`following`
- 圈子动态流：`recommended`、`latest`、`following`

## 范围外

- 评论敏感词过滤
- 搜索召回算法、搜索排序算法
- 管理端敏感词词库管理页面
- 独立的召回服务、向量检索、画像系统
- 新增数据库表、迁移或复杂离线任务
- 前端 UI 改版

## 默认假设

- “发布的内容”优先指帖子与官方文章的标题、正文，不包含评论。
- “各个信息流”优先指现有 `API_ROUTES.feed` 与 `API_ROUTES.circleFeed` 两条主 feed，以及它们已有的三个 tab。
- 本轮不引入数据库 schema 变更，敏感词词库先以内置服务或本地配置方式落地。
- 推荐算法先做在线轻量排序，不做复杂召回拆层；仍以现有 repo 查询出的候选集合为基础重排。

## 关键模块 / 功能列表

1. 发布前敏感词检测服务
2. posts 发布与更新链路接入敏感词校验
3. 推荐排序信号定义与评分函数
4. 首页文章流分 tab 展示规则
5. 圈子动态流分 tab 展示规则
6. 面向实现和回归的测试补齐

## 规则口径

### 敏感词过滤

- 检测对象至少包含：
  - 帖子标题
  - 帖子正文纯文本
  - 官方文章标题
  - 官方文章正文纯文本
- 命中后直接拒绝写入或更新，不做静默替换。
- 错误口径稳定，便于前端展示与后续扩展。
- 敏感词检测逻辑集中在后端服务层，禁止在 route 层或多个业务入口重复拼装判断。

### 信息流展示算法

- `recommended`
  - 使用统一评分框架，但文章与动态允许不同权重。
  - 至少纳入以下信号：发布时间衰减、点赞/收藏/分享/评论、作者关系、内容介质特征、基础质量约束。
  - 在同分时以发布时间倒序兜底。
- `latest`
  - 保持按发布时间倒序，不混入推荐权重。
- `following`
  - 仅展示关注作者内容；集合内按发布时间倒序。

## 风险与开放问题

- 当前 feed 仍是“先查候选，再内存排序”，如果候选集合过窄，推荐效果会受限，但本轮不扩展为新的召回层。
- 现有帖子表缺少更丰富的算法特征，本轮应优先复用已存在字段，避免推动 DB 变更。
- 若后续确认评论、评测、榜单等也要接入敏感词过滤，属于后续扩范围，不在本轮默认范围内。

## 已收敛结论

- 本轮先走后端实现，不要求同步推出管理端词库配置页面。
- 本轮先完成可解释、规则化的推荐算法，而不是黑盒模型。
- 本轮允许生成一个专门设计推荐算法的子代理，先产出策略，再由实现链路消费。
