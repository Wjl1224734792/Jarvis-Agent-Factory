# 移动端 / 微信小程序前端需求文档

## 1. 文档定位

本文档用于指导后续独立开发：

- 移动端 App 前端
- 微信小程序前端

本需求文档以当前仓库内已实现的后端能力和 Web 用户端能力为基准，不额外假设未落地的后端接口。

## 2. 目标与原则

### 2.1 目标

在不改动当前后端主协议的前提下，交付一套可独立开发的移动端 / 微信小程序前端规格，覆盖：

- 页面结构
- 布局形式
- 页面逻辑
- 关键交互
- 登录鉴权
- 发布与上传
- 用户闭环

### 2.2 设计原则

- 与当前 Web 用户端能力保持一致，不额外发明后端未实现能力。
- 优先保证主链路闭环，再考虑增强型体验。
- App 与微信小程序尽量共用信息架构，但允许交互细节不同。
- 管理端能力不进入本期移动端 / 小程序范围。

## 3. 当前后端与 Web 已有能力范围

### 3.1 当前已落地用户端核心能力

- 手机号 + 图形验证码 + 短信验证码登录 / 注册
- 登录后用户资料完善
- 首页 Feed
- 飞友圈 Feed 与详情查看
- 帖子详情、评论、点赞、收藏、分享、举报
- 飞行器列表、详情、评论、评测、互动、举报
- 榜单列表、榜单详情、评分对象详情、评论、评分、举报
- 用户主页、关注关系、用户聚合内容
- 通知列表、单条已读、全部已读
- 个人资料设置、头像上传、手机号换绑
- 发布文章
- 发布动态
- 发布飞行器投稿
- 发布品牌申请
- 创建榜单 / 评分对象
- 上传图片 / 视频 / 举报图片等统一上传流程

### 3.2 当前明确不在本期范围

- 管理端
- 微信登录
- 支付
- 即时聊天
- 离线缓存同步
- 推送系统

## 4. 客户端形态建议

### 4.1 App 形态

- 技术实现：uni-app (Vue 3 + TypeScript)
- 鉴权建议：使用后端现有 `/auth/app/*` 接口
- 会话模式：`accessToken + refreshToken`
- 上传模式：沿用当前 `/uploads/init -> 对象存储直传 -> /uploads/complete`

### 4.2 微信小程序形态

- 技术实现：uni-app (Vue 3 + TypeScript)
- 第一阶段鉴权建议：同样使用 `/auth/app/*` 的 token 模式
- 原因：
  - 当前后端已有 App 登录、注册完成、刷新、登出接口
  - 比 Cookie 方案更适合小程序宿主环境
  - 与 App 共用同一套代码库，降低维护成本

### 4.3 统一要求

- App 与小程序共用：
  - 信息架构
  - 页面命名
  - 核心接口契约
  - 上传与错误处理规则
- App 与小程序允许分别适配：
  - 导航形式
  - 手势与动效
  - 文件选择能力
  - 系统 UI 风格

## 5. 信息架构

### 5.1 底部主导航

建议统一使用 5 个一级入口：

1. 首页
2. 飞友圈
3. 发布
4. 榜单
5. 我的

说明：

- “飞行器”放到首页内二级入口，避免底部导航过多。
- “通知”放入“我的”页内一级功能卡片或右上角入口。
- 小程序和 App 都可以保留一个居中的“发布”强调入口。

### 5.2 一级导航结构

| 一级入口 | 页面目标 |
| --- | --- |
| 首页 | 承接推荐内容、热门榜单、热门飞行器、快捷发布入口 |
| 飞友圈 | 聚焦动态流、互动、内容详情 |
| 发布 | 统一进入发布中心，选择文章 / 动态 / 飞行器 / 品牌 / 榜单 |
| 榜单 | 查看榜单与评分对象 |
| 我的 | 承接登录、个人主页、设置、通知、我的内容 |

## 7. 页面需求

### 7.1 启动页 / 闪屏页

#### 布局

- 品牌 Logo
- 品牌文案
- 加载动画

#### 页面逻辑

- 启动后先检查本地 token / refreshToken
- 若有 token，调用会话恢复逻辑
- 恢复成功进入首页
- 恢复失败清理本地状态并进入未登录首页

### 7.2 登录页

#### 布局

- 手机号输入框
- 图形验证码展示区
- 图形验证码输入框
- 短信验证码输入框
- “获取验证码 / 重新发送验证码”按钮
- 登录 / 注册按钮
- 登录说明

#### 页面逻辑

- 进入页面自动请求图形验证码
- 用户输入手机号、图形验证码
- 点击“获取验证码”后调用短信发送接口
- 成功后进入 60 秒倒计时
- 倒计时结束按钮变为“重新发送验证码”
- 每次发送成功后自动刷新图形验证码
- 提交登录后：
  - 若返回 `authenticated`，进入首页
  - 若返回 `registration_required`，跳到注册补全页

#### 关键状态

- 图形验证码内容
- 图形验证码 challengeId
- 短信发送中
- 短信倒计时
- 错误提示

### 7.3 注册补全页

#### 布局

- 头像上传区
- 用户名输入框
- 完成注册按钮

#### 页面逻辑

- 从登录页带入 `registrationToken`
- 支持上传头像
- 提交后调用注册完成接口
- 成功后进入首页

### 7.4 首页

## 8. 统一交互规则

### 8.1 登录门槛

以下操作要求登录：

- 点赞
- 收藏
- 评论
- 关注
- 举报
- 发布
- 上传
- 评分

未登录触发时，统一弹登录引导。

### 8.2 上传规则

- 所有上传统一走后端两阶段上传流程
- 前端必须根据当前环境变量配置接受后端返回的上传限制提示
- 前端上传前建议做本地大小校验，但以后端返回为准

### 8.3 错误提示

- 优先显示共享 client 输出的安全提示
- 不直接暴露后端原始异常信息
- 上传超限提示直接显示“当前最大允许 X MB”

### 8.4 空状态

所有列表页必须有：

- 加载中状态
- 空状态
- 错误状态

## 9. 鉴权与会话要求

### 9.1 App / 小程序推荐方案

- 使用 App token 流程：
  - `/auth/app/login`
  - `/auth/app/register/complete`
  - `/auth/app/refresh`
  - `/auth/app/me`
  - `/auth/app/logout`

### 9.2 本地存储

- 保存 `accessToken`
- 保存 `refreshToken`
- 启动时尝试恢复会话
- `accessToken` 失效时自动刷新

### 9.3 安全要求

- 不在 UI 中直接展示：
  - `mockCode`
  - `registrationToken`
  - `refreshToken`
  - `accessToken`
- 错误信息必须经过前端安全映射

## 11. App 与微信小程序差异要求

### 11.1 App 额外要求

- 支持更完整的视频选择与播放能力
- 支持页面级返回手势
- 支持更强的图片预览与多媒体体验

### 11.2 微信小程序额外要求

- 页面层级更浅，减少复杂多层跳转
- 优先列表页 + 详情页结构
- 分享能力优先适配微信生态
- 上传和授权流程遵循小程序宿主约束

## 12. 开发排期建议

### 第一阶段

- 首页

### 第二阶段

## 13. 交付物要求

独立客户端开发启动前，至少应具备：

- 页面清单与路由表
- 全局导航结构
- 页面线框图
- 状态流转图
- 接口调用清单

## 14. 结论

基于当前后端和 Web 用户端能力，移动端 App 与微信小程序已经具备独立前端开发条件。

建议后续开发顺序：

1. 先开发 App 与小程序共用的信息架构与状态模型
2. 再按客户端形态分别落页面
3. 优先接通登录、首页、详情、互动、我的四条主链路
4. 最后补齐发布能力
