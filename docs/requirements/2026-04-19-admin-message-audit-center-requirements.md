# 2026-04-19 管理端消息中心与审核系统消息联动需求说明

## 需求摘要

- 在现有站内消息基础上补齐审核相关系统消息，覆盖当前审核链路中尚未打通或表达不完整的状态变化。
- 将系统消息与审核待办能力接入 `apps/admin`，新增管理端消息中心 / 待办视图，用于查看、筛选并快速跳转到对应审核页或业务详情。
- 重新设计 `apps/admin` 的后台信息架构与页面骨架，采用更传统、更稳定的 Ant Design 管理后台布局，同时保留现有业务模块与路由分区思路。
- 本轮默认不做“管理员之间的聊天 / 协作 IM”，而是完善“系统消息 + 审核通知 + 后台消费链路”。

## 目标与成功标准

### 目标

- 建立一套可复用的审核系统消息机制，能稳定表达“谁的什么内容被审核到什么状态、为什么、该去哪里处理/查看”。
- 让管理端具备统一消息入口，能查看系统消息、审核待办与快捷跳转，而不是仅依赖各审核页面手工轮询。
- 将 `apps/admin` 的壳层、导航、首页和关键审核页面整理为传统管理后台布局，降低操作路径成本与信息噪音。

### 成功标准

- 服务端至少覆盖以下系统消息生产与查询链路：
  - 帖子 / 文章 / 动态审核状态变化
  - 评测审核状态变化
  - 榜单审核状态变化
  - 评分对象审核状态变化
  - 机型投稿审核状态变化
  - 品牌申请审核状态变化
- 管理端新增明确的消息中心 / 待办入口，支持：
  - 按消息类型或业务域筛选
  - 查看未读与待处理数量
  - 跳转到对应审核页、详情页或处置入口
  - 对单条消息执行已读或批量已读
- 管理端整体布局调整为传统后台样式：
  - 固定头部
  - 左侧分组导航
  - 右侧内容区
  - 首页以 KPI、待办、最近通知、快捷入口为主
- 共享契约与前后端代码保持一致，`apps/admin` 不再自己拼装一套与消息契约脱节的展示模型。

## 范围内

- `apps/admin`
- `apps/server`
- `packages/schemas`
- `packages/http-client`
- `packages/shared`
- 与本轮直接相关的测试、必要文档与最小样例数据

## 范围外

- 管理员与管理员之间的私聊、群聊、评论协作消息
- App / 小程序端消息页面专项适配
- 无关业务模块的大规模重构
- 脱离 Ant Design 的后台 UI 体系迁移

## 关键模块 / 功能列表

### 1. 审核系统消息补齐

- 盘点当前所有审核状态变更入口，补齐尚未发消息的链路。
- 统一系统消息载荷，至少包含：
  - 消息类型
  - 业务域
  - 目标对象
  - 当前状态
  - 标题与摘要
  - 可跳转目标
  - 可选的驳回原因 / 说明信息
- 保证前端无需靠字符串猜测消息语义。

### 2. 管理端消息中心 / 审核待办

- 在 `apps/admin` 新增消息相关路由与导航入口。
- 设计适合管理端的消息列表视图，而不是简单复制 Web 端消息中心。
- 管理端消息中心需要重点突出：
  - 系统消息
  - 审核待办
  - 最近状态变化
  - 快捷跳转
- 如当前接口不足以支撑管理端消费，可在不破坏边界的前提下扩展服务端查询能力。

### 3. 管理端传统后台布局重整

- 基于 Ant Design 的 `Layout`、导航与内容容器重整后台壳层。
- 调整后台首页与关键审核页面的信息结构，使其更贴近传统运营后台：
  - 顶部状态区
  - 左侧导航
  - 中央待办 / 消息 / KPI
  - 清晰的页面级筛选与操作条
- 复用现有业务页面能力，尽量避免纯视觉重写导致的逻辑回归。

### 4. 关键审核页接线

- 将消息中心与以下页面建立稳定跳转关系：
  - 文章 / 动态审核
  - 评论审核
  - 评测审核
  - 品牌申请审核
  - 机型投稿审核
  - 榜单审核
  - 评分对象审核
- 如果某类消息当前无法直接定位到页面上下文，需要补齐路由或筛选参数能力。

## 已收敛结论

- 本轮主线是“系统消息 + 审核状态通知 + 管理端接入 + 后台布局重整”。
- 默认不做管理员之间的聊天协作消息；若后续要做，作为下一阶段独立需求处理。
- 管理端继续使用 Ant Design，不引入新的主 UI 框架。
- 管理端页面风格以传统管理后台为目标，不追求 Web 端那种偏内容产品的消息展示方式。
- 本轮允许同时触及共享契约、服务端和管理端，但必须保持 `packages -> apps` 的依赖边界。

## 风险与开放问题

- 当前系统消息已覆盖部分业务域，但审核链路并不完整，补齐过程中可能发现消息类型或目标对象枚举需要扩展。
- `apps/admin` 目前没有成熟的消息消费页面，新增后可能牵动导航、首页信息架构与局部样式系统。
- 若现有 API 只面向终端用户消息中心设计，管理端可能需要专门的聚合查询或过滤能力。
- 若某些审核消息缺少稳定跳转目标，可能需要同步调整管理端筛选参数或详情入口。

## 验证要求

- 至少执行仓库根级：
  - `bun run lint`
  - `bun run typecheck`
  - `bun run test`
  - `bun run build`
- 额外执行与消息 / 管理端改动直接相关的定向验证。
