# 飞加 UniApp 小程序 / 安卓前端需求文档

> 文档版本：v1.0  
> 更新时间：2026-04-12  
> 适用范围：微信小程序、Android App（UniApp 同构开发）  
> 关联文档：[移动端登录接入文档](../mobile-login-guide.md)、[OpenAPI 前端对接指南](../openapi-frontend-integration-guide.md)、[移动端 / 微信小程序页面开发任务清单](../mobile-wechat-page-task-checklist.md)

## 1. 文档目标

本文档用于指导飞加移动端前端开发，目标是让 UniApp 版本在不照搬 Web 布局的前提下，完整承接飞加现有核心能力：

- 浏览文章、动态、飞行器、榜单、排行对象和用户主页。
- 完成点赞、收藏、分享、关注、评论、回复、举报等互动。
- 完成文章、动态、飞行器、品牌申请、榜单创建等发布 / 创建流程。
- 支持搜索、通知、个人中心、设置、内容管理等基础用户闭环。
- 在微信小程序与 Android App 上保持一致的信息架构、操作逻辑和视觉语言。

本阶段不包含后台 Admin、支付、即时私信、直播、复杂 IM、厂商认证后台。移动端只做用户侧产品。

## 2. 产品定位

飞加移动端不是 Web 的缩小版，而是“低空飞行内容与机型数据库的随身入口”。移动端的核心体验应该更像“飞友随手浏览、快速互动、轻量创作、随时查机型”。

核心原则：

- **内容优先：** 首页、飞友圈、机型详情、榜单详情都要让内容本身占据第一屏。
- **操作轻量：** 点赞、收藏、分享、关注、评论必须即时反馈，不能让整页闪烁或回到顶部。
- **发布可完成：** 发布页可以比 Web 更简化，但字段规则、审核结果、错误提示必须完整。
- **移动端原生感：** 列表支持下拉刷新、上拉加载、骨架屏、底部抽屉、滑动返回。
- **跨端一致：** 微信小程序和 Android App 使用同一套页面结构与状态模型，只在平台能力上做差异适配。

## 3. 技术与工程假设

### 3.1 技术栈

- 框架：UniApp + Vue 3 + TypeScript。
- 状态管理：Pinia 或轻量 Store，按领域拆分。
- 请求层：统一 `request` 封装，所有接口从同一入口走。
- 样式：优先使用 `scss` 或 UnoCSS / Tailwind 风格工具类，最终以团队实际 UniApp 方案为准。
- 图片与视频：统一封装上传、预览、压缩、失败重试。
- 富文本：移动端第一版建议用简化编辑器，不完整复刻 Web 富文本工具栏。

### 3.2 认证口径

移动端使用 Bearer Token，不使用 Web Cookie。

- 登录接口优先使用 App 认证接口：`POST /auth/app/login`。
- 刷新接口：`POST /auth/app/refresh`。
- 当前用户：`GET /auth/app/me`。
- 请求头统一注入：`Authorization: Bearer <accessToken>`。
- `TOKEN_EXPIRED` 时自动刷新并重试一次。
- 刷新失败时清空本地 token，并跳转登录页。

### 3.3 数据来源

移动端复用现有公开 API 与用户 API。

- 首页：`GET /home/feed`、`GET /rankings`、`GET /models?sort=hot&limit=3`。
- 搜索：`GET /search?q=...`。
- 飞友圈：`GET /circle/feed`。
- 文章 / 动态详情：`GET /posts/{id}`、`POST /posts/{id}/view`。
- 飞行器：`GET /models`、`GET /models/{slug}`、`POST /models/{slug}/view`。
- 榜单：`GET /rankings`、`GET /rankings/{id}`、`GET /rating-targets/{id}`。
- 个人中心：`GET /users/me/profile`、`GET /users/{id}/profile`、`GET /users/{id}/content`。
- 通知：`GET /notifications`、`POST /notifications/read-all`、`POST /notifications/{id}/read`。

## 4. 信息架构

### 4.1 底部导航

建议采用 5 个一级入口：

| Tab | 名称 | 目标 |
| --- | --- | --- |
| 1 | 首页 | 聚合文章、热门榜单、热门机型、全站搜索入口 |
| 2 | 飞友圈 | 动态瀑布流、关注 / 推荐 / 最新动态 |
| 3 | 发布 | 五类发布 / 创建入口 |
| 4 | 榜单 | 榜单列表、榜单详情、排行对象详情 |
| 5 | 我的 | 个人资料、内容管理、通知、设置 |

飞行器库不放底部 Tab，放在首页快捷入口、搜索结果页和“我的”常用入口里。原因是底部 Tab 更适合高频社区场景，机型库更像“查询型入口”。

## 5. 全局交互规范

### 5.1 登录拦截

未登录用户可以浏览公开内容，但以下操作必须弹出登录引导：

- 点赞
- 收藏
- 分享计数写入
- 关注
- 评论
- 回复
- 举报
- 发布 / 创建
- 查看“我的内容管理”

登录引导使用底部弹窗，不跳空白页，不丢失当前页面上下文。登录完成后返回原页面，并尽量恢复用户刚才的操作意图。

### 5.2 列表刷新与加载

所有列表页面统一支持：

- 下拉刷新：刷新当前 Tab / 当前筛选条件。
- 上拉加载：加载下一页或追加更多内容。
- 骨架屏：首次加载展示骨架，不使用空白页。
- 局部刷新：互动后只更新当前卡片，不触发整页重载。
- 回到顶部：长列表滚动超过 2 屏后显示轻量回顶按钮。

### 5.3 互动反馈

点赞、收藏、关注、分享必须采用乐观更新：

1. 点击后立即变更按钮状态和计数。
2. 后台请求失败时回滚，并展示轻提示。
3. 不刷新评论区。
4. 不触发列表重新拉取。
5. 不导致页面抖动或滚动位置丢失。

评论发布、删除、编辑：

- 只更新当前评论树。
- 同步更新当前内容的评论数。
- Feed 列表中的摘要计数可在后台同步更新，不阻塞用户操作。

### 5.4 分享

微信小程序：

- 使用小程序原生分享能力。
- 分享文案优先使用内容标题。
- 分享路径必须带内容 ID / slug。

Android App：

- 使用系统分享面板。
- 分享内容包含标题、摘要和落地页链接。
- 分享成功后再写入分享互动状态。

### 5.5 搜索

全站搜索使用独立结果页，不做第一版下拉联想。

搜索范围：

- 文章
- 动态
- 机型
- 榜单
- 排行对象
- 用户

交互规则：

- 搜索词少于 2 个字符不发请求。
- 搜索结果按类型分组。
- 搜索历史保留最近 10 条，仅本地存储。
- 点击搜索历史直接进入搜索结果页。
- 搜索结果点击后进入对应详情页。

### 5.6 上传

所有上传必须走统一上传模块。

流程：

1. 初始化上传，获取上传目标。
2. 直传文件。
3. 完成回写，获取 `fileId` 与 URL。

移动端额外规则：

- 图片默认压缩到合理体积后上传。
- 视频上传必须展示进度。
- 上传失败允许重试。
- 上传中禁止提交表单。
- 已上传资源可删除、重新排序。

### 5.7 错误与空状态

错误提示优先级：

1. 字段级错误。
2. 页面级错误。
3. Toast / 轻提示。

必须做字段级提示的错误：

- 图形验证码错误。
- 短信验证码错误。
- 昵称已占用。
- slug 非法或重复。
- 文件类型不支持。
- 文件体积超限。

空状态需要给下一步行动：

- 没有搜索结果：提示换关键词，并提供热门机型入口。
- 没有评论：提示“成为第一个评论的人”。
- 没有收藏：提示去首页 / 机型库浏览。
- 没有发布内容：提示去发布中心。

## 6. 视觉风格

### 6.1 设计方向

移动端视觉建议采用“低空仪表 + 内容杂志”的方向：

- 低空仪表：用于机型参数、榜单排名、评分、统计。
- 内容杂志：用于文章、动态、详情页阅读体验。
- 方正骨架：大容器、图片区、列表块以直角或小圆角为主。
- 柔和触点：按钮、输入框、头像、标签保留小圆角，提升触控亲和力。

整体避免 Web 后台感、避免大面积边框堆叠、避免默认紫色渐变和泛用卡片风。

### 6.2 色彩

| `--color-fog` | `#EEF2F5` | 页面背景 |
| `--color-card` | `#FFFFFF` | 内容卡片 |
| `--color-primary` | `#156C8C` | 主按钮、选中态、链接 |

| `--color-sand` | `#E8D7B9` | 榜单、评分辅助色 |

| `--color-danger` | `#D94A3A` | 删除、错误 |

## 7. 页面需求

### 7.1 启动页

目标：

- 恢复本地会话。
- 判断进入首页还是登录页。
- 避免白屏。

布局：

- 全屏品牌背景。
- 中央 Logo + 飞加。
- 底部小字展示“飞友与飞行器社区”。

操作逻辑：

- 有 refreshToken 时尝试刷新会话。
- 刷新成功进入首页。
- 刷新失败清空本地 token，进入首页游客态。
- 启动页最多停留 1.5 秒，避免拖慢进入体验。

### 7.2 登录页

目标：

- 支持手机号 + 图形验证码 + 短信验证码登录。
- 新用户自动进入资料补全。

布局：

- 顶部品牌区：Logo、飞加、简短 slogan。
- 中部表单：手机号、图形验证码、短信验证码。
- 底部说明：用户协议、隐私政策。

操作逻辑：

- 页面进入自动请求图形验证码。
- 点击验证码图片刷新。
- 获取短信验证码前校验手机号与图形验证码。
- 短信验证码按钮进入倒计时。
- 登录成功且已注册进入首页。
- 登录成功但需要注册，进入注册补全页。

### 7.3 注册补全页

目标：

- 完成昵称与头像设置。

布局：

- 顶部说明卡片。
- 头像上传区。
- 昵称输入框。
- 完成按钮。

操作逻辑：

- 头像可跳过。
- 昵称必填，最多 50 字。
- `DISPLAY_NAME_TAKEN` 必须显示在昵称输入框下方。
- 完成后进入首页，并刷新当前用户信息。

### 7.4 首页

目标：

- 承接推荐内容、热门榜单、热门机型和搜索入口。

布局顺序：

1. 顶部搜索栏。
2. 快捷入口：飞行器库、榜单、发布、我的收藏。
3. Feed Tab：推荐 / 最新 / 关注。
4. 内容流卡片。
5. 热门榜单横滑区。
6. 热门机型横滑区。

卡片信息：

- 文章：封面、标题、摘要、作者、浏览量、评论数。
- 动态：媒体预览、标题 / 内容摘要、作者、点赞数、评论数。
- 热门榜单：封面、标题、条目数、评论数、热度标签。
- 热门机型：图、品牌 Logo（有 Logo 才显示）、机型名、收藏数、评论数。

操作逻辑：

- 首页搜索提交进入搜索结果页。
- 点击文章 / 动态进入帖子详情页。
- 点击热门榜单进入榜单详情页。
- 点击热门机型进入机型详情页。
- 关注 Tab 未登录时弹登录引导。

### 7.5 搜索结果页

目标：

- 展示全站搜索结果。

布局：

- 顶部搜索输入框。
- 搜索历史。
- 结果分组列表：文章、动态、机型、榜单、排行对象、用户。

操作逻辑：

- 少于 2 个字符不请求。
- 搜索提交后保存到本地历史。
- 结果分组为空时隐藏该分组。
- 点击结果进入详情页。
- 返回时保留关键词和滚动位置。

### 7.6 飞友圈动态流

目标：

- 浏览社区动态。

布局：

- 顶部 Tab：推荐 / 最新 / 关注。
- 双列瀑布流。
- 悬浮发布按钮。

动态卡片：

- 媒体图 / 视频标识。
- 标题或内容前两行。
- 作者头像、昵称。
- 点赞数、评论数。

操作逻辑：

- 点击卡片进入独立详情页，不使用 Web 弹窗逻辑。
- 点赞只更新当前卡片状态。
- 关注 Tab 未登录弹登录。
- 视频卡片点击详情后再播放，列表内不自动播放。

### 7.7 文章 / 动态详情页

目标：

- 提供沉浸阅读与评论互动。

布局：

1. 顶部返回栏。
2. 媒体区。
3. 标题、作者、发布时间。
4. 正文。
5. 互动栏。
6. 评论区。
7. 底部评论输入栏。

操作逻辑：

- 详情打开时调用浏览记录接口。
- 文章显示真实浏览量。
- 动态公开页不展示浏览量。
- 点赞、收藏、分享只更新互动栏。
- 评论默认按最新排序，可切换热门。
- 评论输入框聚焦时打开键盘，提交后清空草稿。
- 未登录评论弹登录引导。

### 7.8 飞行器列表页

目标：

- 提供机型查询、筛选、跳转详情。

布局：

- 顶部搜索框。
- 快捷筛选 chips：分类、品牌、动力、排序。
- 筛选底部抽屉。
- 双列或单列机型卡片。

机型卡片：

- 封面图。
- 品牌 Logo（有 Logo 才显示）。
- 品牌名、机型名。
- 一句话摘要。
- 收藏数、评论数。

操作逻辑：

- 筛选条件在本地状态和 URL 参数之间保持一致。
- 搜索关键词走 `keyword` 参数。
- 热门排序走 `sort=hot`。
- 筛选抽屉点击确认后刷新列表。
- 清空筛选恢复默认列表。

### 7.9 飞行器详情页

目标：

- 展示机型资料、参数、互动、评论。

布局：

1. 顶部返回栏 + 分享。
2. 图片 / 视频轮播。
3. 品牌与机型标题。
4. 核心参数摘要。
5. 详细参数表。
6. 简介。
7. 互动区：想买、收藏、分享。
8. 热门机型推荐。
9. 评论区。

操作逻辑：

- 详情打开时调用 `POST /models/{slug}/view`。
- 公开页不展示浏览量。

## 8. 组件抽象建议

优先抽这些基础组件：

- `AppPage`：统一页面容器、背景、顶部安全区。
- `AppNavBar`：返回、标题、右侧操作。
- `AppTabBar`：底部一级导航。
- `SearchBox`：顶部搜索输入与提交。
- `ContentCard`：文章 / 动态通用卡片。
- `ModelCard`：机型卡片。
- `RankingCard`：榜单卡片。
- `RatingTargetCard`：排行对象卡片。
- `BrandLogo`：品牌 Logo 展示，有 Logo 才渲染。
- `CommentSection`：评论排序、输入、评论树。
- `InteractionBar`：点赞、收藏、分享、关注。
- `UploadBox`：图片 / 视频上传。
- `EmptyState`：空态。
- `ErrorState`：错误态。
- `SkeletonBlock`：骨架屏。
- `LoginRequiredSheet`：登录引导底部弹窗。

## 9. 状态管理建议

建议按领域拆 Store：

| Store | 职责 |
| --- | --- |
| `authStore` | token、当前用户、登录状态、刷新会话 |
| `appStore` | 平台信息、设备安全区、全局 loading |
| `searchStore` | 搜索历史、最近关键词 |
| `feedStore` | 首页 Feed、飞友圈 Feed 缓存 |
| `publishStore` | 发布草稿、上传进度 |
| `commentStore` | 评论排序、回复目标、评论草稿 |
| `notificationStore` | 未读数、通知列表 |

关键规范：

- 详情页不要依赖列表页数据完整性。
- 列表卡片与详情页同一条数据可以共享缓存，但不能互相强制刷新整页。
- 互动操作只更新命中的当前对象。
- 发布草稿必须本地保存，避免页面返回丢失。

## 10. 性能要求

### 10.1 首屏

- 首页首屏可见时间目标：2 秒内。
- 启动页不做复杂动画。
- 首页 Feed、热门榜单、热门机型并行请求。
- 图片使用懒加载。
- 列表分页加载，不一次性渲染大列表。

### 10.2 滚动

- 长列表使用分页和虚拟列表能力（如果 UniApp 目标端支持）。
- 瀑布流卡片必须固定图片占位比例，避免图片加载后跳动。
- 评论区默认只展示根评论和少量回复，更多回复点击展开。

### 10.3 缓存

- 首页 Tab 缓存最近一次数据。
- 详情页返回列表时保留滚动位置。
- 搜索历史仅本地缓存。
- 个人资料修改后主动刷新 `authStore.user`。

## 11. 平台差异

### 11.1 微信小程序

- 分享使用 `onShareAppMessage`。
- 文件上传使用 `uni.uploadFile`。
- 登录仍使用手机号 + 短信验证码，不默认接微信一键登录。
- 页面栈深度要控制，详情页可用 `redirectTo` 或 `navigateBack` 兜底。
- 底部弹窗避免过高，最高不超过屏幕 85%。

### 11.2 Android App

- Token 建议存储在更安全的本地存储方案中。
- 分享使用系统分享面板。
- 支持系统返回键。
- 图片选择支持相册与拍摄。
- 上传大视频时展示进度与取消按钮。
- 网络弱时显示重试入口，不直接白屏。

## 13. 验收清单

### 功能验收

- 未登录可以浏览首页、飞友圈、机型、榜单和详情。
- 未登录触发互动时弹登录引导，不静默失败。
- 登录后可以点赞、收藏、分享、关注。
- 评论区支持最新 / 热门排序。
- 文章详情展示真实浏览量。
- 动态和飞行器公开页不展示浏览量。
- 作者在“我的内容管理”能看到自己的浏览量。
- 机型卡片、详情、发布页、创建榜单选择器都支持品牌 Logo，有 Logo 才显示。
- 搜索结果页能展示文章、动态、机型、榜单、排行对象、用户。
- 创建榜单的机型选择器可搜索、可筛选、内部滚动顺畅。

### 体验验收

- 列表互动后不整页刷新、不回到顶部。
- 详情互动后评论区不重载。
- 上传失败可重试。
- 发表失败保留草稿。
- 页面切换有骨架或过渡，不白屏。
- Android 返回键符合预期。
- 小程序分享路径能打开对应详情。

### 视觉验收

- 不出现 Web 站点侧边栏式布局。
- 大容器以直角或小圆角为主。
- 按钮和输入框保持易点击。
- 品牌 Logo 不存在时不显示占位图。
- 卡片信息密度适合移动端，不把 Web 右栏内容硬塞进详情页。

## 14. 风险与建议

- UniApp 跨端文件上传、视频预览、富文本能力差异较大，建议先封装适配层，再写页面。
- 小程序页面栈有限，详情页之间连续跳转要注意返回逻辑。
- 评论区和瀑布流是性能风险最高的两个区域，建议先做分页和骨架屏。
- 发布页字段多，第一版要优先保证提交成功率，不要过早追求复杂编辑器体验。
- 如果后续要做 iOS App，也应继续复用本信息架构，不要重新设计一套导航。
