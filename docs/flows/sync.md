# `/sync` — 项目文档同步流程

- **命令**：`/sync`
- **类别**：工具
- **说明**：检查并更新核心文档，确保与项目代码保持一致

## 流程概览

```
扫描项目 → 检查CLAUDE.md → 检查AGENTS.md → 检查README.md → 检查CHANGELOG.md → 清理过时文件 → 报告
```

## 各步骤说明

### 1. 扫描项目

读取项目关键文件以建立当前状态基线：

- `package.json`：版本号、脚本命令、依赖列表
- 配置文件：`tsconfig.json`、`.eslintrc` 等
- 目录结构：`src/`、`docs/`、`.github/` 等核心目录

此步骤为后续对比提供准确的代码现状数据。

### 2. 检查 CLAUDE.md

对比 `CLAUDE.md` 中的内容与项目现状：

- 引用的文件路径是否存在
- 描述的命令、脚本是否与 `package.json` 一致
- 启动说明是否与当前入口文件匹配

### 3. 检查 AGENTS.md

对比 `AGENTS.md` 中的约束与项目现状：

- 架构约定是否与当前代码结构一致
- 工作流引用的 Agent 名称是否仍然有效
- 发布流程步骤是否匹配实际的 CI/CD 配置

注意：不修改 AGENTS.md 的现有约束内容，仅更新过时引用和描述。

### 4. 检查 README.md

对比 `README.md` 中的信息与项目现状：

- 安装命令是否可正常执行
- 启动命令、构建命令是否与 `package.json` 的 scripts 一致
- 目录结构描述是否与项目当前结构匹配
- 版本号、Badge 链接是否有效

### 5. 检查 CHANGELOG.md

对比 `CHANGELOG.md` 的更新状态：

- 最近版本的变更记录是否已包含最新 commit 的功能点
- 版本号是否与 `package.json` 一致
- 格式是否符合 Keep a Changelog 规范

### 6. 修复不一致

在确认差异后自动修复以下内容：

- 更新过时的命令、脚本名称
- 修正无效的文件路径引用
- 补充遗漏的新增功能描述
- 更新版本号至当前值

### 7. 清理过时文件

清理项目中不再需要的临时文件：

- `.claude/` 目录下的孤立文件（无对应配置引用的）
- 临时调试文件
- 已合并分支的残留文件

### 8. 产出报告

以结构化格式输出同步结果：

| 项目 | 状态 | 说明 |
|------|------|------|
| CLAUDE.md | 新增/更新/跳过 | 具体变更内容 |
| AGENTS.md | 新增/更新/跳过 | 具体变更内容 |
| README.md | 新增/更新/跳过 | 具体变更内容 |
| CHANGELOG.md | 新增/更新/跳过 | 具体变更内容 |
| 清理 | N项 | 清理的文件列表 |

## 适用场景

- 代码变更后（新增功能、重构、移除废弃代码）保持文档同步
- 定期维护，确保文档不随代码演进而过时
- 版本发布前做最终文档一致性检查

## 保护规则

同步过程中不会触碰以下内容：

- `dist/`、`build/` 等构建产物目录
- 流水线产生的驱动文档（`docs/YYYY-MM-DD/requirements/`、`docs/YYYY-MM-DD/tasks/`、`docs/YYYY-MM-DD/plans/` 等）
- `node_modules/` 依赖目录
- `.git/` 版本控制目录
