# `/publish` — 一键发布完整流程

- **命令**：`/publish`
- **类别**：流程
- **说明**：一键完成从质量检查到版本发布的全流程自动化

## 使用场景

| 场景 | 说明 |
|------|------|
| 常规版本发布 | 完成开发后的标准发布流程 |
| 质量检查后发布 | 通过所有质量门后自动推进 |
| PR 合并发布 | 创建 PR → 合并 → 打 Tag 三合一 |
| 紧急补丁发布 | 热修复后的快速发布通道 |

## 关键 Agent

| Agent | 职责 |
|-------|------|
| infra-deploy-expert | 部署与基础设施配置 |
| qa-review-expert | 最终质量签核 |

## 流程概览

```
前置检查 → 质量门 → 测试 → 版本递增 → 提交推送 → 创建PR → 合并PR → 打Tag → 完成
```

## 各步骤说明

### 1. 前置检查

确认当前处于正确的开发分支（通常为 `dev`），工作区干净无未提交变更。若存在未提交文件则提示用户先处理。

### 2. 质量门

执行代码质量检查，确保无基础性问题：

- **Lint**：运行 ESLint / Prettier 检查代码风格
- **Type-check**：运行 TypeScript 类型检查（`tsc --noEmit`）
- **Build**：执行项目构建，确认产物可正常生成
- **Deps Audit**：检查依赖安全漏洞（`npm audit`）

任一步失败即中止流程并输出错误信息。

### 3. 测试

运行项目完整测试套件（`npm test`），确保所有现有功能不受影响。测试失败则中止流程。

### 4. 版本递增

读取当前 `package.json` 中的版本号，按指定策略递增：

- `patch`：修复版本（1.0.0 → 1.0.1）
- `minor`：功能版本（1.0.0 → 1.1.0）
- `major`：破坏性变更（1.0.0 → 2.0.0）

默认使用 `patch`，可通过参数指定递增策略。

### 5. 提交推送

将更新后的 `package.json` 提交（commit message 含新版本号），然后推送当前分支到远端。

### 6. 创建 PR

使用 `gh pr create` 创建 Pull Request，目标分支为 `main`，PR 标题和描述自动生成为发布相关内容。

### 7. 合并 PR

使用 `gh pr merge --merge` 将 PR 合并到 `main` 分支，保持提交历史完整。

### 8. 打 Tag

在 `main` 分支上创建版本 Tag（如 `v1.0.1`），标注 release 版本，并推送到远端仓库。

### 9. 完成

输出发布摘要：新版本号、PR 链接、Tag 名称。

## 适用场景

- 任何需要发布新版本的项目
- 常规迭代发布（patch / minor）
- 重大版本升级（major）
