## 3. TDD vs 直接开发路由建议

### 原则

依据项目规范，以下场景需 TDD：
- 核心业务规则（本次无）
- 权限验证（本次无）
- 资金 / 配额 / 统计（本次无）
- 幂等性 / 重试 / 故障恢复（本次无）
- **状态机 / 状态转换**（本次无）
- **高风险接口契约**（REQ-011 CLI hash 路径影响升级安全）
- **可复现 Bug**（REQ-006、REQ-008 为可复现 Bug，条件分支多）

### 分类结果

#### TDD（测试驱动开发） — 3 条

| REQ | 名称 | TDD 理由 |
|-----|------|---------|
| REQ-006 | 文档加载失败修复 | 路径拼接逻辑涉及多个分支（日期目录 vs 扁平目录，编码 vs 不编码），不同 filepath 格式产出不同结果。需单元测试覆盖日期路径、扁平路径、含特殊字符路径等场景。 |
| REQ-008 | 当前 Gate 文档立即可见 | 条件分支逻辑：有 checkpoints 走日期匹配、无 checkpoints 走当天日期目录扫描、当天目录不存在返回空数组。三种场景需分别测试覆盖。 |
| REQ-011 | 全局 CLI Hash 路径统一 | 三条 hash 存储路径一致性影响 upgrade/diff 功能的正确性。路径错误会导致用户文件被误覆盖或误跳过。需测试全局/项目两种模式下的 hash 读写一致性。 |

## 7. 推荐执行策略

### 策略选择：并行分组 + 风险优先

由于全部为直接开发或 TDD，无需 DDD 建模阶段，可直接进入实现。

**推荐分组策略：** 按功能域（组1~组5）并行推进，每组内部串行。

```
组1: X6 面板 UI 修复（REQ-001~005）
  内部顺序：REQ-004(RAF) → REQ-002(FlowChart) + REQ-003(AgentGraph) → REQ-001(分割线) → REQ-005(视觉验证)
  注意：REQ-004 为 REQ-002/003 的前置依赖（动画 hook 先重构，再修改调用方）

组2: 文档系统修复（REQ-006~008, REQ-010）
  内部顺序：REQ-006(路径修复) → REQ-008(实时扫描) → REQ-007(目录锚定) → REQ-010(目录迁移)

组3: CLI 工具修复（REQ-011）
  独立任务，可并行

组4: 指令/模板同步（REQ-012~013）
  内部顺序：REQ-012(重命名) → REQ-013(模板补齐)
  注意：REQ-013 依赖 REQ-012 的输出（需先重命名再复制到模板）

组5: 文档规范同步（REQ-009）
  独立任务，可并行
```

### 并行机会

| 并行对 | 说明 |
