# 智能体行为准则

## 核心原则

这是一组所有项目智能体都要遵守的行为规范，用来减少常见 LLM 编码错误：隐含假设、过度设计、无关 diff、未经验证就宣布完成。

与项目级 `AGENTS.md`、`.codex/AGENTS.md` 和任务 Execution Packet 合并执行。若规则冲突，不要静默选择；先说明冲突并向用户或编排者确认。

## 编码前先想清楚

非平凡任务开始前，必须先明确：

- **假设**：你基于哪些前提行动。
- **歧义**：是否存在多种合理解释。
- **取舍**：是否有更简单或更保守的做法。
- **成功标准**：怎么验证任务完成。

如果关键信息不清楚，停止推进并提问；子智能体要回退给编排者，不得替用户补全需求。

简单任务可以压缩表达，但不能跳过不确定性、范围边界和验证。

## 简洁优先

只写解决当前问题所需的最少代码：

- 不添加用户没要求的功能。
- 不为单次使用创建抽象。
- 不增加没有明确需求的可配置性。
- 不处理现实中不可能发生的错误路径。
- 如果 200 行能缩成 50 行且更清晰，重写成 50 行。

自检问题：资深工程师会不会认为这份改动过度复杂？如果会，先简化。

## 手术式变更

每一行改动都要能追溯到用户请求、需求文档或 Execution Packet：

- 只改必须修改的文件和行。
- 不顺手美化相邻代码、注释或格式。
- 不重构没有坏掉、也不属于本任务的代码。
- 匹配现有风格，即使你个人会换一种写法。
- 只清理由本次改动制造的未使用 import、变量、函数和孤儿代码。
- 发现无关死代码时只报告，不删除，除非用户明确要求。

在并发或多智能体任务里，默认还有其他人同时工作；不要回滚、覆盖或重写他人的改动。

## 目标驱动执行

把任务转换成可验证目标，并循环到证据支持结论：

- “修 Bug” → 先复现或写回归测试，再修复，再验证症状消失。
- “加校验” → 覆盖有效和无效输入，再让测试通过。
- “重构” → 明确保留行为，必要时先跑基线验证，改后再跑同等验证。
- “接入功能” → 写清输入、输出、边界、验收条件和验证命令。

多步骤任务先写短计划：

```md
1. [步骤] -> 验证：[检查方式]
2. [步骤] -> 验证：[检查方式]
3. [步骤] -> 验证：[检查方式]
```

没有运行验证，就不能宣称测试、构建、修复或交付成功。未运行的验证必须说明原因。

## 红线

以下情况必须停下并纠正：

- 没有说明关键假设就开始实现。
- 存在两种解释却静默选择其中一种。
- 为“未来可能需要”增加功能、抽象或配置。
- 为了顺手清理扩大 diff。
- 修改 Execution Packet 之外的路径。
- 测试失败后直接改实现，未定位失败原因。
- 没有验证证据却暗示“已完成”“已修复”“通过了”。

## 交付自检

最终输出或交接前，逐项确认：

- 范围是否只覆盖用户请求或上游文档。
- 假设、取舍和残余风险是否已经说清。
- 所有新增或变更代码是否必要。
- 本次制造的未使用代码是否已清理。
- 验证命令、结果和未运行项是否真实记录。
