# 文档对齐闸门（Alignment Gates）

多代理流程中的每个阶段，除产出文档外，还必须满足进入下一阶段的最低对齐条件；未满足时不得继续推进。

---

## Gate A：需求 → 任务分解

**触发时机：** 阶段 1 完成，准备进入阶段 2（spawn `task_design`）

**必须满足条件：**
1. 需求摘要已写明
2. 目标与成功标准已写明
3. 范围内 / 范围外已写明
4. 关键模块 / 功能列表已写明
5. 风险与开放问题已写明
6. 已收敛结论（用户已确认或主会话明确）

**不通过处理：**
- 不得进入 `task_design`
- 必须回到主会话继续澄清

---

## Gate B：任务分解 → 执行规划

**触发时机：** 阶段 2 完成，准备进入阶段 3（spawn `planner`）

**必须满足条件：**
1. 任务 ID 完整（TASK-XXX 格式）
2. 任务名称完整
3. 类型完整（前端 / 后端 / 共享 / 测试）
4. 优先级完整
5. 完成标准完整
6. DDD 分类完整
7. TDD / test_after / manual_only 分类完整
8. 风险任务已标注
9. 文件所有权 / 共享路径提醒已写明

**不通过处理：**
- 不得进入 `planner`
- 必须回退 `task_design` 补齐

---

## Gate C：执行规划 → 实现

**触发时机：** 阶段 3 完成，准备进入阶段 5（spawn 实现代理）

**必须满足条件：**
1. 当前轮次目标已写明
2. 当前轮次范围已写明
3. 执行代理分工已写明
4. 共享区域唯一责任方已指定
5. 每个任务都有 Execution Packet
6. test_strategy 已指定
7. 风险提醒已写明
8. 实现者交接信息已写明

**不通过处理：**
- 不得进入实现阶段
- 必须回退 `planner` 补齐

---

## Gate D：实现 → 评审

**触发时机：** 阶段 5 完成，准备进入阶段 6（spawn `review_qa`）

**必须满足条件：**
1. 当前实现目标已写明
2. 输入依据已写明
3. 变更文件 / 范围已写明
4. 实现说明已写明
5. 测试 / 验证结果已写明
6. 边界与异常处理已写明
7. 风险 / 未解决项已写明
8. 对前端 / 后端 / 共享契约的影响已写明
9. 推荐下一步已写明

**不通过处理：**
- 不得进入 `review_qa`
- 必须回退对应实现代理补齐

---

## 闸门检查职责

| Gate | 检查者 | 检查方式 |
|------|--------|----------|
| A | 编排者（主会话） | 直接核对需求文档 |
| B | 编排者 + planner（规划前检查） | planner 自主检查，编排者复核 |
| C | 编排者 | 直接核对计划文档 |
| D | 编排者 | 直接核对实现文档 |

## 关键原则

- 闸门不是"建议"，是"硬性门槛"
- 缺一项就不得进入下一阶段
- 编排者有责任在 spawn 前完成闸门检查
- planner 有责任在规划前检查 Gate B
