# AGENTS.md

## 仓库预期

- 保持变更紧密围绕用户请求。
- 优先选择最小的正确变更而非广泛的重构。
- 编辑前先阅读相关代码路径。
- 引入新抽象前先复用现有模式。
- 不要进行不相关的清理变更。

## 工作风格

- 对于模糊、多步骤或高影响的工作，在编码前先澄清和规划。
- 对于代码库发现、模块边界检查或风险评估，在编辑前先探索。
- 对于实现工作，当任务大于简单修复时，将规划与编码分离。
- 保持中间推理简洁，聚焦于决策、风险和输出。

## 多智能体路由

- 当请求仍然模糊或成功标准不明确时，使用 `clarify_spec`。
- 当需求已知但实现任务仍需分解时，使用 `task_design`。
- 在进行非平凡编码工作前，使用 `planner` 定义任务范围、顺序、所有权和验证。
- 对于只读探索、代码库映射、依赖追踪和边界检查，使用 `repo_explorer`。
- 仅用于前端实现工作时使用 `frontend_implementer`。
- 仅用于后端实现工作时使用 `backend_implementer`。
- 对于有意义的变更，在宣布完成前使用 `review_qa`。

## 实现规则

- 遵循仓库现有的架构、命名和文件组织。
- 除非新边界有明确理由，否则优先编辑现有模块而非创建新模块。
- 除非任务明确需要变更，否则保持公共接口稳定。
- 当变更共享合约、模式、配置或类型时，先检查下游影响。
- 避免占位符逻辑、推测性抽象和死代码。

## 并行工作

- 仅当任务边界清晰且文件或所有权不重叠时才并行化。
- 不要对相同的共享合约、模式、配置或核心入口点运行并行实现。
- 如果共享区域的所有权不明确，请在实现开始前在规划中解决。

## 验证

- 先运行最小相关的验证，然后按需进行更广泛的检查。
- 完成前，对所触及区域运行适用的构建、测试、lint 和类型检查命令。
- 如果无法运行某个验证步骤，请明确说明并解释原因。
- 没有验证证据不要声称完成。

## 评审预期

- 确认结果与原始请求一致。
- 检查所触及区域可能的回归。
- 清晰地指出假设、风险和后续工作。
- 保持最终总结简洁且聚焦于实现。

## 仓库目录架构

顶层布局（随迭代会增删文件，以仓库实际为准）：

```
feijia/
├── AGENTS.md                 # 本文件：代理协作与完成标准
├── package.json              # workspace 根脚本（dev / typecheck / test / build / db:*）
├── vitest.config.ts          # schemas、http-client、web 等与 server 测试入口
├── apps/                     # 可运行应用 → apps/AGENTS.md
├── packages/                 # 共享库与数据库包 → packages/AGENTS.md
├── docker/                   # 本地 PG / Redis / MinIO → docker/AGENTS.md
├── docs/                     # 产品、计划、工作流文档（约定不在此目录放 AGENTS.md）
└── .codex/                   # Codex 技能与代理侧配置
```

## 代码层级指引

- 应用总览与 `apps/` 树：`apps/AGENTS.md`；共享包与 `packages/` 树：`packages/AGENTS.md`。
- 本地基础设施：`docker/AGENTS.md`；各应用子目录内另有对应 `AGENTS.md`。

## 文档

- 当行为、合约、设置或开发者工作流变更时，更新附近的文档。
- 将详细的工作流手册、规划模板和评审清单放在此文件之外。
- 将较长的流程指导放入专用文档并在需要时引用。
- 对于详细的多步骤交付流程、交接结构和评审手册，请遵循 `docs/workflows/workflow.md`。

## 完成定义

任务完成当：

- 请求的变更已实现或阻塞项已明确识别，
- 相关验证已运行或已明确说明未运行，
- 重要风险和假设已记录，
- 且结果已准备好进行评审或交接。
