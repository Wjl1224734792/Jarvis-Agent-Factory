# AGENTS.md

## 仓库预期

- 保持变更紧密围绕用户请求。
- 优先选择最小的正确变更而非广泛的重构。
- 编辑前先阅读相关代码路径。
- 引入新抽象前先复用现有模式。
- 不要进行不相关的清理变更。

## 工作风格

- 对于模糊、多步骤或高影响的工作，在编码前先澄清和规划。
- 对于代码库发现、模块边界检查或风险评估，在编辑前先探索。
- 对于实现工作，当任务大于简单修复时，将规划与编码分离。
- 保持中间推理简洁，聚焦于决策、风险和输出。

## 需求澄清（主控职责）

以下由**主会话**执行，不对应可委派的子代理。

- **要做**：把自然语言需求收敛为清晰、可实现、可验证的理解；明确目标、成功标准、范围、关键流程与对象、模块交互；必要时先调用 `repo_explorer` 再映射到代码结构；识别风险与开放问题；在对话中确认假设并视需要写入需求文档。
- **禁止**：在未与用户对齐的情况下独自补全模糊需求；将澄清工作交给子代理。
- **需求文档路径（可选）**：`docs/requirements/YYYY-MM-DD-<topic>-requirements.md`，建议包含：需求摘要、目标与成功标准、范围内/外、模块与功能列表、关键流程、关键对象、模块交互草案、与当前代码结构的映射、风险与开放问题、推荐下一步。

## 实现规则

- 遵循仓库现有的架构、命名和文件组织。
- 除非新边界有明确理由，否则优先编辑现有模块而非创建新模块。
- 除非任务明确需要变更，否则保持公共接口稳定。
- 当变更共享合约、模式、配置或类型时，先检查下游影响。
- 避免占位符逻辑、推测性抽象和死代码。
- 遵守rules，详情见`.codex\AGENTS.md`
- **Tailwind CSS**：组件与页面样式**仅允许**通过标记上的**内联类名**表达（如 `className="..."`，必要时常与 `cn` / `clsx` 等在**同一 TSX/模板处**拼出字符串）；**禁止** `@apply`、在 `.css` / CSS Module 中用工具类冒充组件样式、或任何不依托内联类名的等价写法。各应用入口样式表中为 Tailwind 工程服务的 `@import "tailwindcss"`、`@theme`、全局基础层等**配置性**内容除外，不在此限。

## 并行工作

- 仅当任务边界清晰且文件或所有权不重叠时才并行化。
- 不要对相同的共享合约、模式、配置或核心入口点运行并行实现。
- 如果共享区域的所有权不明确，请在实现开始前在规划中解决。

## 验证

- 先运行最小相关的验证，然后按需进行更广泛的检查。
- 完成前，对所触及区域运行适用的构建、测试、lint 和类型检查命令。
- 对 `test_strategy: tdd` 的任务：**完成声明前**须有一次「测试失败 → 同一套测试通过」的可核对记录（提交顺序、命令输出或评审备注均可）。
- 如果无法运行某个验证步骤，请明确说明并解释原因。
- 没有验证证据不要声称完成。

## 评审预期

- 确认结果与原始请求一致。
- 检查所触及区域可能的回归。
- 对标记为 `tdd` 的变更：核对是否遵循 **Red → Green**（必要时 **Refactor**）；若仅有最终实现、缺少失败用例阶段说明，应打回或要求补证据。
- 清晰地指出假设、风险和后续工作。
- 保持最终总结简洁且聚焦于实现。

## 代码层级指引

- 应用总览与 `apps/` 树：`apps/AGENTS.md`；共享包与 `packages/` 树：`packages/AGENTS.md`。
- 本地基础设施：`docker/AGENTS.md`；各应用子目录内另有对应 `AGENTS.md`。
- **Codex 代理**：`.codex/agents/*.toml`。

## 多代理编排

完整编排流程封装在 `.codex/skills/agent-orchestration/` 技能中，仅在用户显式触发时加载。编排详情请查阅该技能的 `reference/` 目录。

简单任务和单步查询无需走编排流程，按需 spawn 单个 agent 即可。

## 完成定义

任务完成当：

- 请求的变更已实现或阻塞项已明确识别，
- 相关验证已运行或已明确说明未运行，
- **`tdd` 任务**：具备可核对的 Red/Green（及需要的 Refactor）说明或历史，
- 重要风险和假设已记录，
- 且结果已准备好进行评审或交接。
