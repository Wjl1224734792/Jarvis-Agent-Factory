# AGENTS.md

## 仓库预期

- 保持变更紧密围绕用户请求。
- 优先选择最小的正确变更而非广泛的重构。
- 编辑前先阅读相关代码路径。
- 引入新抽象前先复用现有模式。
- 不要进行不相关的清理变更。

## 工作风格

- 对于模糊、多步骤或高影响的工作，在编码前先澄清和规划。
- 对于代码库发现、模块边界检查或风险评估，在编辑前先探索。
- 对于实现工作，当任务大于简单修复时，将规划与编码分离。
- 保持中间推理简洁，聚焦于决策、风险和输出。

## 多智能体路由

- 当请求仍然模糊或成功标准不明确时，使用 `clarify_spec`。
- 当需求已知但实现任务仍需分解时，使用 `task_design`。
- 在进行非平凡编码工作前，使用 `planner` 定义任务范围、顺序、所有权和验证；**每条待实现任务须标注测试策略**（见下节 `test_strategy`），并与 `docs/tasks` 等文档中的「TDD」表述对齐。
- 对于只读探索、代码库映射、依赖追踪和边界检查，使用 `repo_explorer`。
- 仅用于前端实现工作时使用 `frontend_implementer`（对 `test_strategy: tdd` 的任务必须按 **Red → Green → Refactor** 顺序执行）。
- 仅用于后端实现工作时使用 `backend_implementer`（同上）。
- 对于有意义的变更，在宣布完成前使用 `review_qa`（对 TDD 任务核对 Red/Green 证据）。

## TDD 编排（Red / Green / Refactor）

`planner` 为每条实现类任务指定 `test_strategy` 之一：

| 取值 | 含义 |
|------|------|
| `tdd` | 测试驱动：先失败用例，再最小实现，再按需重构 |
| `test_after` | 实现后补测或扩展现有测试，仍须在交付前跑通 |
| `manual_only` | 仅手工/浏览器验证；须写明范围与限制（默认少用） |

**`tdd` 时子代理须遵守的顺序：**

1. **Red**：新增或修改测试，使当前行为**明确失败**（断言目标行为或拒绝错误行为）；运行对应测试命令并保留失败输出或日志说明。
2. **Green**：编写**最小**生产代码令该测试通过；不顺带做大范围重构。
3. **Refactor**：在测试仍绿的前提下整理结构、去重、命名；若有行为变化须回到 Red。

**约束：**

- 共享合约、路由前缀、DB schema 等高风险变更若标为 `tdd`，与其实现相关的测试步骤**不要**与另一代理并行抢改同一文件。
- 无法自动化测试的边界（如纯 UI 动效）可标 `test_after` 或 `manual_only`，并在 `planner` 产出中写清验收方式。

## 实现规则

- 遵循仓库现有的架构、命名和文件组织。
- 除非新边界有明确理由，否则优先编辑现有模块而非创建新模块。
- 除非任务明确需要变更，否则保持公共接口稳定。
- 当变更共享合约、模式、配置或类型时，先检查下游影响。
- 避免占位符逻辑、推测性抽象和死代码。

## 并行工作

- 仅当任务边界清晰且文件或所有权不重叠时才并行化。
- 不要对相同的共享合约、模式、配置或核心入口点运行并行实现。
- 如果共享区域的所有权不明确，请在实现开始前在规划中解决。

## 验证

- 先运行最小相关的验证，然后按需进行更广泛的检查。
- 完成前，对所触及区域运行适用的构建、测试、lint 和类型检查命令。
- 对 `test_strategy: tdd` 的任务：**完成声明前**须有一次「测试失败 → 同一套测试通过」的可核对记录（提交顺序、命令输出或评审备注均可）。
- 如果无法运行某个验证步骤，请明确说明并解释原因。
- 没有验证证据不要声称完成。

## 评审预期

- 确认结果与原始请求一致。
- 检查所触及区域可能的回归。
- 对标记为 `tdd` 的变更：核对是否遵循 **Red → Green**（必要时 **Refactor**）；若仅有最终实现、缺少失败用例阶段说明，应打回或要求补证据。
- 清晰地指出假设、风险和后续工作。
- 保持最终总结简洁且聚焦于实现。

## 文档

- 将详细的工作流手册、规划模板和评审清单放在此文件之外。

## 完成定义

任务完成当：

- 请求的变更已实现或阻塞项已明确识别，
- 相关验证已运行或已明确说明未运行，
- **`tdd` 任务**：具备可核对的 Red/Green（及需要的 Refactor）说明或历史，
- 重要风险和假设已记录，
- 且结果已准备好进行评审或交接。
