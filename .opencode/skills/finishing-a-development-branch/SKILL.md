# 完成开发分支

## 概述

"代码写完了"不等于"分支完成了"。功能分支的完成是一个多步骤的收尾流程，涉及审查、清理、验证、合并、部署和文档归档。跳过任何一步，都会给后续开发和维护埋下隐患。

**核心原则：** 完成的分支 = 代码已审查 + 测试已通过 + 已合并到目标分支 + 功能分支已删除 + 文档已更新。

**必需背景：** 你必须理解 `verification-before-completion`（验证清单）、`git-workflow-and-versioning`（分支与提交规范）。

## 何时使用

**适用场景：**
- 功能开发完成，准备合并
- Bug 修复完成，准备合并
- 重构完成，准备合并
- 需要将功能分支的代码集成到主分支

**不适用场景：**
- 探索性分支（spike/prototype）——不需要走完整流程，但需记录探索结论
- 长期存在的共享分支（develop/main 等）——这些永不"完成"
- 尚未通过验证的中间状态

## 完成流程（按序执行）

### 第一步：自检

在请求审查前，先自我检查：

```
□ 所有验收标准已满足（对照 TASK-XXX 的 acceptance_criteria）
□ 自动化验证已通过（lint + typecheck + 测试）
□ 无遗留调试代码（console.log、注释掉的代码、临时测试数据）
□ 变更范围未越界（对照 Execution Packet 的 allowed_paths）
□ commit history 干净（每个 commit 是独立的逻辑变更）
□ 敏感信息已排除（密钥、token、密码未提交）
```

### 第二步：同步上游

在合并前确保你的分支与目标分支同步：

```bash
# 方式 1：Rebase（推荐——线性历史）
git checkout feature/xxx
git fetch origin
git rebase origin/develop

# 方式 2：Merge（安全——不改变你的 commits）
git checkout feature/xxx
git fetch origin
git merge origin/develop
```

**同步后必须重新验证：**
- 重新运行测试（合并可能引入冲突）
- 解决冲突后，确认代码仍符合预期

### 第三步：代码审查

根据项目规范提交 PR/MR：

- **PR 标题**：简洁的描述（如 `[TASK-001] 用户注册功能`）
- **PR 描述模板**：
  ```
  ## 变更概述
  <一句话描述>
  
  ## 关联需求/任务
  - REQ-001：用户注册
  - TASK-001：用户注册功能
  
  ## 变更文件
  - src/auth/register.ts（新增）
  - src/db/schema/user.ts（修改）
  - tests/auth/register.test.ts（新增）
  
  ## 验证证据
  - 单元测试：12 通过 / 0 失败
  - 集成测试：3 通过 / 0 失败
  - lint：零警告
  
  ## 审查要点
  - <提示审查者重点关注的部分>
  ```

### 第四步：处理审查意见

收到审查意见后：

1. **不要防御**：审查是对代码质量的意见，不是对你能力的质疑
2. **逐条回复**：每条意见要么接受修改，要么解释为什么不改
3. **修改后追加 commit**：不要 squash 以保持审查历史（除非审查者要求）
4. **标记已处理**：在 PR 中标记已解决的对话

### 第五步：合并

审查通过且 CI 通过后，合并到目标分支：

```bash
# Squash merge（推荐——干净的主分支历史）
# 将整个功能分支的所有 commit 压缩为一个 commit 合并
# commit message 保留为功能概述

# Merge commit（需要保留详细历史时使用）
# 产生一个合并 commit，保留功能分支的所有 commits
```

### 第六步：清理

合并完成后：

```bash
# 删除本地功能分支
git branch -d feature/xxx

# 删除远程功能分支
git push origin --delete feature/xxx

# 如果使用了 git worktree
git worktree remove ../feature-xxx
```

### 第七步：部署验证

如果涉及部署：

1. 在 staging 环境验证功能
2. 执行冒烟测试（核心路径是否正常）
3. 监控错误日志和性能指标
4. 确认无回归后，推进到生产环境

### 第八步：文档归档

- 实现文档已输出到 `docs/implementation/`
- 如涉及 API 变更，API 文档已更新
- 如涉及架构决策，ADR 已记录
- 如有操作手册变更，运维文档已更新

### 第九步：发布（如适用）

参照 `shipping-and-launch` 技能执行发布流程。

## 多分支并行开发

当多个功能分支同时开发时：

1. **尽早合并**：每天至少同步一次上游分支，避免大规模冲突
2. **小功能优先**：小分支优先合并，给大分支更频繁的同步机会
3. **冲突协商**：如果两个分支修改了同一个共享区域，先沟通、后合并
4. **不要等所有分支都完成**：完成一个合并一个，不要积攒

## Git Worktree 使用（可选）

当需要在多个分支间快速切换时：

```bash
# 创建 worktree（在新目录中检出另一个分支）
git worktree add ../project-hotfix hotfix/critical-bug

# 查看所有 worktree
git worktree list

# 完成后清理
git worktree remove ../project-hotfix
```

使用 worktree 的好处：
- 不需要 stash/pop，主打功能分支和被临时打断的修复互不干扰
- 可以同时在两个分支上运行测试

## 常见借口

| 合理化借口 | 现实 |
|-----------|------|
| "功能测过了，不用删除分支，留着吧" | 死分支堆积让仓库混乱。分支历史已在 merge commit 中保留。删掉。 |
| "这个功能不急，代码留在分支上就行" | 未合并的代码 = 未集成的风险。分支越老，合并冲突越多。尽早合并。 |
| "文档以后再补，先把代码合并了" | 没有文档的合并 = 隐式知识。其他人（包括未来的你）不知道这段代码为什么这样写。 |
| "CI 通过了就行，不用本地再测" | CI 和本地可能有环境差异。合并后的第一个 commit 必须是你本地验证过的。 |
| "审查意见太多了，重新开一个 PR 吧" | 审查历史是变更上下文的一部分。继续在原 PR 上修改，不要另起炉灶。 |

## 红线——停下来，按流程走

如果你发现自己在想：
- "小改动，直接 merge 不用 review"
- "这个功能完成了，直接合 main"
- "审查通过了，不用再跑测试"
- "分支太多了，合完再一起删"
- "文档写了别人也不看，省了吧"

**以上这些都意味着：你在跳过完成流程。回到第一步，逐步完成。**

## 验证清单

分支完成前确认：

- [ ] 自检清单全部通过
- [ ] 已与目标分支同步且无冲突
- [ ] 代码审查已完成且意见已处理
- [ ] CI/CD 全部通过
- [ ] 已合并到目标分支
- [ ] 功能分支已删除（本地 + 远程）
- [ ] 部署验证已通过（如涉及）
- [ ] 实现文档已输出
- [ ] 相关文档（API/ADR/运维）已更新
- [ ] 发布已完成（如适用）
